## 全体の流れ

intent-planner は、AI に大きめの実装を頼む**前**に「何を作りたいか・何を守りたいか」を AI と一緒に整理し、実装が終わった**あと**に学びを整理した文書へ書き戻す、というひとめぐりのループです。

```
①意図を整理する          ②実装に渡す        ③実装する         ④学びを戻す
discover → compass        export             （cc-sdd 等        writeback
  → packets                                    で実装）          → improve
        └──────────────────── 迷ったら status / validate ───────────────────┘
```

- **①整理する** — `/intent-discover`（全体像）→ `/intent-compass`（守るべき基準）→ `/intent-packets`（作業単位に分解）
- **②渡す** — `/intent-export-cc-sdd`（または `/intent-export-openspec`）で、選んだ作業単位を実装ツールの下書きに変換
- **③実装する** — 下書きを cc-sdd / OpenSpec に渡して実装（intent-planner は下書きまで担当）
- **④戻す** — `/intent-writeback`（実装の学びを記録・反映）、節目に `/intent-improve`（全体の整合を取り直す）

各ステップの成果物は `.intent/` フォルダの中の Markdown です。レビューしてから次へ進めます。**現在地に迷ったら `/intent-status`** — いまどこにいて、次に何をすればいいかを1つだけ教えてくれます。

### 案件に必要な分だけ使う

intent-planner は、設計のずれや説明のやり直し、統合の手戻りが大きくなりやすい案件に、**必要最小限の判断材料**を渡すための道具です。指示を増やすこと自体が目的ではありません。細かな手順は減らしながら、外してはいけない意図を明確にすることを目指します。vibe coding で十分な小さな試作には重すぎる場合があるため、packet から `direct` で実装するか、この工程自体を省略できます。軽量版のために別のモードを増やすのではなく、既存の経路から必要な分だけ使います。

実装時の情報は、拘束力に応じて4段階に分かれます。Invariant は必ず守る約束、packet の Scope / Acceptance は今回の実装で満たす範囲と受入条件、Decision は現在の設計基準、Preference / Heuristic は参考にする候補です。この**最小十分**な情報で境界を示し、AI が合意済みの範囲内を自分で進める形を**境界付き自律**と呼びます。範囲を越える案だけを人へ戻します。また、**確定材料**と、実装時に必要になった場合だけ見直す**再確認候補**を分け、必要な時点で必要な分だけ渡します。関係のない履歴をすべて渡すことはしません。

③と④では、必要なときだけ `.intent/execution-contract.md` を読みます。この文書は、必ず守る約束、今回の範囲と受入条件、現在の設計基準、参考にとどまる候補を区別するための短い共通契約です。範囲内の実装方法は AI に任せます。範囲を越える価値のある案が見つかった場合は、現行設計を維持するか、設計変更を承認するか、次の packet に送るかを人が選ぶまで待ちます。この契約がない旧環境では、従来どおり packet と Compass を使います。

|---|---|---|
| `/intent-discover` | 最初に | 作りたいものの全体像を組み立てる |
| `/intent-compass` | discover のあと | 守るべき基準（不変則・進んではいけない方向）を作る |
| `/intent-packets` | compass のあと | 実装に渡せる作業単位（packet）に分ける |
| `/intent-export-cc-sdd` | 実装に渡すとき | 作業単位を cc-sdd の下書きに変換する |
| `/intent-export-openspec` | 実装に渡すとき | 作業単位を OpenSpec の下書きに変換する |
| `/intent-writeback` | 実装が終わったら | 実装で得た学びを記録して文書へ反映する |
| `/intent-improve` | 数件こなした節目 | 文書と実装のズレをまとめて直す |
| `/intent-status` | 迷ったらいつでも | 現在地と「次の一手」を1つ教える（読むだけ） |
| `/intent-validate` | 実装に渡す前 | 文書どうしの矛盾・抜け・重複を検査する（読むだけ） |
| `/intent-overview` | 全体を見たいとき | 散らばった成果物を1枚に集約して見せる（読むだけ） |
| `/intent-from-spec` | 既存の仕様書や断片メモがあるとき | 仕様書・メモから「書かれていない意図」を見つける（読むだけ） |

| `/intent-release-note` | リリース時 | git 履歴から「なぜ変わったか」を補ったリリースノートを作る（読むだけ） |
| `/intent-db-design` | DB 設計を伴うとき | 意図・不変則・既存スキーマから DB 設計の叩き台を作り、DB 固有の検査軸で点検する（手動発動） |

「読むだけ」のコマンドは何も書き換えないので、AI が文脈に応じて自動で実行することもあります。文書を書き換えるコマンド（discover / compass / packets / writeback / improve / export）は、あなたがスラッシュで明示的に呼んだときだけ動きます。

## 各コマンドがやること

### 計画フェーズ（最初にこの順で）

**`/intent-discover` — 全体像を組み立てる**
課題やアイデアについて AI がいくつか質問し、意図を階層（L0 目的 〜 L4 作業候補）に整理して `.intent/intent-tree.md` に書き出します。あわせて、この案件をどう進めるか（[モード](#モード進め方の切り替え)）を推奨・確定し、入口でどこまで詰めるか（[詰めの問い](#詰めの問いdesigner-questions)）を確認して記録します。解がまだ発散しているときは、AI が仮説・反例・別の問題設定をすべて暫定として示します。ここで AI は結論を決めず、次の `/intent-compass` で人が許容範囲・除外・不変条件を定めます。以後の実装は、その確定した compass と packet の境界内だけで動きます。

**`/intent-compass` — 守るべき基準を作る**
判断の基準を `.intent/intent-compass.md` に作ります。中身は主に次の4つです。

- **North Star（目指す姿）** — このプロジェクトが向かう先
- **Anti-direction（進んではいけない方向）** — やらないと決めたこと
- **Invariants（不変則）** — 何があっても壊してはいけない性質
- **Decision Rules（判断ルール）** — 「何と比べて・なぜそう決めたか」を残した決定の記録

これらは実装中の AI に毎回渡る基準になり、「正しく動くけど設計意図からズレている」変更を防ぎます。

実装時に読むのは全履歴ではなく、案件の領域や変更対象に関係する `active` な Invariant / Decision だけです。無関係な判断や `superseded`（置換済み）の判断は対象から外します。関係の有無を判断できないものは黙って捨てず、確認候補として示します。`Revisit when` に書かれた見直し条件が成立しても、判断を**自動で失効・削除せず**、従来の判断、新しい事実、成立した条件をまとめて示します。人が承認した更新だけを writeback で反映します。

**`/intent-packets` — 作業単位に分ける**
実装に渡せる単位（packet）に分解します。各 packet は1ファイルで、親の意図への参照・スコープ・守るべき不変則を持ちます。聞き漏らしやすい技術判断（整合性・冪等性・エラー時の挙動・認可など）を「決めるべきスロット」として埋め込み、決まっていないものは理由つきで保留します。最初に着手すべき packet を理由つきで1つ推薦もします。

### 実装に渡す（どちらか）

**`/intent-export-cc-sdd`** — 選んだ packet 1つを [cc-sdd](https://github.com/gotalab/cc-sdd) の下書きに変換します。下書きの requirements には、packet の期待挙動・適合基準（受入をどう測るか）が「受入基準の材料」（`## Acceptance Material` 節）として含まれ、下流の要件生成が受入基準を創作せずに書けるようになっています。受け渡しの案内には、design / tasks フェーズへ進むときに該当ヒントファイル（design.md / tasks.md）の本文を渡す手順も含まれます（steering が未整備でも下書きが文脈を供給します）。
**`/intent-export-openspec`** — 選んだ packet 1つを [OpenSpec](https://github.com/Fission-AI/OpenSpec) の proposal 下書き＋ヒントに変換します。

どちらに渡すかは案件の性質で選ばれます（決め打ちではありません）。`/intent-packets` の「次の一手」が、案件の出口指定・モード・前提（`.kiro/` や `openspec/` フォルダの有無）から出口を提案します。一意に決まらなければ候補を並べます。enforcement を設定していれば、渡す前に書き戻し漏れを検査します。

### 維持フェーズ（実装のあと）

**`/intent-writeback` — 学びを戻す**
実装で分かったこと（新しく決まったこと・不変則の違反・暗黙の挙動・保留が解けたこと）を、まず `.intent/deltas.md` に「delta（差分の候補）」として記録します。この時点では元の文書は書き換わりません。あなたが承認した項目だけが Intent Tree / Compass / Packets に反映されます。承認の手間は学びの種類で変わり、判断基準を変えるような重い学びだけは1件ずつ確認、それ以外は「止めたい項目だけ指定すれば残りは一括反映」です。

### リリース後の成果を記録する

通常の実装学習とは別に、リリース後の利用者成果も `/intent-writeback` で記録できます。利用者が「成果についての学びを記録する」と明示したときだけ、この経路を使います。

1. Intent Tree の対象 L1 に `成果の物さし:` を1行で書きます。これは「利用者価値が出たと何で分かるか」であり、開発の受入に使う `計測基準:` や、守る約束の破損を調べる `検査オラクル:` とは別です。
2. `/intent-writeback` に成果についての学びを伝えます。結果は `価値が出た | 価値が出なかった | まだ分からない` から選び、生データを貼らず要約します。出所として `誰が計測したか`、`いつ計測したか`、`どこで計測したか` も添えます。
3. 記録はまず Packet 単位の delta に `pending`（承認待ち）で追記されます。この段階では Intent Tree は変わりません。同じ意図を繰り返し観測した場合も、過去の記録を上書きせず追記します。
4. 内容を確認して承認すると、対象 L1 の `成果についての学び:` に最新の結果と要約が反映されます。見送った記録は残りますが、L1 は変わりません。
5. `/intent-status` と `/intent-overview` は、物さしがあり承認済み結果がなければ `リリース後の結果待ち`、結果があれば3値と要約を示します。pending の記録を確定結果には使いません。

出所が足りない場合、`outcome-provenance-missing`（成果記録の出所不足を知らせる警告）が欠けた項目を示します。警告は記録・承認・検証を止めません。この機能は外部サービスから成果を自動取得せず、数値の自動判定も行わないため、人が確認した要約だけを扱います。不具合の分類とも自動統合しません。

**`/intent-improve` — 全体を整え直す**
数件こなした節目やリリース前に、`.intent/` と実装の現実を completeness（抜け）/ correctness（正しさ）/ coherence（整合）の3軸で突き合わせ、ズレの是正案を提示します。反映は承認後だけです。packet 単位の writeback では拾えない「実装にあるのに文書に無い／文書にあるのに実装と食い違う」を、まとめて検出します。

### 読むだけのコマンド（いつでも）

何も書き換えないので安全に使えます。

- **`/intent-status`** — 現在地の要約と「次に人が決めること」を1つだけ推奨します。その1件を先頭に置き、続けて「工程の状態」「未決の設計判断」「利用者成果」を分けます。工程が順調でも成果が出たとはみなさず、根拠がなければ利用者成果は「未観測」と表示します。工程レールや Candidate Packets は維持し、書き戻し漏れ・整合違反など危険な知らせ以外の詳細は後ろへ退避します。総合PASSやスコアにはまとめません。
- **`/intent-validate`** — 実装に渡す前に、文書どうしの矛盾・抜け・境界の重複を深刻度つきで報告します。曖昧な言い回しや、Compass の基準が各 packet に行き渡っているかも検査します。
- **`/intent-overview`** — 散らばった成果物を横断して、ツリー図・工程レール・進捗・ギャップを1枚に集約して見せます。ここでも工程の状態、未決の設計判断、利用者成果を別々に映し、利用者成果の根拠が無ければ「未観測」のまま保ちます。
- **`/intent-from-spec`** — 既存の仕様書（PRD・issue など）や断片メモ・走り書き・音声書き起こしを入力に、書かれていない意図（不変則・前提）を物差しに照らして「抜け」として炙り出します。断片のときは話題ごとに束ね「決まっている／迷っている」を仕分けてから抽出するので、頭が散らかったままでも intent の入口に立てます。抽出は仮説として提示します。
- **`/intent-to-spec`** — 意図を1本の読める自然言語の仕様書に書き起こします。根拠の無い記述には「推測」の印を付けて、でっち上げを防ぎます。どこまで詳しく書くか（詳細度: 概要 / 標準 / 詳細）を選べ、指定しなければ生成前に一度だけ確認します（範囲・体裁と直交する軸で、「詳細」を選ぶと packet 本文まで素材にした深い文書が出ます。読み手向けに絞る体裁〔一枚もの・定例報告・意思決定メモ〕は厚み固定で詳細度を適用しません）。
- **`/intent-release-note`** — git のコミット履歴を読み、各コミットを意図と照合して「なぜ変わったか」を補ったリリースノートを作ります。意図に紐づかないコミットは薄い行で残し、文書と実装の落差を見せます。

### DB 設計の専用ビュー（手動発動）

- **`/intent-db-design`** — 永続データモデルを設計する packet に対し、意図・不変則・既存スキーマ（migration/DDL）の三層を読み、テーブル定義・制約・インデックス・命名を、各記述を射影元へトレースしながら（根拠の無い記述には「推測（inferred）」「未確認（unverified）」の印を付けて）`.intent/db-design/` に DB 設計の**叩き台**を作ります。出力は設計の叩き台であって要件ではなく、export（requirements）には混ぜません。あわせて DB 固有の検査軸（正規化崩れ・欠落インデックス・N+1 誘発・制約漏れ・命名一貫性）と、不変則への適合（immutable 等）・不可逆性（データ投入後の migration コスト）を点検します（すべて警告のみ・止めない・書き換えない）。実装後は `/intent-validate` が「叩き台 vs 実際のスキーマ」のズレを見つけ、`/intent-writeback` が「なぜ違う設計にしたか」を記録します。発動は手動で、`/intent-status` がこのコマンドを促すことはありますが自動では動きません。

## 作られるファイル（`.intent/`）

intent-planner の成果物は、すべて `.intent/` フォルダ内の Markdown です。基本方針は、「1単位を1ファイルにする」「進行中と完了済みを分ける」「チームで共有する正本と個人の作業用ファイルを分ける」の3つです。

```
.intent/
├── intent-tree.md        # 意図の階層（L0 目的 〜 L4 作業候補）
├── intent-compass.md     # 判断基準: North Star / Anti-direction / 不変則 / Decision Rules
├── compass-archive.md    # 覆された判断ルールの退避先（履歴。下記の分割記録の1つ）
├── packets/
│   ├── index.md          # 進行中 packet の一覧（自動生成・手で編集しない）
│   ├── plan.md           # 計画レベルの記録
│   ├── active/           # 進行中の packet。1 packet = 1 ファイル
│   └── archive/<年>/     # 完了・置換済みの packet。消えずにここへ移る
├── cc-sdd/<スラッグ>/    # 実装ツールへ渡す下書き（ローカル作業物・Git 非追跡）
├── deltas.md             # 実装から得た学びの受け皿（承認後に本文書へ反映。下記の分割記録の1つ）
├── export-log.md         # 実装に渡した履歴（1 行 = 1 回。下記の分割記録の1つ）
└── mode.md, modes/       # 進め方モードの記録と定義
```

> **追記型の記録（deltas / export-log / drift-log / compass-archive）は複数のファイルに分けて保存されます。** 上の `deltas.md` などは、現在の内容をまとめて表示するために再生成できるファイルです。実際の記録は `deltas/<単位>.md` のように**1単位1ファイル**で保存され、完了した項目は `deltas/archive/` へ移ります。こうすることで、複数の作業が同じファイルの末尾を書き換える事態を避けられます。履歴は Git と `archive/` に残ります。分散した記録をまとめて読みたいときは、`/intent-overview` が進行中と保管済みを分けて表示するため、手作業で集約する必要はありません。

### あなたが触るもの・コマンドに任せるもの

| 関わり方 | 対象 | やること |
|---|---|---|
| **読んで承認する** | active/ の packet・deltas.md の学び・compass の基準 | コマンドが出す案を承認または修正する。これが人の主な仕事 |
| **直接書いてもよい** | tree / compass の Open Questions への回答 | 直接編集するか、会話で伝えれば次のコマンド実行時に反映される |
| **触らない（自動管理）** | index.md・export-log.md・cc-sdd/ の下書き | 手で編集する必要なし |

### packet は消えない

packet は active/ に1ファイルで生まれ、あなたの承認で進行中になり、実装と書き戻しの完了で archive/ へ移ります。計画を見直して置き換えた packet も「superseded（置換済み）」として残ります。**削除はされないので、何度やり直しても過去の判断は失われません。**

### Git はそのままコミットすればよい

`.intent/` の大半はコミット対象で、チームで共有する記録になります。ローカルだけで使うファイル（cc-sdd/ の下書きと `.intent/discovery/` 配下の作業ディレクトリ）は、インストーラが `.gitignore` に追加します。「どの packet を実装に渡したか」は、コミットされる `export-log.md` から全員が同じ情報を確認できます。

選択中のモードや質問の進め方など、個人やセッションごとに変わる情報は、`/intent-discover` が実行ごとに作る作業ディレクトリ `.intent/discovery/<スラッグ>-<rand>/mode.md` にローカル保存されます。discover は作成したディレクトリ名を出力し、後続の `/intent-compass` や `/intent-packets` はその名前を引き継いで `mode.md` を読みます。並行するセッションは別々のディレクトリを使うため、設定を上書きしません。旧形式の `.intent/mode.local.md` も互換性のため引き続き読み取れます。チームで共有する enforcement / drift-watch の設定だけが `.intent/mode.md` に保存され、コミット対象になります。

## モード（進め方の切り替え）

プロジェクトの状況に応じて、意図を整理する進め方を「モード」として切り替えられます。`/intent-discover` が状況に合うモードを提案し、選んだ結果を記録します。

- **standard** — 既定の汎用モードです。新規プロダクトにも、既存プロジェクト内でまだ言語化されていない機能にも使えます。
- **refactor** — 大規模な既存プロジェクトのリファクタリングや再設計向けです。コードから意図を読み取る手順を含みます。
- **behavior-unknown** — 仕様書がなく、現在の振る舞いも分からない既存システム向けです。
- **feature-growth** — 稼働中のシステムへ機能を追加するときに使います。既存機能への影響を調べ、追加内容を作業単位へ分けます。
- **non-code** — 文書、業務、研究など、プログラム以外の成果物向けです。cc-sdd / OpenSpec を経由せず、読める文書を直接生成します。

新しいモードは `.intent/modes/` にファイルを1枚足すだけで追加できます（`.intent/modes/README.md` 参照）。

`non-code` を選び、非プログラム向けの出口で `/intent-to-spec` を実行すると、cc-sdd/openspec を経由せず `.intent/nl-spec/` に読める成果物（記事構成案・業務手順書・調査ブリーフ等）が出力されます。

## 詰めの問い（designer-questions）

モードとは別の軸として、入口で**どこまで詰めるか**を選べます。`/intent-discover` が、フローが代わりに問うてくれること（計測できる成功基準、最初に E2E を貫くか＝walking skeleton、UI があれば画面ラフ、検証案件なら仮説と GO/NO-GO）を説明し、要否を確認して記録します。

- **on** にすると、共通の3質問（計測基準・walking skeleton・画面ラフ）と検査が有効になります。さらに「検証（PoC）か本番か」を確認し、PoC なら「何が観測されたら仮説を棄却するか（反証条件）」と GO/NO-GO 基準の質問が加わります。
- **off** のときの増分は、要否を聞く1問だけです。

on を選ぶと、続けて詰めの深さを確認します。未記載または不明な値は **standard（標準）** として扱い、これまでどおり最大1往復で進みます。**deep（深掘り）** は利用者が明示的に選んだ場合だけ有効になり、AI が自動で切り替えることはありません。

deep では、機能の中身、前提、エッジケース、反例、失敗とみなす条件まで確認します。一度に尋ねるのは最大4問です。必要な論点を「未確認」「回答済み」「後で確認」「不明」「該当なし」として扱い、回答済み・後で確認・不明・該当なしのいずれかですべて片付くまで、未確認の論点だけを次の質問束で確認します。質問束の回数には固定の上限を置きませんが、片付いた論点を言い換えて再質問することもありません。

各質問束では、その場で終了することも選べます。終了した場合は新しい質問を出さず、残る未確認の論点を `.intent/intent-tree.md` の Open Questions に記録して先へ進みます。回答から新しい論点が必要になったときも、利用者の回答か既存資料に根拠があり、既存の論点と同じ意味でないものだけを追加します。

on/off に関わらず、要望から目指す構成がほぼ一意に決まると判断したときは（cron 化・CLI 化など）、中立な選択肢で回り道させず「目指す構成はこれですよね」と1問で先に当てて確認します。

## 観点別レビュー

観点別レビューは、内容を詰める質問を受け、**deep（深掘り）を選んだときだけ**使います。質問を受けない場合と standard（標準）では始まらず、既存の質問量も増えません。職種の人格を演じるのではなく、案件に必要な「製品を決める観点」「進行を管理する観点」「利用体験を設計する観点」を責任範囲として別々に確認します。この3つは固定された全職種一覧ではなく、案件に必要な別の専門観点も追加できます。

- 製品を決める観点では、解決する問題と根拠、対象者と利用状況、代替手段、提供価値と成功の見分け方、優先順位、今回の範囲と範囲外を確認します。
- 進行を管理する観点は、複数人、複数の作業線、外部依存、期限、承認、引き継ぎ、リリース調整のどれか一つがある場合だけ使います。判断する役割、依存関係、順序、承認点、リスク、代替手段、リリース条件、切り戻しを確認します。
- 利用体験を設計する観点では、利用前から利用後までの流れ、接点、見える部分と裏側の支援、待ち、引き継ぎ、失敗、離脱、再開、アクセシビリティ、文言とトーンを確認します。

たとえば、外部依存や期限など、上の条件が一つもない単独開発では、進行の観点を非該当として閉じ、進行の質問を増やさないまま製品と利用体験を確認します。一方、単独開発でも期限があれば進行を管理する観点を使います。チームで進め、外部承認がある案件でも進行を管理する観点を使い、判断する役割、依存関係、承認点、リリース条件を確認します。

担当者がいる観点では、その担当者へ問いを直接示します。担当者がいない観点では、AIが代行していると明示したうえで、根拠を添えた暫定回答を示します。回答は「確認済みの事実」「根拠付きの推測」「未確認」「非該当」に分けます。根拠のないAIの回答は未確認のままにし、調査した事実として扱いません。

観点間で判断が食い違う場合は、それぞれの判断と根拠、残る未確認事項、判断が必要な役割を分けて示します。人の判断を得る前に一つの仕様へ自動統合しない規則です。質問の進め方は既存のdeepをそのまま使い、一度に最大4問です。利用者が終了を選んだら質問を止め、残る未確認の論点を Open Questions に記録します。

体験設計のフレームは任意の整理手段です。採用しなくても接点、失敗、裏側の支援を確認でき、フレームの存在や採用を観点別レビューの前提にしない設計です。画面の情報の優先順位、画面間の移動、レイアウト、見た目はここでは決めず、後続の画面設計で扱います。

## 体験設計のフレーム候補

`/intent-discover` は、案件に体験を設計する観点が合う場合だけ、既存のロールレンズを手がかりに確立したフレームの候補を示します。候補は、ペルソナ、Jobs to be Done、カスタマージャーニーマップ、サービスブループリント、ユーザーストーリーマッピングの5種です。案件との適合が明確なものに絞り、候補ごとに適合する理由を1行で添えます。体験を設計する必要がない案件や、適合が弱い案件では何も増やさず従来どおり進みます。

候補を見たあと、人が採用・見送り・保留を決めます。採用前には生成しないため、候補を見ただけでファイルが増えることはありません。採用した場合だけ、`.intent/nl-spec/design-frame-<frame-id>.md` に派生のMarkdown下書きを作ります。この下書きは推測を含み、再生成できる参考資料です。Intent Tree、Intent Compass、packetを自動では変更しません。

Compassでは「体験の約束」として、トーンと文体、アクセシビリティ、エラー時の姿勢、体感速度と処理中の状態提示の4観点を確認できます。特に、通常どおり進まないときに利用者を責めず、次の行動が分かるかを先に考えます。ここでも提案は候補であり、人が採用した内容だけが既存の分類を経て判断基準の候補になります。

この機能は実装前の事前検討に限定しています。画像や図は生成しない、アナリティクスで行動を実測しない、体験段階フィールドや段階ビューを追加しない、数値の優先度・日付コミット・進捗率を追加しない、という境界があります。

## enforcement（書き戻し漏れの検査・任意）

実装後の `/intent-writeback` を飛ばしたまま次へ進むと、`.intent/` は実装の現実から静かにズレていきます。enforcement は、この「書き戻しの実行漏れ」を機械的に検出する**任意**のレイヤーです。**既定は off** で、設定しない限り何も変わりません。

強度は3段階。`.intent/mode.md` の「Enforcement」セクションを直接編集して切り替えます。

| 値 | 動作 |
|---|---|
| `off`（既定） | 検査しない |
| `remind` | 漏れを検出したら警告のみ。止めない |
| `gate` | 漏れを検出したら export / push を止める |

検査されるのは主に「記録したまま承認・反映されずに残っている学び（pending delta）」です。push の関所が欲しければ、インストール時に `--enforce` を付けると pre-push フックも配置されます。

> enforcement が保証するのは「書き戻しという**手続きが実行されたこと**」だけで、書き戻した中身の正しさは保証しません（それは `/intent-improve` と人間レビューの仕事です）。誤検知が構造的に残るため既定は off にしています。止まっても、明示的に続行を指示すれば export は実行でき、push は `git push --no-verify` で通せます。

## drift-watch（ズレの監視・任意）

意図を立てても、実装を進めるほど当初の意図から少しずつ外れることがあります（AI が「分割して疎結合に」といった美徳に過剰適応して意図より複雑な構造を生むなど）。drift-watch は、この逸脱を**外れきる前に捕まえる**任意のレイヤーです。**既定は off**。`.intent/mode.md` の「Drift-watch」セクションを `on` にすると有効になります。

`on` のとき、3つの工程に軽いフックが差さります。

| 工程 | 何をするか |
|---|---|
| `/intent-discover` | 着手前に「逸脱しやすい場面」を型カタログと照合して名指しし、先に守るべき方向を書かせる（予防） |
| export 時 | 実装に渡す直前に、Compass の基準と照らして外れていれば警告する。渡した packet のスコープを超える実装指示が来ていないかも見る（水際） |
| `/intent-improve` | 節目に逸脱を記録し、効いた／効かなかったを集計したレポートを出す（事後） |

**いずれも警告のみで、止めません**（誤検知を前提とするため）。検知は `.intent/drift-log.md` にローカル記録されるだけで、外部に送信されることはありません。記録は「防げた／捕まえた」と「見逃した／誤検知だった」を対称に残す設計で、都合のいい記録だけ残す確証バイアスを構造的に避けます。

型カタログ（`.intent/drift-patterns.md`）は網羅ではなく、あなたが自分の現場で踏んだ逸脱を型として足して育てる前提のファイルです。

## 造語の管理（用語のブレを防ぐ・任意）

使い込むうちに、AI がチームで合意していない新しい言葉を増やすことがあります。同じ概念を別々の言葉で呼ぶようになると、意図のすり合わせが難しくなります。intent-planner は、この問題を**予防・検出・言い換え**の3段階で扱います。

- **正規語彙の台帳** — `.intent/glossary.md`（正しい用語＋表記ゆれ＋一行説明）に「これが正しい用語」を集約します。あなたが育てる台帳で、コマンドが勝手に書き換えることはありません（あなたが1語ずつ承認した内容だけ、代わりに書き足せます）。
- **語の状態（任意）** — 各行に「その語をチームがどこまで認めたか」を書けます。`承認済み`（チームで合意した共通語）／`暫定`（登録した人の暫定の語）／`否認済み`（採用しないと決めた語）の3つです。**状態を書かなくても従来どおり動きます**（書かれていない行は「暫定」として読みます。古い3列のままの台帳も、そのまま有効です）。否認した語は行を消さずに残すと、同じ語が後からまた作られたときに「一度やめた語の作り直し」として気づけます。登録も、暫定から承認済みへ上げるのも、**1語ずつ・一行説明を確かめてから**です（まとめて承認はできません）。
- **軽い検出と言い換え（term-drift が配置されていない場合）** — term-drift 配置物が一つもない環境では、`/intent-validate` が従来どおり用語台帳にない語を「造語の疑い」として示し、既存用語への言い換え案を添えます。これは警告だけで、作業を止めません。固有名詞、一般的な英語の技術用語、初出時に説明された正当な新語は対象外です。実際に書き換えるのは、あなたが承認したあとだけです。
- **予防** — 配布される規約文書（CLAUDE.md / AGENTS.md）に「台帳に無い新語を勝手に作らない／作るなら初出に一行説明を付ける」規律が含まれます。加えて、**利用者へ質問を出す直前**の点検（それ単体で意味が通るか・専門用語を詰め込みすぎていないか・識別子に言い換えを添えたか）と、**下流の spec ツールへ渡す下書きへの同梱**（「ここから作る文書・質問は初見に通じる言葉で書く」）も入ります。指示は文脈が深くなるほど効きが薄れるので、予防は必ず事後の検査と対で置きます。
- **すでに造語が増えてしまった場合（標準で同梱される別ツール）** — [term-drift](https://github.com/ijust/term-drift) は、台帳にない新語だけでなく、「配線」のような一般語を内輪の意味で使っている箇所も見つけます。書き換えるのは、**あなたが1語ずつ確認して承認した箇所だけ**です。

  term-drift 0.3.0 は、intent-planner がバージョンを固定して依存する npm パッケージとして標準導入されます。通常のセットアップでは、選択したエージェントの情報を term-drift の公式 installer または公式 update に渡し、`./.term-drift/` と専用スキルをプロジェクト内に配置します。term-drift 0.3.0 では、承認済みの一般語の分類を既存の用語台帳に保存し、別のセッションでも再利用できます。ただし、同じ表記を内輪の意味で使っている箇所や、曖昧な文章まで承認済みになるわけではありません。

  配置物が見つかったとき、`/intent-validate` は用語や互換性を独自判定せず、`npx intent-planner . --agent <選択中のagent> --dry-run` で通常の installer が示す導入状態と予定処理を確認するよう案内します。`ready` になった後、本格的な用語点検は term-drift 専用 skillから始めます。この専用経路は `/intent-validate` の構造検査を置き換えません。

## 制約の叩き台（定石の供給と蓄積・任意）

`/intent-compass` で Anti-direction / Invariant（守りたいこと・やってほしくないこと）を毎回ゼロから書くのは骨が折れます。intent-planner は、ドメインの定石（「SQL は必ずプレースホルダで組む」「スライドは結論から書く」など）を**叩き台の候補**として手前で差し出し、あなたが採用したものを次回も使えるよう貯められます。これは判断基準づくり（C2）を**置き換えるものではなく、その前段に候補を足す補助**です。

- **候補の提示** — `/intent-compass` の導出の前に、同梱の定石カタログ（`.intent/constraint-starters.md`・出典付き）と、あなたが育てた台帳（`.intent/constraint-library.md`）を読み、いまの案件に合いそうな定石を「叩き台としてどうですか」と候補提示します。**compass へ自動では書き込みません**。採否はあなたが決め、採用したものだけを人の手で取り込みます（合わなければ黙ります）。
- **採用したら台帳へ貯める** — 候補のうち「これは自分の定番だ」と採用した制約は、個人台帳 `.intent/constraint-library.md` へ追記しておくと、次の案件で同梱カタログと並んで候補に出ます（あなただけの定石が育っていきます）。**追記はあなたの手か明示の承認のもとだけ**で、勝手に蓄積はしません。台帳が無い・何も採用しない場合は黙って何もしません。
- **このプロジェクトの中だけに閉じます** — 貯めた制約は、そのプロジェクトの `.intent/` の中だけに置かれます。プロジェクトをまたいだ共有や、リポジトリの外への持ち出しはしません（機密案件の制約が外へ漏れないように、横断蓄積はあえて持たない設計です）。個人台帳は再インストールでも上書きされません。

## cc-sdd / OpenSpec への引き継ぎ

intent-planner が作るのは**下書きまで**です。spec の本体は下流の実装ツールが生成し、各フェーズであなたがレビューします。

**cc-sdd** — 配置先に cc-sdd（`.kiro/`）があるとインストーラが検出して案内します。`/intent-export-cc-sdd` が生成した下書きを cc-sdd の `/kiro-spec-init` に渡すと、整理した意図が requirements → design → tasks のフローへそのまま流れます。不変則と上位の意図は tasks へ転記されやすい形で渡るので、実装段階でも全体意図が効き続けます。

**OpenSpec** — もう1つの出口として [OpenSpec](https://github.com/Fission-AI/OpenSpec) に対応します。`/intent-export-openspec` は選んだ packet を proposal 下書き（Why / What Changes / Impact）＋ delta spec のヒントに変換し、続行を指示すると `/opsx:propose` を起動して change-proposal フローへ橋渡しします。cc-sdd 版と同じく、入力は対象 packet 1つ＋compass に限定する低コスト契約で、起動までで止まります（spec 本体の完成は OpenSpec 側に委ねます）。

## 自動で回す（`/loop`）ときの注意

intent-planner の工程は、人が `/intent-status` で次の一手を受け取って進めるのが既定ですが、`/loop` のようなハーネスに駆動を委ねて自走させることもできます。

ただし、文書を書き換えるコマンド（discover / compass / packets / writeback / improve / export）は**意図的に人間の承認を前提**にしています。無監視で走り続ける開発（vibe coding）への歯止めとして承認ゲートを置いているからです。`/loop` で承認を飛ばすと、速さと引き換えに次を失います。

- **ズレに気づく機会** — 各段で人が成果物を見て「意図とズレている」と気づく機会が減る
- **文書の保護** — 誤った学びがそのまま意図文書に反映されうる
- **重い分岐のレビュー** — 人が確定すべき問いを承認なしで埋めると、推測が確定として固定されうる

**おすすめはハイブリッド**です。内側の「実装→テスト→修正」は `/loop` に委ねて速く回し、段の切れ目（compass 確定・packet 分解・writeback 承認・重い問いの確定）では人が割り込む。読むだけのコマンド（status / validate / overview）は承認不要なので、`/loop` の中でも安全に判断材料を出し続けます。「全部を承認レスで回す」より「効く一点に承認を集中する」のが、軽さと安全を両立させる使い方です。

## インストールのオプション

```bash
npx intent-planner                       # カレントディレクトリへ
npx intent-planner ./my-project          # 指定ディレクトリへ
npx intent-planner --dry-run             # 何が起きるか先に確認
npx intent-planner --lang en --agent codex   # 英語 + Codex
npx intent-planner --enforce             # pre-push フックも配置
npx intent-planner --agent codex  # Codex向けintent-plannerとterm-driftを標準配置
```

| オプション | 説明 |
|---|---|
| `dir` | 配置先ディレクトリ（既定: カレント） |
| `--force` | 同名ファイルがあっても全て上書きする（`.intent/` のあなたのデータも失われる。対話環境では実行前に確認。既定: スキップ。ルート文書は追記） |
| `--update-shared` | 共有ファイル（CLAUDE.md / AGENTS.md / GEMINI.md / pre-push）も配布版へ更新する（上書き前に `<ファイル>.bak` へ退避。`.intent/` のデータには触れない） |
| `--dry-run` | 書き込まず、配置/スキップ予定の一覧だけ表示する（全一覧を表示） |
| `--verbose` | 配置/スキップしたファイルを1件ずつ全て列挙する（既定は件数サマリのみ） |
| `--lang <value>` | 言語指定: `ja`（既定）/ `en` |
| `--agent <value>` | 対象エージェント: `claude`（既定）/ `codex` / `gemini` |
| `--enforce` | pre-push フックを配置する（既定: 配置しない） |
| `--with-term-drift` | term-drift 0.3.0 は標準配置されるため、現在は指定不要。既存スクリプトとの互換性のため受け付ける旧オプション |
| `--yes`, `-y` | 既存ルート文書への追記確認を省いて同意する（非対話環境では既定でスキップ） |
| `--help`, `-h` | ヘルプを表示する |

配置されるもの:

```
.claude/skills/intent-*/   スラッシュコマンドの実体（--agent codex / gemini の場合は .agents/skills/ + AGENTS.md / GEMINI.md）
.intent/                   Intent Tree / Compass / Packets / deltas / modes などの雛形
CLAUDE.md / AGENTS.md / GEMINI.md  使い方を AI に教える薄い入口
```

既存のルート文書（`CLAUDE.md` / `AGENTS.md` / `GEMINI.md`）があっても**上書きはせず**、確認のうえ**非破壊で追記**します（既存内容のバイト列は変更しません）。Claude Code / Gemini CLI は quickstart 本体を別ファイル（`CLAUDE_intent.md` / `GEMINI_intent.md`）に置き、ルート文書へ参照1行（`@CLAUDE_intent.md` / `@./GEMINI_intent.md`）を足します（再帰 import で読み込まれます）。Codex は `AGENTS.md` の末尾に quickstart 節を直接追記します（Codex は `@import` 記法を持たないため）。再実行しても重複追記しません（冪等）。非対話環境（CI 等）では追記を見送り案内に留めます。`--yes` で同意を前渡しできます。

term-drift 0.3.0 は intent-planner がバージョンを固定して依存する npm パッケージで、通常のセットアップに含まれます。対象のリポジトリとエージェントが決まると、依存パッケージ内の公式処理を起動し、`./.term-drift/` とエージェント専用スキルを配置します。`--yes` は従来どおりルート文書への追記確認だけを省略するもので、term-drift の配置条件には影響しません。`--dry-run` では公式処理を起動せず、term-drift の版、対象エージェント、予定している処理（`install` / `update` / no-op）と、その処理を実行または見送る理由を表示します。

導入状態は `not-installed`（未導入）、`ready`（版、rules、選択したエージェント用のスキルが互換な一式）、`inconsistent`（一部だけ導入されている、または内容に不整合がある）のいずれかです。`inconsistent` はさらに、安全に不足分だけを追加できる `additive-compatible`、互換性を確認済みの 0.2.3 / 0.2.5 など公式 update を試せる `update-attemptable`、自動処理しない `blocked` に分かれます。`install-failed` はその回の公式処理が失敗したことを示す結果で、現在の配置状態とは別に表示されます。失敗後も、同じ検査方法で現在の状態を改めて確認します。`ready` なら再導入せず、選択中のエージェントの term-drift 専用スキルから用語点検を始められます。`blocked` の場合は、再実行を勧める前に問題箇所を示します。

新規導入と安全な追加は term-drift 0.3.0 の公式 installer、対応済みの旧版からの更新は公式 update に任せます。公式処理が安全上の理由で拒否した場合、intent-planner は処理後の状態を表示しますが、独自に巻き戻すことはしません。`ready` なら何もせず、未知の版や 0.3.0 より新しい将来版にも自動追随しません。term-drift 所有の rules、skill、glossary を intent-planner が独自に修復、上書き、削除することもありません。

## 分割形式の compass への移行

新規配置では `.intent/compass/` の分割形式を参照します。既存プロジェクトは、必要に応じて [移行ガイド](migration.md) の手順で移行できます。移行は任意で、従来の単一ファイル形式も引き続き読み取れます。
