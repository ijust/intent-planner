# ⏱ 10分ウォークスルー — インストールから「実装直前」まで

**このページの約束**: 約10分で、intent-planner のインストールから「実装に渡す下書き」ができるまでを、**実際の画面と生成物**を見ながら一巡します。題材は「ログイン機能を作りたい」。読み終えたら、自分のプロジェクトで discover → compass → packets → export の一巡を、次に何を打つか迷わず自走できます。

> **正直な注記**: このページの端末出力・生成物は、すべて実際の実行から採取したものです（作り物のサンプルはありません）。AI の応答（質問文・言い回し）は毎回変わります — 流れと成果物の形だけを頼りにしてください。所要時間はこの実走の実測で、AI の生成時間は合計約3〜4分でした。あなたが確認・回答する時間を含めた目安が「約10分」です。

## 前提

- Claude Code（Codex / Gemini CLI でも同じ流れ。`--agent` で選ぶ）
- Node.js（インストーラの実行だけに使う）

## Step 0 — インストール（1分弱）

プロジェクトの直下で:

```bash
npx intent-planner
```

実際の出力（この実走からの抜粋。末尾の応援のお願い2行だけ省いています）:

```
新規配置しました (149):
  (ファイル1件ずつの一覧は --verbose を付けると表示されます)

.gitignore を作成しました (.intent/cc-sdd/ の下書きを Git 非追跡化)

配置エージェント: claude
  skill: .claude/skills/intent-*/
  ルート doc: CLAUDE.md を配置しました。

次にやること:
  1. Claude Code を開く
  2. プロンプトに /intent-discover と入力して実行する（意図の詰めがここから始まります）
```

案内のとおり、Claude Code を開いて次へ。

> ちなみに: 既に導入済みのプロジェクトで再実行すると、案内は「続きから再開する: `/intent-status` を実行する」に変わります。途中まで進めた作業は上書きされません。

## Step 1 — `/intent-discover`（この実走では AI 生成 約1分 + あなたの回答時間）

プロンプトにこう打ちます:

```
/intent-discover ログイン機能を作りたい
```

AI がまず「意図の詰め方モード」を推奨し、確認してきます（実走からの抜粋・言い回しは毎回変わります）:

> 新規機能で意図がまだ言葉になっていないので、標準モード（standard）を推奨します。このモードでよいですか？
> あわせて、内容を詰める質問をしてよいですか？（受ける=でき上がりの正確性が増す／飛ばす=速度が上がる）

回答すると、AI が目的・成功の捉え方を推論して**追認**を求めてきます:

> 目的をこう捉えました — 「利用者が自分のアカウントで安全にログインし、自分のデータに続きからアクセスできる」。想定ユーザーは「週数回訪れる一般利用者（パスワードを忘れがち）」。この捉え方で合っていますか？

「合っている」と答えると、`.intent/intent-tree.md` に意図の階層（Intent Tree）が書かれます。実際の生成物の抜粋:

```markdown
## L0: Product Purpose
- 利用者が自分のアカウントで安全にログインし、自分のデータに続きからアクセスできるようにする

## L2: Capabilities
- C1 メール+パスワード認証 / C2 外部アカウント連携（OAuth） / C3 セッション管理

## L4: Candidate Packets
- P1 セッション管理の基盤（C3 を支える。P2/P3 が乗る土台）
- P2 OAuth 連携 / P3 ログイン UI

## Open Questions
- パスワードレス（メールリンク認証）を将来入れるか
- セッションの妥当な期間（数時間か数週間か）
```

ポイントは3つ。**確定と推測が分かれている**（推測は Assumptions 欄へ）、**決めていないことが Open Questions として残る**（推測で埋めない）、**作業単位の候補（P1〜P3）に土台からの順序がある**。

## Step 2 — `/intent-compass`（この実走では AI 生成 約30秒 + 確認）

判断基準を作ります。いちばんの主役は **Anti-direction（避けるべき局所最適）** — 「AI がやりがちな脱線」を先回りで名指しします。実際の生成物の抜粋:

```markdown
## Anti-direction
1. 「ついでに」ユーザーテーブル全体を再設計する — 今回の意図はログインであって
   会員情報の再モデリングではない。気づいた課題は Open Questions へ。
2. セッション ID を URL・localStorage に置く近道 — httpOnly cookie の意図を
   実装の都合で崩さない。
4. テストなしのログイン分岐追加 — 認証はエラー系（失敗・期限切れ・多重ログイン）が
   本体。ハッピーパスだけの done を作らない。
```

守るべき不変条件（パスワード平文禁止など）と、迷ったときの判断ルール（セッションはサーバ側 + httpOnly cookie）もここに固まります。実装中の AI はこのページの**関係する項目だけ**を読みます（全文は読みません）。

## Step 3 — `/intent-packets`（この実走では AI 生成 約30秒 + 確認）

意図を「実装に渡せる作業単位（packet）」に切り出します。この実走では **P1 セッション管理の基盤**が最初の1個として起案され、こう推薦されました:

> 最初に着手すべき packet は「セッション管理の基盤」。P2（OAuth）も P3（UI）も、ログイン状態を保持する土台が無いと動きません。入力→処理→出力を最初に貫く1本です。

packet には「やること／やらないこと／完了後に観測できる振る舞い／守る不変条件／決めていないこと（理由と再訪条件つき）」が入ります。**決めていないことを決めたことのように書かない**のが肝です。

## Step 4 — `/intent-export-cc-sdd`（この実走では AI 生成 約30秒）

P1 を下流の spec ツール（cc-sdd）へ渡す下書きに変換します。実際の生成物（`.intent/cc-sdd/セッション管理の基盤/requirements.md`）の冒頭:

```markdown
# Project Description（cc-sdd /kiro-spec-init へ渡す下書き）

ログイン状態の保持・失効の土台を作る。正しい資格情報でログインするとセッションが
発行され、以後のリクエストがログイン状態として通る。…

## Invariants
- INV-1: パスワードを平文で保存・ログ出力しない（MUST NOT）
- INV-2: セッション ID を URL に載せない。cookie は httpOnly / Secure（MUST）

## Acceptance Material
- 発行→検証→失効の一連が観測できること（MUST）: ログアウト・期限切れ・
  不正 cookie の3異常系で 401 が返る。
```

cc-sdd を導入していれば、この下書きをそのまま `/kiro-spec-init` に渡して requirements → design → tasks → 実装へ進めます。cc-sdd を使わない場合も、この下書き自体が「実装 AI に渡す指示書」として読めます。

## そのあと

- 実装が一巡したら `/intent-writeback` — 実装で得た学びを意図の文書へ戻します（ここまでが1周）。
- **迷ったらいつでも `/intent-status`** — いまどこにいて、次に何を打てばいいかを1つだけ教えてくれます。

## この実走の所要時間（実測）

| 工程 | AI の生成時間（実測） |
|---|---|
| インストール | 1分弱 |
| discover（Intent Tree） | 約1分 |
| compass（判断基準） | 約30秒 |
| packets（作業単位） | 約30秒 |
| export（下書き） | 約30秒 |
| **合計** | **約3〜4分**（あなたの確認・回答を含めた目安は約10分） |

## 最短コース（さらに軽く一巡したいとき）

新しい機構は不要で、**既存の選択肢を軽い側に倒すだけ**で短くなります:

1. discover の最初の確認で「質問を飛ばす（off）＝速度優先」を選ぶ（詰めの質問がなくなる）
2. packets では**最初の1個だけ**を確定して先へ進む（残りは候補のまま）
3. そのまま `/intent-export-cc-sdd` へ一直線

正確性より速度を優先する選び方なので、大事な案件では既定（質問を受ける）に戻すのがおすすめです。
