# ⏱ 10分ウォークスルー — `intent-plan`で「実装直前」まで

このページでは、intent-planner のインストールから「実装に渡す下書き」ができるまでを、**実際の画面と生成物**を見ながら約10分で体験します。題材は「ログイン機能を作りたい」です。入力する工程名は `intent-plan` の1回だけで、途中は人が決める必要のある点でのみ確認します。

> **所要時間について**: このページの端末出力と生成物は、実際の実行結果から抜粋しています。AI の質問文や言い回しは実行のたびに変わるため、同じ文面ではなく、流れと成果物の形を確認してください。この実行では、AI の生成に合計約3〜4分かかりました。内容の確認と回答を含めた全体の目安が約10分です。

## 前提

- Claude Code（Codex / Gemini CLI でも同じ流れ。`--agent` で選ぶ）
- Node.js（インストーラの実行だけに使う）

## Step 0 — インストール（1分弱）

プロジェクトのルートで実行します。

```bash
npx intent-planner
```

実際の出力例です。連携ツール（term-drift / handoff-bridge）の状態表示と末尾の案内1行は省略しています。

```
結果: 新規 164 件・更新 0 件を配置しました (スキップ 0 件)／対応が要る項目はありません
新規配置しました (164):
  (ファイル1件ずつの一覧は --verbose を付けると表示されます)

.gitignore を作成しました (.intent/cc-sdd/ の下書きを Git 非追跡化)

配置エージェント: claude (skill: .claude/skills/intent-*/)
  ルート doc: CLAUDE.md を配置しました。

次にやること:
  1. Claude Code を開く
  2. プロンプトに /intent-plan と入力して実行する（必要な確認を挟みながら、計画から受け渡しまで進みます）
```

案内に従って Claude Code を開き、次へ進みます。

> すでに導入済みのプロジェクトで再実行すると、案内は「続きから再開する: `/intent-status` を実行する」に変わります。途中まで進めた作業は上書きされません。

## Step 1 — `/intent-plan`を1回実行する

プロンプトにこう打ちます:

```
/intent-plan ログイン機能を作りたい
```

`intent-plan` は、discover → compass → packets → export を順に進めます。各段階の成果物と安全性は従来と同じですが、次の工程名を毎回入力する必要はありません。AI がまず、意図を整理するためのモードを提案し、確認します。

> 新規機能で意図がまだ言葉になっていないので、標準モード（standard）を推奨します。このモードでよいですか？
> あわせて、内容を詰める質問をしてよいですか？（受ける=でき上がりの正確性が増す／飛ばす=速度が上がる）

回答すると、AI が目的と成功の捉え方を推測し、合っているかを確認します。

> 目的をこう捉えました — 「利用者が自分のアカウントで安全にログインし、自分のデータに続きからアクセスできる」。想定ユーザーは「週数回訪れる一般利用者（パスワードを忘れがち）」。この捉え方で合っていますか？

「合っている」と答えると、`.intent/intent-tree.md` に意図の階層（Intent Tree）が書かれます。実際の生成物の抜粋:

```markdown
## L0: Product Purpose
- 利用者が自分のアカウントで安全にログインし、自分のデータに続きからアクセスできるようにする

## L2: Capabilities
- C1 メール+パスワード認証 / C2 外部アカウント連携（OAuth） / C3 セッション管理

## L4: Candidate Packets
- P1 セッション管理の基盤（C3 を支える。P2/P3 が乗る土台）
- P2 OAuth 連携 / P3 ログイン UI

## Open Questions
- パスワードレス（メールリンク認証）を将来入れるか
- セッションの妥当な期間（数時間か数週間か）
```

ここでのポイントは3つです。

- **確定した内容と推測が分かれている** — 推測は Assumptions 欄に入ります。
- **未決事項が Open Questions に残る** — AI が推測で埋めることはありません。
- **作業単位の候補（P1〜P3）に順序がある** — 土台となる作業から並びます。

## Step 2 — 守る判断基準を確認する

discover の整理が終わると、`intent-plan` はそのまま判断基準づくりへ進みます。中心になるのは **Anti-direction（進んではいけない方向）**です。AI が選びそうでも、今回の目的には合わない方向をあらかじめ明示します。

```markdown
## Anti-direction
1. 「ついでに」ユーザーテーブル全体を再設計する — 今回の意図はログインであって
   会員情報の再モデリングではない。気づいた課題は Open Questions へ。
2. セッション ID を URL・localStorage に置く近道 — httpOnly cookie の意図を
   実装の都合で崩さない。
4. テストなしのログイン分岐追加 — 認証はエラー系（失敗・期限切れ・多重ログイン）が
   本体。ハッピーパスだけの done を作らない。
```

守るべき不変条件（パスワード平文禁止など）と、迷ったときの判断ルール（セッションはサーバ側 + httpOnly cookie）もここに固まります。実装中の AI はこのページの**関係する項目だけ**を読みます（全文は読みません）。

## Step 3 — 最初の作業単位を確認する

`intent-plan` は次に、意図を実装に渡せる作業単位（packet）に分けます。packet が複数あっても、既定で受け渡すのは最初に着手する1件だけです。この例では、最初の packet として **P1 セッション管理の基盤**が作られ、次のように推薦されました。

> 最初に着手すべき packet は「セッション管理の基盤」。P2（OAuth）も P3（UI）も、ログイン状態を保持する土台が無いと動きません。入力→処理→出力を最初に貫く1本です。

packet には、やること、やらないこと、完了後に確認できる振る舞い、守る不変条件、未決事項とその理由・見直し条件が入ります。**決めていないことを、決定済みのように書かない**のが大切です。

## Step 4 — 実装に渡す下書きを確認する

P1 の確認後、`intent-plan` は導入済みの出口を見て、最初の1件を下流の spec ツールへ渡す下書きに変換します。出口を1つに決められない場合だけ、どこへ渡すかを確認します。実際の生成物（`.intent/cc-sdd/セッション管理の基盤/requirements.md`）の冒頭:

```markdown
# Project Description（cc-sdd /kiro-spec-init へ渡す下書き）

ログイン状態の保持・失効の土台を作る。正しい資格情報でログインするとセッションが
発行され、以後のリクエストがログイン状態として通る。…

## Invariants
- INV-1: パスワードを平文で保存・ログ出力しない（MUST NOT）
- INV-2: セッション ID を URL に載せない。cookie は httpOnly / Secure（MUST）

## Acceptance Material
- 発行→検証→失効の一連が観測できること（MUST）: ログアウト・期限切れ・
  不正 cookie の3異常系で 401 が返る。
```

cc-sdd を導入していれば、この下書きをそのまま `/kiro-spec-init` に渡して requirements → design → tasks → 実装へ進めます。cc-sdd を使わない場合も、この下書き自体が「実装 AI に渡す指示書」として読めます。

## そのあと

- 実装が一巡したら `/intent-writeback` — 実装で得た学びを意図の文書へ戻します（ここまでが1周）。
- **迷ったらいつでも `/intent-status`** — いまどこにいて、次に何を打てばいいかを1つだけ教えてくれます。

## この手順を実際に行った際の所要時間（実測）

| 工程 | AI の生成時間（実測） |
|---|---|
| インストール | 1分弱 |
| discover（Intent Tree） | 約1分 |
| compass（判断基準） | 約30秒 |
| packets（作業単位） | 約30秒 |
| export（下書き） | 約30秒 |
| **合計** | **約3〜4分**（あなたの確認・回答を含めた目安は約10分） |

## 最短コース（さらに軽く一巡したいとき）

新しい仕組みを使わなくても、既存の選択肢から軽い進め方を選べば短縮できます。

1. discover の最初の確認で「質問を飛ばす（off）＝速度優先」を選ぶ（詰めの質問がなくなる）
2. packets では**最初の1個だけ**を確定して先へ進む（残りは候補のまま）
3. 出口が一意なら、`intent-plan` にそのまま下書き作成まで進めてもらう

正確性より速度を優先する選び方なので、大事な案件では既定（質問を受ける）に戻すのがおすすめです。
