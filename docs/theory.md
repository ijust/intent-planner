# intent-planner の理論的背景

intent-planner は、**Intent Driven Development（IDD、意図駆動開発）**を軽量に実践するための仕組みです。Intent Tree、Compass、packets に「なぜ作るのか」「何を作るのか」を記録し、discover → compass → packets → export → writeback という一定の工程に沿って進めます。これにより、実装手順ではなく意図が、次に作るものを決める状態を作ります。

進行段階（state）は記録しますが、状態を自動で進める仕組みは内蔵しません。次へ進める役割は、人のレビューや `/loop` のような外部の実行ループに任せます。それでも、次に何を作るかを決めるのは記録された意図です。詳しくは「[進捗の表現 — 状態は持つが、状態機械は持たない](#進捗の表現--状態は持つが状態機械は持たない)」で説明します。なお、ここでいう IDD は、ほかの分野で使われる同名の概念ではなく、一定の工程と軽量なスキル・ひな形によって意図駆動の開発を実現する intent-planner 独自の呼び方です。

intent-planner の各機能は、要求工学（Requirements Engineering）やソフトウェアアーキテクチャ研究で確立されてきた考え方を、AI コーディングエージェントとの協働向けに軽量化したものです。このドキュメントは「なぜこの手順なのか」を理論の側から説明します。

## Less instruction / clearer intent

intent-planner の目的は、指示を増やすことではありません。設計のずれや統合の手戻りが大きな損失につながる案件に、**最小十分な判断材料**を渡すことです。Invariant、Scope / Acceptance、Decision、Preference / Heuristic の4段階で拘束力を分ければ、実装方法を細かく命令しなくても、AI は越えてはいけない境界を判断できます。AI が合意済みの範囲内を自分で進め、範囲を越える変更だけを人へ戻す形を、**境界付き自律**と呼びます。

実装時に渡す情報も、同じ原則に従います。今回の判断に必要な確定情報と、実装中に必要になった場合だけ見直す候補を分け、関係のない Intent Tree、Compass、履歴を常にすべて渡すことはしません。必要な判断材料を残したまま、不要な指示を減らすためです。そのため、小さな試作や vibe coding で十分な案件では、`direct` を選ぶか、intent-planner の工程を省略できます。

進捗も、1つの総合判定にはまとめません。**工程の状態**、**未決の設計判断**、**利用者にもたらした成果**は、それぞれ別の情報です。工程が順調でも、狙った成果が出ているとは限りません。成果の証拠がなければ「未観測」と表示し、総合 PASS や点数は付けません。代わりに、人が次に判断すべきことを1つだけ前面に出します。

**読まなくても使えます。** intent-planner は、フローに沿って質問に答えていけば必要な成果物が埋まるように設計されており、理論の予習を前提にしていません。このドキュメントは、各ステップで何を書くべきか迷ったとき、あるいは手順の意図そのものを知りたいときの参照用です。

## なぜ「spec の手前」に1段挟むのか — 問題空間と解空間

要求工学では、**問題空間**（何を達成したいか・なぜか）と**解空間**（どう実現するか）を区別します。Michael Jackson の Problem Frames が示すように、要求はソフトウェア（machine）の側ではなく、それが作用する世界（world）の側に存在します。spec（requirements → design → tasks）は解空間へ向かって具体化していく工程であり、その入力となる「何を・なぜ」が曖昧なまま spec を書き始めると、**形式は整っているのに方向が違う仕様**ができあがります。`.intent/` は、問題空間の記述を解空間の記述（spec）から分離して保持するレイヤーです。

もう1つの動機は **architectural drift** です。Perry & Wolf（1992）は、設計者の意図したアーキテクチャから実装が徐々に逸れていく現象を drift / erosion として定式化しました。AI エージェントによる大量のコード生成はこの進行を加速させます。各変更は局所的には妥当でも、横断的な意図を参照しないまま積み重なると全体が崩れるからです。古典的な対策は意図を「参照可能な形」で外在化することであり、それを AI が毎回読める文書（steering context）として用意するのが intent-planner の中心的な仕掛けです。

## 損失付き射影 — 仕様・意図・コードは互いに情報を落とす

spec（仕様）・Intent（意図）・code（実装）は、同じ対象の異なる**射影**です。どの射影も対象の全情報を保持しません。各方向の変換で、必ず何かが落ちます。

自然言語の仕様書と構造化された意図を、**情報を失わずに完全に相互変換することはできません**。Gotel & Finkelstein（1994）は、要求トレーサビリティをあらゆる方向から完全に解決する方法はあり得ないと論じています。自然言語には曖昧さがあり、別の形式へ変換してから戻しても、元と同じ意味を保てる保証がないためです。また、正確に追跡するには細部まで確定する必要がありますが、開発が進む間に必要な情報そのものが失われることもあります。

さらに、仕様書を情報の欠落なくコードへ変換できるなら、その仕様書自体が、別の記法で書かれたプログラムだと言えます。実行可能な完全仕様は、最終的にはプログラミング言語と同じ役割を持つからです。仕様とコードの間にある差は、Brooks のいう**本質的な複雑さ**に属し、表現やツールの工夫だけでは消せません。

そのため intent-planner は、仕様を完全にするとは主張しません。代わりに、**変換の途中でどの情報が失われたかを示し、その情報が後続の設計を左右するものかどうかを見分けます**。情報の欠落自体は避けられません。防ぎたいのは、後続の正しさを支える重要な情報が、気づかれないまま失われることです。

落差は辺ごとに非対称です。

| 辺 | 落ちるもの | intent-planner の対応 |
|---|---|---|
| spec → Intent（取り込み） | 仕様書に書かれていない目的・不変条件・進んではいけない方向 | 外部仕様書や断片メモを基準と照らし、書かれていない項目を抜けとして示す（`/intent-from-spec`） |
| Intent → spec（生成） | Intent にない詳細を、生成時にもっともらしく補ってしまうこと | 各記述の根拠を元の文書まで追跡し、根拠のない記述には inferred（推測）の印を付ける（`/intent-to-spec`） |
| Intent → code | 実装が Intent の要求を満たさない・破った | drift として検出する（drift-watch / improve） |

完全性スキーマ（決定スロット）、不変則の固定カテゴリ、Requirements Smells、PBR の4つの観点は、失われた情報が設計を左右するかどうかを判断するための基準です。見つけた抜けは仮説として示し、最後に要点をまとめて人へ確認します。警告は出しますが、作業は止めません。これは「LLM は補完案を出すことには向いていても、完全性を最終判断する役には向かない」という方針の一例です。

ここには、要求工学上の重要な違いがあります。**書かれている文章の曖昧さや矛盾は文章内のパターンから見つけられますが、そもそも書かれていないことは、文章だけを読んでも見つけられません。** 抜けを見つけるには、ドメインモデルや品質特性の分類など、照合するための外部基準が必要です。完全性検査の研究でも、この点は繰り返し指摘されています。

固定カテゴリと完全性スキーマは、その外部基準を小さく保ったものです。重い知識ベースや形式化を持ち込まず、最終判断は人に残します。研究には「ある要求を外すと、最適なアーキテクチャの選択が変わるか」を定量的に調べる ARLO の AIR（architecturally influential requirements）があります。intent-planner はこの発想を、数理的な判定ではなく、決定スロットと固定カテゴリを使った定性的な確認に応用しています。

仕様書から Intent を取り込む処理と、Intent から仕様書を生成する処理は、同じ情報を逆方向に変換します。どちらも、目的・不変則・判断基準、実装全体に関わる制約、個別の要求を扱いますが、注意すべき点は逆です。

- **取り込み（`/intent-from-spec`）**では、仕様書に書かれていない不変則などの抜けを見つけることが中心です。
- **生成（`/intent-to-spec`）**では、元の Intent にない詳細を LLM がもっともらしく補わないようにすることが中心です。

確認方法が異なるため、2つは別のスキルにしています。取り込みでは固定カテゴリとの照合を使い、生成では生成物の各記述を元の文書まで追跡します。

取り込み元は仕様書だけではありません。仕様書が残っていない既存のコードベースには、`/intent-from-code` を使えます。このスキルはコードを読み取り専用で調べ、L0〜L3 に相当する意図の候補、不変条件の候補、コードからは分からない抜けを `.intent/code-ingest/` にまとめます。すべての候補に `inferred`（推測）の印と根拠を付け、コードに書かれていない意図を確定事項のように補うことを防ぎます。`/intent-from-spec` と同様に、出力は人が承認するまで仮説として扱います。

> **現在の実装範囲と、まだ検証していないこと**: 上表の3方向はすべて実装済みです。Intent → code のずれは drift-watch / improve、spec → Intent の抜けは `/intent-from-spec`、Intent → spec での根拠のない補完は `/intent-to-spec` が扱います。`/intent-from-spec` は、`/intent-discover` が対話で集める情報を文書から読み取る入口です。結果は確定事項ではなく Assumptions（仮説）として示し、最後に要点をまとめて人へ確認します。まとまった仕様書だけでなく、断片メモ、走り書き、音声の書き起こしも入力できます。その場合は、話題ごとにまとめ、「決まっていること」と「迷っていること」に分けてから抽出します。この分類にも推測の印を付け、本人のレビューまで確定しません。なお、仕様書から抜けていた不変則のうち、どの程度が後の実際の不具合に関係したかは、まだ測定していません。Git 履歴を使った測定方法を設計できる段階にとどまります。

生成先は、仕様書やコードに限りません。Intent の上流部分は出力形式に依存せず、プログラム固有の情報は packets と export の段階で加わります。そのため、文書・業務・研究などの案件にも、上流を作り直さず対応できます。

非プログラム案件では、`non-code` モードを選び、`/intent-to-spec` から `.intent/nl-spec/` に読める成果物を出します。これは、同じ Intent を仕様書やコード以外の成果物へ変換する経路です。

出力形式は読み手に応じても変えられます。`/intent-to-spec` の「ステークホルダー向けの一枚もの」と「定例報告」は、同じ素材を結論から読める順に並べ、Invariant、Decision Rule、packet id などの内部用語を普通の言葉へ言い換えます。「意思決定メモ」は、compass に記録された採用・不採用の判断、理由、見直し条件を先に示します。元の文書にない評価や決定は補いません。判断材料が compass に書かれていなければ生成せず、先に整理するよう案内します。

読み手が顧客、経営、営業など、リポジトリの内部事情を知らない人であれば、`scripts/symbol-labels.json` の対応表を使って内部記号を言い換えます。対応表に短い呼び名がない packet id やスキル名は、推測で訳さず本文から省きます。リポジトリの当事者向け文書では、追跡に必要な参照として記号を残します。この規則は1つのファイルにまとめ、各出力形式から共通して参照します。

さらに、読み手だけでなく、どこまで詳しく書くかも選べます。詳細度は `概要 / 標準 / 詳細` の3段階です。概要では packet の証拠欄まで、詳細では期待する振る舞い、制約、受入基準を含む packet 本文まで参照します。詳しい文書を作るときも、元の資料にない記述を足して量を増やしてはいけません。どの詳細度でも、記述の根拠を元の文書まで追跡します。詳細度が指定されていなければ、生成前に一度だけ確認します。一枚もの、定例報告、意思決定メモは、情報を絞ること自体が形式の目的なので、詳細度は固定です。

もう一つの出力先は **Git の履歴**です。`/intent-release-note` はコミット履歴を読み取り専用で調べ、各コミットを packet 名、上位の意図、deltas などと照合して、「何が変わったか」だけでなく「なぜ変えたか」を補います。出力形式には、一般的な変更履歴、GitHub Releases 向け、利用者への影響を先に示す顧客向け変更履歴、PR 説明の下書き、そして自分向けの変遷ノートがあります。PR 説明には、意図の要約、関係する判断基準、レビューの要点、コミットと意図の対応を含めます。変遷ノートは、中断をはさんで考えごとを再開する将来の自分に向けて、`.intent/` の変更履歴から「いつ・何を決めた・何を変えたか」を新しい順の時系列で辿り直す内向きの1枚です。理由が記録から辿れない変更は「理由の記録なし」と明示し、推測の動機で物語化しません。GitHub へは書き込まず、packet や compass の本文もそのまま外部へ転記しません。

Git の履歴には「何が変わったか」は残りますが、「なぜ、どの意図のために変えたか」は残らないことがあります。`/intent-release-note` は、その不足を intent と照合して補います。対応の判断方法は2つあります。

- **記録に基づく対応** — コミットメッセージ末尾に任意で付ける `Intent: <packet 名> (<packet_id>)` という Git trailer を使います。これはコミット作成者が明示した対応です。
- **推測による対応** — trailer がないコミットは、コミットメッセージと intent の文章を照合します。これはツールが後から推測した対応です。

この2つは確度が異なるため、出力でも区別します。文章の照合だけでは確信を持てない場合、断定せず候補として示します。intent と結び付けられないコミットも省かず、簡潔な行として残します。trailer は任意で、付け忘れを警告することはありません。記載するのは packet 名と packet_id だけにし、機密情報は含めません。Git に対して行うのは読み取りだけで、commit、tag、push は実行しません。なお、どの程度のコミットを正しく intent と対応付けられるかや、対応しない変更の割合とプロジェクトの健全性の関係は、まだ測定していません。

さらに、Intent から **DB 設計の叩き台**も作れます。`/intent-db-design` は、永続データモデルを扱う packet について、packet の意図、compass の不変則、既存のスキーマや migration を読み取り専用で確認します。そのうえで、テーブル定義、制約、インデックス、命名の根拠を元の文書まで追跡できる形で `.intent/db-design/` にまとめます。元の資料にない記述には `inferred`（推測）または `unverified`（未確認）の印を付けます。

この機能は、下流の仕様作成ツールが全体設計と細かなテーブル定義を同じ文書で扱う際に、DB 設計を推測で埋めてしまう問題を切り分けるためにあります。出力は要件ではなく設計の叩き台なので、requirements には混ぜません。起動は人が明示的に行い、`/intent-status` や `/intent-packets` が必要に応じてコマンドを案内するだけです。

叩き台は、正規化の崩れ、必要なインデックスの欠落、N+1 を招く設計、制約漏れ、命名の一貫性、不変則への適合、後戻りの難しさという観点で点検します。いずれも読み取り専用の警告で、処理を止めません。実装後は `/intent-validate` が叩き台と実際のスキーマの違いを示し、`/intent-writeback` が変更理由を delta に記録します。下流の設計へ取り込むことは強制しません。なお、叩き台が実装でどの程度参照されたかや、DB 固有の警告と後の不具合の関係は、まだ測定していません。

## Intent Tree — ゴール指向要求工学

L0–L4 の階層は、ゴール指向要求工学（GORE: Goal-Oriented Requirements Engineering）の**ゴール精化（goal refinement）**を簡略化したものです。KAOS や i* といった手法では、上位ゴール（why）を下位ゴール（what）へ、さらに実現手段（how）へと分解し、どの要素からも「なぜそれが存在するのか」を上位へ辿れるようにします。

| Intent Tree | ゴール指向要求工学での対応 |
|---|---|
| L0: Product Purpose | 最上位ゴール |
| L1: Desired Outcomes | 達成したい状態変化（outcome） |
| L2: Capabilities | ゴールを支える能力（機能要求の母体） |
| L3: Behavioral / Architectural Intents | 品質要求・アーキテクチャ上の制約 |
| L4: Candidate Packets | 実現タスクへの橋渡し |

「下から上へ why を辿れる」ことの実用上の意味は、変更時の判断にあります。ある packet を変更・削除するとき、その parent intent を見れば「何が崩れるか」を機械的に確認できます。`/intent-export-cc-sdd` が packet に上位 intent を添えて渡すのは、この追跡可能性（traceability）を実装段階まで届けるためです。

ただし、上位の意図を下書きへ添えるだけでは、下流の生成過程で一部が失われます。2026年7月に intent-planner 自身の36件の export と、その後に生成された仕様を照合したところ、下書きに含めた意図の参照のうち、requirements に残ったものは約3分の2でした。各タスクまで不変条件の参照が引き継がれた割合は、1割強でした。

主な原因は、packet の期待する振る舞いや適合基準がヒントファイルにだけ書かれ、下流の requirements 生成処理がそのファイルを自動では読まないことです。Project Description だけから受入基準を作ると、元の基準が薄まるか、新しい内容が補われてしまいます。

そこで、次の2点を下書きと案内に組み込みます。

1. requirements の下書きに `## Acceptance Material` を必須で設け、期待する振る舞いと適合基準を転記します。要求の強さは RFC 2119 に沿って明確にします。ここでは EARS 形式の完成した要求までは作らず、下流が受入基準を「転記して整える」ための材料を渡します。
2. design や tasks へ進むときは、該当するヒントファイルの本文も渡すよう案内します。下流ツールは自動でヒントを読まないため、利用者またはエージェントが明示的に渡します。渡されなくても処理は止めません。

これでも引き継ぎを保証できるわけではありません。intent-planner は、下書きと案内によって失われる確率を下げますが、下流ツール自体は変更しません。

Intent Tree が canonical（確定）と Assumptions（AI の推測）を分けるのは、要求の**出所（provenance）**の管理です。推測を確定した意図と混ぜると、後からどれを検証すべきか判別できなくなります。

なお KAOS のゴール木は本来**可変深度**で、「単一のエージェントに割り当て可能になるまで」といった停止基準で分解を続けます。L0–L4 という固定の層はその簡略化であり、**粒度の層ではなく役割（目的 / 成果 / 能力 / 設計意図）の層**です。木全体で粒度を揃えることは求めません — GORE の主流文献にもそのような規範はなく、揃えるべきは作業単位（packet）の側で、それは behavior-preserving / testable / rollbackable という操作的基準が担います。

## L1 の計測基準 — 計測できない要求は判定できない

Tom Gilb は、品質要求には尺度（scale）と測定方法（meter）を持たせるべきだと主張し、Planguage という記法に落とし込みました。Basili らの **GQM（Goal-Question-Metric）** も、ゴールを問いに、問いを指標に展開する同系統の手法です。

designer-questions が on のとき `/intent-discover` が各 L1 項目に「計測基準:」を求めるのは、この実践の最小形です。「使いやすくする」のような計測不能な outcome は、実装後に達成・未達成を判定できません。検証目的（PoC）の開発では特に致命的で、判定できない仮説は何も学びを生みません。

## Compass — 判断基準の外在化

### 発散では探索を強め、判断と実装では境界を強める

AI に同じ強さの制御を全工程でかける必要はありません。`/intent-discover` で解が発散しているときは、AI が仮説・その仮説を崩す反例・別の問題設定を暫定案として積極的に出します。ただし、これらは canonical ではなく Assumptions / Open Questions に留めます。次の `/intent-compass` で人が「何を許容し、何を除外し、何を不変とするか」を確定し、その境界が後続へ渡る判断基準になります。実装の自律性は広げず、確定済みの compass と packet の範囲に閉じる bounded autonomy とします。これは discover 単体が実装を制御する機構を持つという意味ではなく、discover が後続へ渡す設計原則です。

### Invariants — 保存される性質で正しさを定義する

Bertrand Meyer の**契約による設計（Design by Contract）**では、クラスは常に保たれるべき不変条件（invariant）を持ちます。これをシステムレベルへ持ち上げたものが、Ford らの **architectural fitness function**（『Building Evolutionary Architectures』）です。共通するのは「変更の正しさを、変更内容そのものではなく、**変更後も保存される性質**で定義する」という発想です。

intent-planner の Invariants は機械検証ではなく文書として AI に渡されますが、発想は同じです。AI に「この変更は良い変更か」を判断させる代わりに、「この性質は保たれているか」というチェック可能な形に判断を変換しています。

### Decision Rules — 軽量 ADR と設計根拠

Compass の Decision Rules（Context / Decision / Why / **Alternatives considered** / Consequences / **Revisit when** の6欄）は、Michael Nygard が2011年に提案した **ADR（Architecture Decision Record）** の軽量版です。さらに遡ると、設計上の決定を「何を選んだか」だけでなく「何と比較して・なぜ選んだか」ごと残すべきだという **design rationale** 研究（Rittel の wicked problems 論に始まる IBIS など）の系譜にあります。

- **Alternatives considered** は、Markdown ベース ADR の標準形である **MADR** が "considered options" として定式化した欄に対応します。不採用案とその理由を残すことで、同じ代替案が後から無自覚に再提案されるループを断ちます。
- **Revisit when**（見直し条件）は、決定を「永久に正しいもの」でなく**有効期限つきの仮説**として扱う欄です。Twin Peaks モデルが言う要求と設計の往復を、「どんな観測があればこの決定を再訪するか」という条件の形で前もって書き残します。`/intent-improve` がこの条件の成立を検出するのは、Living Documentation の「文書が読者に働きかける」発想の実装です。

判断記録が増えても、毎回その全件を関門にすると新しい変更の認知負荷が履歴量に比例します。そこで読み手は `active` かつ案件に関係する判断だけを JIT で選び、無関係・`superseded`・archive は除外します。関係が曖昧なら確認へ上げ、`Revisit when` の成立も「自動失効」ではなく再検討の材料として扱います。これは design rationale の履歴性を保ちながら、現在の判断に必要な working set だけを小さくする考え方です。

根拠を残さない決定は、後続の変更者 — 人間でも AI でも — に無自覚に覆されます。決定を意図的に覆すときは、新しいエントリを追加し、旧エントリへ superseded と後継参照を付けて `compass-archive.md` へ退避します（前述の保管構造の節を参照）。

### Agent Contract — 制約としての仕様

`.intent/execution-contract.md` に置く実行契約（読むべきもの・拘束力・境界を越えたときの判断形式）は、仕様を「生成の手順」ではなく**生成の境界**として渡す発想です。既存の置き場を Invariant / Scope・Acceptance / Decision / Preference・Heuristic の4段として読み、境界内の手段選択は AI に残します。価値ある案が境界を越える場合は、黙った棄却と無断実装のどちらにも倒さず、現設計維持・設計変更・次の packet という人の判断へ上げます。契約による設計が呼び出し側と実装側の責務を固定するように、LLM エージェントにも「どこまで自律し、どこから確認するか」を実行時だけ参照する短い契約として外在化します。

### Anti-direction — 「やらないこと」の明文化

ゴール指向要求工学の **obstacle analysis**（ゴール達成を阻害するシナリオの分析）や、スコープ定義における non-goals の慣行に対応します。生成 AI は「もっともらしいが全体意図に反する局所最適」を選びやすいため、進んではいけない方向を明示しておく価値は、人間のチーム相手よりさらに大きくなります。

### 使い方の伝え方 — 人には流れで、AI には規約で伝える

使い方の伝え方は、相手が人か AI コーディングツールかで異なります。人に対しては、理論を先に教えるのではなく、流れに沿って質問に答えれば必要な仕様が整理される形を目指します。一方、AI には、起動時に読む規約文書を通して、作業の流れ、入口、最低限のルールを明示します。この2つは対象が異なるため、両立します。

### 必要な情報だけを、その都度読む

AI に使い方を明示することと、仕様や不変則の全文を常に読み込ませることは別です。すべてを常時読み込むと、責務が増えるほど固定の読み込み量も増え、使えるコンテキストを圧迫します。そのため、規約文書には存在、作業の流れ、入口、最低限のルールだけを置きます。仕様や不変則の本文は、実装の直前に関係する範囲だけを読みます。作業単位ごとに必要な制約だけを渡す設計です。

この短い入口は、AI ツールごとのルート規約文書として配布されます。Claude Code では `CLAUDE.md`、Codex では `AGENTS.md`、Gemini CLI では `GEMINI.md` です。新しい AI ツールへ対応するときは、インストーラのエージェント一覧に配置先とルート文書を1件追加します。スキル、ひな形、ルート文書の配置処理は共通化しているため、ツールごとに別の分岐を増やす必要はありません。Gemini CLI は `.agents/skills/` を読むため、Codex と同じ配置先を共有します。

### サービスデザインの手法も、候補としてだけ提示する

技術的な定石と同様に、サービスデザインの確立した手法も読み取り専用で照合し、候補として提示できます。案件の性質とカタログに書かれた適用条件を内容で照らし合わせ、名前が似ているだけの手法は候補にしません。

適合が弱い場合は候補を提示せず、そのまま沈黙します。提示された候補については、人が採否を決めるため、採用・見送り・保留のどれかを選びます。採用前には何も生成しないため、候補を見ただけでファイルは増えません。採用後だけ `.intent/nl-spec/design-frame-<frame-id>.md` に派生の下書きを作ります。Intent Tree、Intent Compass、packet の正本は変更しないため、候補を示しただけで設計判断が確定することはありません。

画像や図、アナリティクス、体験段階、数値の優先度、日付コミット、進捗率は対象外です。画像や図は生成せず、アナリティクスで行動を実測せず、体験段階や管理用の数値も追加しません。確立したフレームは問いを補助する道具であり、プロジェクト管理や行動計測の仕組みへ責務を広げません。

## 発問設計 — 聞くべきことを想起させ、見落としを補正する

intent-planner は「推論で埋める」前に「ユーザーに聞く」発問を各フェーズに束ねて挟みます（compass の Invariant 収集、discover の目的・成功・想定ユーザーの追認、packets の優先順位）。この聞き方 — 何を・どの順で・どんな言い回しで聞くか — は、要求工学・認知科学・LLM 応用研究の交差点から設計しています。

### 固定カテゴリ枠 — 網羅性は安定した枠で担保する

要求抽出には古くから「カテゴリ枠を先に置く」発想があり、Volere のチェックリスト、ISO/IEC 25010 の品質特性、NFR Framework、SQUARE などが確立しています。compass が Invariant・制約を「データ/個人情報・外部依存・運用/障害時・セキュリティ/プライバシー/法令・性能/可用性・技術的制約・不変条件」という固定カテゴリ枠で毎回聞くのは、この「抜け漏れ防止の外骨格」を最小化したものです。

構造化された問いが自由な聞き取りより多くの要求を引き出すことには実証があります。Browne & Rogich は prompting technique の比較実験で構造化された問いがより多くの要求を引き出すことを、Burnay らは6つの context category を elicitation checklist として使うと意思決定に有意な影響が出ることを示しました。重要度の高いカテゴリ（データ/個人情報・外部依存）を先に提示するのは、認知負荷の制御です。

正直な注記: Volere や ISO 25010 といった**個別名義の手法そのもの**が欠落を何%減らすか、という head-to-head の強い実証は乏しいままです。Carrizo らのレビューでも、技法効果を比較した実証研究はごく少数でした。実証が相対的に強いのは手法名よりも「カテゴリ化された問い・決定木・トリガ質問」という構造化発問の側であり、固定カテゴリ枠は「枠を見せれば自動的に網羅される」ものではありません。だからこそ枠を単なる見出しに留めず、各カテゴリで具体的に問う設計にしています。

このカテゴリに技術的制約を含めるのは、外から課された条件と、実装時に選ぶ方法を区別するためです。この文書では、前者を**要求としての How**、後者を**解としての How**と呼びます。

会社標準、契約、規制によって「使う必要がある」「使ってはいけない」と定められた技術、基盤、ライセンスは、実装者が自由に選べるものではありません。そのため、上位の Invariant として早い段階で確定します。一方、「キャッシュを使って速くする」のような実装方針は、cc-sdd へ委譲します。

Michael Jackson の Problem Frames では、要求を、実装する機械そのものではなく、機械が働きかける現実世界の側に置きます。したがって、外から課された技術的制約は **world 側＝problem space に属します**。この考え方に沿えば、技術的制約を上流で確認しても、実装方法を先に決めることにはなりません。分ける基準は、**破ると外的に問題になる制約か（＝要求 How）／実装の都合で選んだだけか（＝解 How）**です（[cc-sdd 連携](#cc-sdd-連携--仕様駆動開発と-ears)、[問題空間と解空間](#なぜspec-の手前に1段挟むのか--問題空間と解空間)を参照）。

### 動的例示と fixation の回避 — 例は手がかりであって答えの空間ではない

カテゴリの例示を scaffold に固定文字列で埋め込まず、プロジェクト文脈（技術スタック・ドメイン・既存コード）から動的生成するのは、認知科学の **encoding specificity**（Tulving & Thomson）に基づきます。手がかりが有効なのは、それが記憶の符号化文脈と一致する場合であり、文脈に合わない一般的な強い手がかりは自由想起より逆効果になりうる、というのが要点です。雑な一般例を外から与えるより、本人の文脈に沿う手がかりのほうが想起を助けます。

例示を「1個の強い代表例」でなく「性質の異なる2〜3個の弱い手がかり」とし、必ず「これは網羅ではない、他にあれば」と非網羅を明示するのは、**fixation** への対策です。Jansson & Smith の設計固着研究以来、提示された例に思考が引っ張られて他の解を探索しなくなる現象が繰り返し示されてきました。ソフトウェア工学でも、要求の提示の仕方が後続の設計・テストの探索空間を狭めることが問題化されています。文脈から例を生成できない場合にカテゴリの枠（見出し）だけを出す fallback も、誤った例で誘導しないための同じ配慮です。

### 解決方法がほぼ決まっている場合は、先に確認する

前節では、外部から課された技術的制約と、実装上の選択を区別しました。ただし、実装方法が常に複数あるとは限りません。定期実行なら cron、コマンドとして提供するなら CLI など、要望そのものから構成がほぼ決まる場合もあります。`/intent-discover` は、要望を受け取った時点で、解決方法がほぼ1つに決まるか、複数の妥当な案があるかを文章の意味から判断します。判断できなければ推測で確定せず、Open Questions に残します。

解決方法がほぼ決まっている場合は、中立な選択肢を並べるより、「目指す構成はこれでよいですか」と1問で確認するほうが早く、間違っていてもその場で訂正できます。これは、目的・成功・想定利用者を推測してから確認する discover の進め方と同じです。確認した構成は記録するだけで、discover の段階では実装やコード生成を行いません。

アンカリングを避けるために既定値を出さない規律は、複数の妥当な案がある設計判断で使います。選択肢が複数あるときは、最初に示した案が「推奨」や「標準」として無批判に採用されないよう、中立に質問します。一方、要望から解決方法がほぼ決まっている場合まで案を伏せると、確認に余計な時間がかかります。アンカリングへの対策は、複数案がある場面に限定して使います。

### 否定形の発問 — 暗黙の前提を失敗シナリオから掘り起こす

Invariant を「〜は必要か」という肯定形ではなく「これを完全に無視したら最悪何が起きるか」「外部依存が落ちても守るべきことは何か」「絶対に消してはいけないデータは何か」という失敗前提・否定形で問うのは、Burnay らの **default requirements**（ステークホルダーが当然視して語らない暗黙の前提）研究に対応します。「絶対に壊してはいけない制約」「運用上の不文律」は、肯定形の確認では思考停止の追認になりがちで、損失シナリオから逆算したほうが想起されやすいためです。

「実際の要求はインタビュー後にアナリストが補い、特に security と privacy が後から追加されていた」という How do requirements evolve during elicitation? の実証は、これらが自由な会話だけでは自発的に想起されにくいことを示します。否定形で得た回答のうち**真に守るべきものだけ**を Invariant 化し過剰な前提を混ぜないのは、掘り起こしと過剰生成は表裏だからです。

### omission recap — LLM の補完は提案には向くが「完全」の判定には向かない

発問の最後に、収集した制約・非機能要件・不変条件を短く要約して返し「抜け・過剰はないか」を1回確認する omission recap を置きます。これは LLM を発問・補完に使うことの限界に対する防御策です。近年の LLM4RE 研究は、LLM が同じ prompt でも出力が一貫しないこと、keywords や personas が scope を狭めて重要な観点を落としうること、hallucination が信頼性を損なうことを指摘しています。

ここから導かれる役割分担は明確です。LLM の推論補完は「質問案・抜け漏れ案の提案」には向きますが「これで完全だ」と判定する役には向きません。だから intent-planner は機能要件を全面的な能動質問へは振らず「推論+確認」の哲学を維持しつつ、確認を受動的な追認で終わらせないために recap を機能要件（discover の L0–L3）にも適用します。第一パスで質問・収集、最後のパスで要約と欠落検査、という役割分離は LLM 向けの prompt engineering 研究とも整合します。

「すべてを質問で決めない」という方針は、既定の `question-depth=standard` に適用します。確認を増やしたい利用者は、**deep（深掘り）**を明示的に選べます。deep では、機能の内容、前提、例外、反例、非機能要件まで質問の範囲を広げます。標準の動作は変わりません。

deep でも、次の歯止めを保ちます。

- 1回に尋ねるのは最大4問
- 各問いで「後で確認」「不明」「該当なし」を選べる
- 質問の途中で終了できる
- 尋問のような言い方をしない
- 既定値を先に示して回答を誘導しない

つまり、deep が広げるのは質問の範囲であって、回答を迫る強さではありません。

質問を始める前には、仕組みの説明を長く並べず、「質問をしてよいか」を直接確認します。質問を受けると意図との食い違いを減らせます。飛ばすと開発は速くなりますが、AI が利用者の想定と異なるものを作る可能性が残ります。この違いを示したうえで、利用者が選べるようにします。

deep の終了を固定した往復回数で決めると、必要な論点が多い案件ほど未確認のまま先へ進みます。そのため固定の往復数は終了条件にしないで、必要な各論点が回答済み・後で確認・不明・該当なしのいずれかになったかを基準にします。未確認だけを次の質問対象にし、すべて片付けば終えるため、質問回数を固定しなくても不要な質問は増えません。利用者が途中で終了した場合は、残る未確認の論点を Open Questions に記録することで、回答を強制せず取りこぼしも見える状態にします。

固定上限を外すだけでは、AI の連想で論点が増え続けるおそれがあります。そこで、新しい論点は利用者の回答または既存資料を根拠として示せる場合だけ追加し、既存の論点と同じ意味なら追加しません。すでに回答済み・保留・不明・非該当になった論点も再質問しません。「何を片付ければ終わるか」と「何を新しく加えてよいか」を対にすることで、必要な確認を落とさず対話を収束させます。

正直な注記: LLM4RE は2023年以降に急増した若い領域で、評価の多くは controlled environment に偏り industry-scale の長期評価はまだ限られます。omission recap は「LLM は補完案の提案者であって完全性の判定者ではない」という確度の高い区別に立つ防御設計であり、特定手法の優位を主張するものではありません。

## 意図の単位と完了の単位 — 分けて、トレースで結ぶ

packet は、「何を実現したいかを表す単位」と「実装作業の完了を管理する単位」という、性質の異なる2つの役割を1つの ID で結び付けます。この2つを混同せず、別々の情報として持つことが、進捗・依存関係・検証結果を扱ううえで重要です。

要求工学では、両者は異なるライフサイクルに属します。**ゴール指向要求工学（GORE / KAOS）のゴール**は「何を、なぜ達成すべきか」を表し、時間、スケジュール、特定の実装には依存しません。一方、**WBS（Work Breakdown Structure）のワークパッケージ**は「どう作業するか」を表し、担当、見積もり、実行順序を持ちます。van Lamsweerde は、要求を特定のゴールを達成する1つの方法と位置付け、上位のゴールほど安定すると述べています。

packet に当てはめると、意図は比較的安定していますが、作業の状態は頻繁に変わります。両者を同じ情報として扱うと、作業が中止されたときに意図まで失われたり、別の実装方法を検討できなくなったりします。また、「関数をマージした」という作業の完了を、「決済が安全に完了する」という目的の達成と取り違えるおそれもあります。

intent-planner は、packet を別々のファイルへ分けるのではなく、packet 内の節と frontmatter でこの違いを表します。Parent Intent / Why / Scope / Safety-Invariants には意図を、`state` には進行段階を、`## Evidence` には検証結果を、`depends_on` には工程上の依存関係を記録します。これらは packet の ID と `spec_refs`（下流の仕様への参照）で結び付けます。

Gotel & Finkelstein（1994）は、要求が生まれるまでの経緯を追う関係と、要求から設計・実装・検証へ進む関係を区別しました。後の研究では、それぞれ pre-RS / post-RS と呼ばれます。intent-planner では、North Star、不変条件、Decision Rules が意図の由来にあたり、packet の実装、検証、完了が実現の記録にあたります。

なお、goal/need → requirement → design/code/test という製品アーティファクト連鎖の traceability が RE で広く扱われてきたものであり、goal（GORE）と work package（WBS）のような**作業側の単位**まで含めたリンクは標準と矛盾しないものの、それを中核 doctrine として明示する主要文献までは確認されていません。本ツールが意図と作業の単位を分けてリンクするのは、この標準的 traceability の射程に収まる範囲での設計判断です（「別種の単位なので、必要なら明示的にリンクを張る」）。

work-item の明細（packet 内の下位タスク単位の進捗）を intent-planner に持たず下流（cc-sdd / kiro の tasks.md）へ委ねるのも、この分離の延長です。意図と完了の単位は分けたうえで、work の明細はさらに下流に置き、`spec_refs` でリンクするに留めます。同じ情報を二箇所で正本化しない（single source of truth）ためです。

## Packets — 小さく・縦に・戻せる単位

packet に求められる性質（behavior-preserving / testable / rollbackable、Scope と Non-scope の明記）は、アジャイル開発のストーリー分割の知見に対応します。

- **INVEST**（Bill Wake）— 良い作業単位は Independent / Negotiable / Valuable / Estimable / Small / Testable
- **垂直スライス** — レイヤー単位（DB だけ・UI だけ）ではなく、観測可能な振る舞いを貫く単位で切る
- **behavior-preserving** — Martin Fowler の『Refactoring』における定義。「外から見た振る舞いを変えずに内部構造を変える」を変更の単位に保つことで、検証が「振る舞いが保たれたか」という観測可能な問いになる
- **rollbackable** — 継続的デリバリーにおける小さなバッチの原則。失敗時に戻せる単位で進めることが、AI に大きな変更を任せる際の安全装置になる

packet の**個数は固定ではありません**。「3〜7 個に収める」といった目標個数を立てて数合わせをするのではなく、**改修の見込みの大きさ**で可変に切ります（1〜7 を緩い目安に・ごく小さな改修なら 1 packet で良い）。大きさは工数やストーリーポイント・行数といった数値メトリクスではなく、**触れる concern の数 × 既存境界への波及の広さ**という質的な軸で測ります。INVEST の Small は「小さく」を求めますが、何個に割るかは規模が決めるのであって、規模に対して数を水増しすると 1 packet = 1 concern（垂直スライス）の単位が壊れます。

分解の終端は、ある packet が複数の条件をすべて満たした時点です — (1) 一つの packet が一つの主要 concern に対応する、(2) 受入基準が観測可能な入力・条件・期待結果に還元される、(3) 解空間の境界（固定/裁量/禁止）が明示される、(4) **cheap-to-reverse（戻すのが安い）**、(5) トレース先（parent intent / spec_refs）が辿れる、(6) **単体完結（standalone completeness）** — packet 単体の done が、利用者・呼び出し側から見て中途半端でない一貫した振る舞いの区切りになっている。(6) は (4) とは別の独立条件です。(4) が「**作る側**のロールバック安全性（途中状態を戻せる）」を見るのに対し、(6) は「**呼び出し側**から見た完了形の意味的一貫性（half-done な振る舞いの done を作らない）」を見ます — 観測する主体が違うため、(6) を (4) に畳み込みません。

`/intent-packets` が「最初に着手すべき packet」を定性的な理由つきでちょうど1つ推薦するのは、**リスク駆動の順序付け**（Boehm のスパイラルモデル以来の「最大の不確実性から先に潰す」）と、Lean の**最小の実験**（PoC では仮説を最も安く反証できる packet を先に）の適用です。順序の根拠を明文化することで、「作りやすいものから着手して核心の不確実性が最後まで残る」という典型的な失敗を避けます。

## Walking Skeleton と画面ラフ

**Walking skeleton** は Alistair Cockburn による用語で、システムの全レイヤーを最小限に貫く E2E 実装を最初に作るプラクティスです。Hunt & Thomas（『達人プログラマー』）の**曳光弾（tracer bullet）**も同型の考え方です。狙いは統合リスク — 部品は個々に動くのに繋がらない — を最初に潰すことにあります。`/intent-packets` が designer-questions on のとき「最初の packet は E2E を貫くか」を確認するのはこのためです。

**画面ラフ**の確認は、要求工学における**プロトタイピング**（要求の獲得と妥当性確認の手段）に対応します。低忠実度のスケッチは「書かれた要求からは出てこない要求」を引き出す最も安価な手段の1つで、UI を持つプロダクトでは文章の要求だけで進むことが手戻りの主要因になります。

ラフの有無確認の先に、**画面デザインの詰め**（deep または画面を設計する観点を選んだ UI 案件のみ）があります。画面ごとの目的・情報の優先順位・主要状態（通常・空・読込中・失敗・権限不足・完了）・見た目の方向を対話で確かめるのは、情報アーキテクチャと状態設計を「実装者の暗黙の推測」から「レビューできる要求の材料」へ引き上げる同じプロトタイピングの延長です。素材が無い部分は推測（inferred）標識付きの派生下書き（`.intent/nl-spec/screen-design-brief.md`）として残し、確定と推測を混ぜないことで、仕様上は動くが凡庸で見づらい画面——推測が確定のふりをして実装に流れ込む drift——を上流で止めます。下書きの先には、望んだときだけ生成される見て触れるモック（`.intent/nl-spec/screen-design-mock.html`・派生・自己完結）があり、依頼者は実装前に目で確かめて修正を指示できます。モックは `.intent/` の派生に閉じ、実装（walking skeleton 以降）は従来どおり下流工程の領分です。

## PoC の仮説・反証条件・GO/NO-GO — 反証可能性

purpose が poc のときに問われる3点は、科学哲学とリーン開発の実践に根ざしています。

- **仮説と反証条件** — Karl Popper の**反証可能性**: 何が観測されたら棄却するかを言えない仮説は検証になりません。「動いた部分だけを見て成功と判断する」確証バイアスへの対策として、棄却条件を実装前に書き残します。
- **GO/NO-GO 基準** — Eric Ries の『リーン・スタートアップ』が説く validated learning と同じく、判定基準を**結果を見る前に**固定します。結果を見た後に基準を作ると、サンクコスト（ここまで作ったのだから）が判定を歪めるためです。

## Writeback / Improve — ドキュメントを生かし続ける

工程の完了と利用者成果は別の観測です。実装とテストが終わったことは、狙った価値が出た証拠にはなりません。L1 の `成果の物さし:` は価値が出たと何で分かるかを先に置き、リリース後の `成果についての学び` は Packet 単位の delta に履歴として蓄積します。そのうち人の承認を得た最新結果だけを L1 に戻すことで、工程の記録を成功の代用にせず、観測から意図へ戻る学習ループを閉じます。

- **Lehman のソフトウェア進化の法則**(1980) — 実際に使われるソフトウェアは変化し続け、意識的に手を打たなければ構造は劣化していきます。ドキュメントも同様で、実装と同期し続ける**仕組み**がなければ必ず陳腐化します。
- **Twin Peaks モデル**（Nuseibeh, 2001） — 要求とアーキテクチャは「要求 → 設計」の一方通行ではなく、互いに行き来しながら並行して詳細化されます。`/intent-writeback` は、この「実装から要求側への登り」を毎 packet 後の手続きとして固定したものです。
- **Living Documentation**（Martraire） — ドキュメントは、実装の現状を反映しているほど役に立ちます。intent-planner は、まずすべての学びを delta に記録し、人の確認後に正本へ反映します。判断基準や不変条件に影響する学びは、項目ごとに明示的な承認を求めます。それ以外は反映先を一覧にし、止めたい項目を指定できるようにしたうえで、残りをまとめて反映します。これにより、すべてを1件ずつ確認する形骸化を避けながら、確認なしに正本を書き換えない原則を守ります。

  delta を経由する規則は、実装後に学びを書き戻す `/intent-writeback` にだけ適用します。実装前に判断基準や作業単位を作る `/intent-compass` と `/intent-packets` は、正本を直接編集します。実装前の起草と実装後の書き戻しでは、変更の目的と危険が異なるためです。
- `/intent-improve` の3軸 **completeness / correctness / coherence** は、要求仕様の古典的な品質特性（ISO/IEC/IEEE 29148 などにおける完全性・正確性・無矛盾性）に対応します。packet 単位の writeback が拾えない全体レベルの乖離を、節目にまとめて検査します。coherence 軸が Decision Rules の **Revisit when 条件の成立**を検出するのも同じ発想で、「見直し条件を書かせる」だけでなく「成立したら知らせる」ところまでを仕組みにしています。

## Validate — 意図文書の lint と、問いの期限

`/intent-validate` の各検査が安定した**検査 ID**（invariant-conflict, stale-assumptions, …）を持つのは、コードに対する lint・静的解析の設計をそのまま意図文書へ持ち込んだものです。指摘が ID で参照できると、報告の再現・抑制・追跡が可能になり、検査の追加・変更も互換性として管理できます。

このうち **stale-assumptions** は、AI が推測した前提（Assumptions）が検証されないまま滞留することへの検査です。前提は時間とともに「検証されていない」から「事実として扱われている」へ静かに昇格してしまうため、未検証のまま古びた前提を明示的に報せます（canonical と推測を分ける provenance 管理の、時間軸方向の補完です）。

**groundless-conclusion** は、結論だけが残り、根拠（rationale）をたどれない意図を「根拠なき結論の疑い」として示し、あとから見直せる状態かを点検する検査です（[結論だけでなく、見直しに必要な根拠を残す](#結論だけでなく見直しに必要な根拠を残す)節を参照）。`coinage-suspect` と同じく、読み取り専用で候補を示すだけです。意味を踏まえて判定するため誤検知の可能性があり、作業は止めません。

**unverified-hypothesis** は、裏付けがないまま確定された仮説（暫定の確信）を示し、`.intent/` に反証や未検証の記録がないかを点検する検査です（[決定前に、仮説を既存の記録と照合する](#決定前に仮説を既存の記録と照合する)節を参照）。`groundless-conclusion` が結論の根拠不足を見るのに対し、こちらは仮説を支える証拠の不足を扱います。参照できる証拠がない場合は警告せず、判定できないことを明示します。この検査も読み取り専用で、意味を踏まえて候補を示すだけです。誤検知の可能性があり、作業は止めません。

### `dangling-reference` — 参照先が残っているか

compass の `Anti-direction N`、`INV N`、`DR N` といった番号付き参照が、項目の移動、統合、削除によって参照先を失っていないかを調べます。これは Doorstop の suspect-link に相当する検査です。

対象は compass 内の番号付き参照に限ります。別のリポジトリにある記録までは確認できません。LLM が実際の記述を読み、参照先の有無を意味に沿って判断します。読み取り専用で候補を示すだけなので、誤検知の可能性があり、作業は止めません。

### `invariant-oracle-missing` — 不変条件を確かめる方法があるか

Invariant のすぐ近くに、違反を見つけるための確認方法が書かれているかを調べます。確認方法は自動テストに限らず、手順でも構いません。性質上確認できない約束には、「ものさし無し」と明記します。どちらもない場合、その Invariant は守れているか判断できない努力目標になってしまいます。

判定は LLM が compass を読んで行い、正規表現だけでは決めません。確認方法の追加は、利用者が承認した別の作業として行います。

変更したファイルから関係する Invariant を探す補助機能もあります。変更パスと、Invariant に記録した影響先を照合し、「どの約束に触れ、何で確認するか」を示します。この逆引きは既定で無効で、警告だけを出します。規模が大きくなっても、ベクターデータベース、依存グラフ、GraphRAG は自動導入しません。利用者が必要性を判断して明示的に選んだ場合だけ、将来の選択肢になります。

### `requirement-oracle-check` — 受入基準で誤実装を見分けられるか

cc-sdd や OpenSpec へ渡す下書きの受入基準が、誤った実装を落とせる観測可能な基準になっているかを調べます。「使いやすくする」「適切に動く」のように、入力、条件、期待結果を確認できない基準は候補として示し、具体化を勧めます。

`requirements-smell` が曖昧な表現を、`export-draft-mismatch` が packet との不一致を見るのに対し、この検査は基準が誤実装を見分けられるかを意味で判断します。同じ箇所が複数の検査に挙がることはあります。

検査は読み取り専用で、結果の深刻度は「推奨」です。下書きの作成や実装を止めず、修正は利用者の承認後に行います。まずは export の下書きだけを対象とし、`/intent-to-spec` の成果物は対象に含めません。

### `draft-content-dropped` — 下書きの内容が下流に残ったか

export の下書きと、そこから下流のツールが生成した仕様を読み比べます。確認するのは、次の2点です。

- 下書きに含めた親の意図、判断基準、守る約束が requirements、design、tasks に残っているか
- `## Acceptance Material` に書いた判定条件が、下流の受入基準に残っているか

比較する仕様は、export 記録に残した機能名から特定します。その記録がない場合は packet 名でも補助的に探しますが、対応を確定できなければ「比較できない」と報告し、推測では結び付けません。下流に仕様がない直接実装の案件では、この検査を行いません。

結果は、どの内容がどのファイルで失われたかを示すところまでです。割合やしきい値で採点せず、export や実装も止めません。

### `compass-rule-decay` — 判断基準そのものが古くなっていないか

compass の Invariant や Decision Rule が、参照先は存在していても、現在の実装や前提から外れていないかを調べます。対象は、前提が崩れた項目、長く使われた形跡がない項目、完了済みの packet からしか参照されない項目です。

`dangling-reference` は参照先が消えた場合を、`invariant-stale-vs-compass` は packet が compass の更新に追い付いていない場合を扱います。この検査は、参照先が残っていても判断基準の内容自体が古くなった場合を扱います。

判定は LLM が compass と参照関係を読み、必要に応じて影響先のパスが存在するかを補助的に確認します。期間や参照回数の固定しきい値は設けず、Git の履歴も読みません。前提が崩れた可能性が高い項目は、根拠となる記述を引用して個別に示します。使われていない疑いだけの項目は、既定では件数をまとめて示します。

更新や履歴への移動は、人または `/intent-improve` が別に行います。この検査は読み取り専用で、候補を示すだけです。

### 所見の根拠固定（evidence-anchored finding）

これは新しい検査項目ではなく、`/intent-validate` がすべての所見を報告するときの共通規則です。評価する AI の出力が毎回安定していても、その内容が正しいとは限りません。そのため、各所見には intent-tree、intent-compass、packets から、根拠となる記述をそのまま引用します。引用で裏付けられない所見は、深刻度を「要修正」にせず「推奨」以下に留めます。

引用が本当に所見を支えているかは、LLM が意味を読んで判断します。単なる文字列一致では決めません。golden dataset、kappa、pairwise 評価、promptfoo などの評価基盤を製品へ組み込むものでもありません。あくまで報告の規則であり、自動採点や停止条件には使いません。

### compass conformance — 憲法が効いているかを後から照合する

compass（Invariants と Decision Rules を束ねた「プロジェクトの憲法」）には、不変条件と判断基準を**書く場所**が整っており、packet 起案時に `/intent-packets` がそれを各 packet の `## Safety / Invariants` へ反映する一方向のプッシュも存在します。しかし「反映されたか」「compass を後から更新したとき既存 packet が追随しているか」「packet の決定が Decision Rules に反していないか」を後から**照合する場所**は弱いままでした。憲法は書く場所より効いているかを照合する場所が肝だ、という観点（BMAD-METHOD の project-context.md＝全フェーズの不変入力として常に効く "constitution" との比較から得たもの）に基づき、`/intent-validate` は read-only の照合軸を3つ持ちます。いずれも宣言の突合のみで、算出・採点・自動修正・自動継承を行いません。

- **`invariant-uninherited`（継承欠落）** — compass のプロジェクト普遍 invariant のうち、各 packet の `## Safety / Invariants` に継がれていないものを列挙します。これは要求工学の **requirements traceability**、とりわけ Gotel & Finkelstein の pre-RS トレーサビリティ（意図→要求の変換が最初の切断点）を、「憲法→packet の継承トレース」へ当てはめたものです。既存の `invariant-conflict`（packet が compass invariant と**衝突**する＝矛盾）とは検出観点が異なり、本軸は「衝突ではなく**沈黙の欠落**」を対象とします — テキスト内のパターンから検出できる矛盾と違い、欠落（沈黙）は照合の基準となる外部の枠（ここでは compass の Invariants 集合）がなければ認知できない、という[損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす)の節で述べた完全性検査の含意の、compass↔packet 辺への適用です。
- **`invariant-stale-vs-compass`（compass 更新の遡及漏れ）** — compass の節更新日（Invariants 節 / Decision Rules 節）が packet の `updated_at` より新しい packet を stale 候補として提示します。これは Lehman のソフトウェア進化の法則（憲法も実装も使われる限り変化し続け、放置すれば乖離する）と、Twin Peaks の往復（要求↔設計の相互更新）が**遡及されないまま取り残される**断面を、stale-assumptions の時間軸検査と同型に捉えたものです。compass のどの項目が当該 packet に効くかを機械同定できないため要修正と断定せず、人に追随確認を促す安全側（推奨以下）で出し、両側の更新日が実打刻されたペアのみを比較します（断定より未検証対象の明示を優先する fail-safe）。
- **`decision-rule-mismatch`（ADR 乖離）** — compass の Decision Rules（Context / Decision / Why / Alternatives considered / Consequences / Revisit when の軽量 **ADR**）と各 packet の `## Decisions` 節を突合し、Decision Rules に反する packet 決定を報告します。突合面は ADR の `Decision`（採る選択肢の主文）対 packet スロットの確定値に限定し、Why / Alternatives は根拠の引用元に使うに留めます。既存の `l3-intent-mismatch`（intent-tree L3 との突合）とは突合相手が異なり、本軸は compass の Decision Rules との突合のみを対象とします。

3軸とも invariant / Decision Rules に機械的 ID が無い自由文を相手にするため、`l3-intent-mismatch` で確立した**意味照合＋人ゲート**の粒度に倒します — 機械的完全一致を前提にせず、明示記述と直接矛盾＝要修正／解釈の余地がある乖離＝推奨／迷ったら推奨に倒し根拠引用を添えて人に委ねる、推論禁止（packet 内容から該当性を推論しない）。これらは単なる lint ではなく、**compass＝意図の射影**と**packet＝完了の射影**の間の落差を観測可能にする装置です。[損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす)の節が述べたとおり、落差そのものは避けられない — 避けたいのは、後続の正しさを支えていた憲法の制約が黙って継承漏れ・追随漏れ・乖離を起こすことです。この落差の検出こそが、本ツールが「仕様を完全にする」のではなく「落ちたものを見せて人に仕分けさせる」という一貫した姿勢の、compass↔packet 辺への延長です。

#### 決定の波及先を列挙し直す — 決定を取り巻く4方向

決定（compass の Decision Rules / packet の `## Decisions`）は**記録**できても、その決定が前提とする上流・成立トリガ・文体に紛れた未確定・名指しする下流コードへ、影響を**展開し直す**機構は弱いままでした。これは前述の継承照合（compass→packet の1辺）を、決定を中心とした4方向へ広げる延長です。いずれも read-only の宣言突合に徹し、canonical を自動書き換えしません。

- **上流＝起票漏れ（discover スキップ）** — `.kiro/specs/` に進行/完了 spec があるのに `.intent/intent-tree.md` の L0〜L4 とテキスト照合できないものを、`/intent-status` が「Packet を経ずに実装が進んだ疑い」として候補提示します。これは前述の[孤児 spec 検査](#横の網羅性--完全性スキーマ--観点別レビュー--トレース)（Packet 層・writeback-phase-boundary で導入した起票漏れ⇔writeback漏れの双対）を**最上流（intent-tree 層）へ1段押し上げた**もので、tree 層 / packet 層 / writeback 層の3階層で棲み分け、同一 spec の二重警告を避けます。
- **トリガ＝Revisit when 発火の拾い上げ** — Decision Rules の `Revisit when`（[有効期限つき決定](#decision-rules--軽量-adr-と設計根拠)）は「成立したら知らせる」ところまでを仕組みにする欄です。`/intent-improve` の coherence 軸が、実装の現実と deltas（実装から得た学びの記録）から条件の成立を読み取り、見直しを再提案します。
  以前は、利用者が「本番構成を確定」のような節目を `.intent/milestones.md` に書き、各 Decision Rule の `Revisit when` と照合する仕組みもありました。しかし、実際には使われなかったため撤去しました。

  intent-planner 自身の143件の `Revisit when` を調べると、ほとんどが「誤検知が多いと分かったとき」のように、リポジトリ内の記録や対話から確認できる条件でした。外部の節目を別台帳へ書く必要がある条件は、実質的にありませんでした。また、利用者が milestones へ記入した実績もありませんでした。そのため、使われない台帳と照合経路を残さず、deltas から見直し条件を検出する既存の経路へ一本化しました。節目の記録は、学びなら deltas、完了なら packet の archive、コミットとの対応なら Intent trailer、外向けの説明なら release note が受け持ちます。

- **文体＝未確定動詞の繰延タスク化** — 「想定 / 流用 / 予定 / TBD / 暫定」のような未確定動詞が確定の文体に紛れ込むと、決定の未確定性が黙って失われます。起草フェーズ（`/intent-packets` / `/intent-compass`）はこれを推測で確定せず Open Questions / 理由付き未定スロット（[投与量](#縦の深度--可視の設計規則を固定し規則の内側を委譲する)の遅延側）への変換案として提示し、`/intent-validate` の `ambiguous-deferred-phrasing` 軸が `## Decisions` の区分外・Revisit when 併記なしの同種表現を「情報」severity で引用します（言い換えは正本へ書き戻さない read-only 報告）。これは Requirements Smells（[横の網羅性](#横の網羅性--完全性スキーマ--観点別レビュー--トレース)）と同型の表層検査を、決定の未確定性へ向けたものです。
- **下流＝決定↔コード乖離** — `/intent-validate` の `decision-rule-code-alignment` 軸は、Decision Rule の主文/Context がコードモジュール（ファイル名・モジュール名）を名指すとき、そのモジュールを Grep で読み、Rule 主文と実装の意味的乖離を検出します。これは**read-only 検査層で唯一、限定的な AI 意味照合（推論）を許す軸**です — 代償（誤検知の再現性低下）を、出力に「AI による意味照合の推定」ラベルを付し断定しない一方向報告・推奨 severity 止まり・名指し起点限定で補償し、この例外を他軸へ波及させません。突合面はコード実装の意味で、packet スロット値のテキスト突合に限る `decision-rule-mismatch` とは分離します。同じ視野は `/intent-writeback` §2 の `[invariant-violation]` 抽出にも明示し、名指しモジュールの grep 突合を含めます。

この4方向はいずれも「[損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす)の落差を観測可能にして人に仕分けさせる」一貫した姿勢の適用であり、決定という1点を起点に上流・トリガ・文体・下流の各辺へ広げたものです。

これらの検査は、存在するだけでは役に立ちません。`/intent-validate` を export 前にしか実行しないと、export しない期間には、compass の更新へ packet が追い付いていない状態がたまります。

そこで `/intent-status` は、compass の更新日と packet の `updated_at` を読み比べます。compass の方が新しい packet が一定数以上あれば、次の一手として `/intent-validate` を勧めます。これは検査結果の確定ではなく、実行する頃合いの案内です。両方に実際の更新日がある場合だけ判断し、作業は止めません。

`/intent-improve` で Decision Rule や Invariant に関わる変更を反映した場合も、続けて validate するよう案内します。status、improve、validate はいずれも正本を書き換えません。「最後に validate した日時」のような新しい状態は保存せず、compass と packet の更新日という既存の情報だけを使います。

drift-watch の記録に、利用者の判定がない項目が3件以上たまった場合は、status が `/intent-improve` を次の一手として勧めます。drift-watch が無効な場合や記録がない場合は、何も表示しません。ここでも自動実行はせず、提案だけに留めます。

Open Questions の **`[export まで]` タグ**は、Lean ソフトウェア開発の **last responsible moment**（決定はそれを下すべき最後の責任ある瞬間まで遅らせる）の実装です。すべての問いに即答を求めると planning が止まり、放置すると未決定のまま実装に流れ込みます。「いつまでに答えればよいか」を問いに付けることで、遅延してよい決定と、export という不可逆点の前に必要な決定を区別します。export 前にタグ付きの未回答を確認するのは、この期限の執行です。

## 要求を広く深く確認する — 横の網羅性と縦の深度

AI コーディングエージェントへ実装を委譲するとき、最大の失敗要因は「タスクの分解が粗い（縦）」よりも「そもそも触れられていない話題（横）」です。エラー時の挙動・整合性・冪等性・認可・後方互換などが仕様に書かれないまま委譲されると、エージェントは**暗黙の（多くは安直な）デフォルト**でコードを埋め、本番で深刻な不具合を生みます。intent-planner はこれを**横の網羅性**と**縦の深度**の二軸で抑えます。この二軸の設計指針は、前提を伏せた中立な調査でも独立に支持されました（要求工学・アーキテクチャ・認知科学の一次情報との突合）。

### 横の網羅性 — 完全性スキーマ × 観点別レビュー × トレース

要求品質の標準（ISO/IEC/IEEE 29148・INCOSE GtWR）は、完全性を「よく書けた一文」ではなく**要求集合が必要な関心事を十分に含むこと**として扱います。したがって横漏れは「よい質問」だけでは防げず、**関心事スキーマ（聞き漏らしの目次）× 観点別レビュー × トレーサビリティ**の複合でしか安定して防げません。

- **完全性スキーマ（決定スロット）** — 聞き漏らしやすい技術的決定を packet の `## Decisions` に**スロット**として列挙します（整合性・冪等性・エラー意味論・認可など）。これは Bass らの **ASR（Architecturally Significant Requirements）** を「required-how を上流で確定する」形に落としたもので、固定リストではなくモード別差分で文脈に合わせます。各スロットは「回答済み / 未定（理由付き）/ 非該当 / ADR候補」のいずれかで必ず閉じ、「黙って飛ばす」を構造的に防ぎます。スロットの正本は単一ファイル（`decision-slots.md`）に集約し、「表に行を足すだけ」で拡張します。
- **観点別レビュー（PBR）** — Basili らの **Perspective-Based Reading** は、利用者・運用・テスト・保守など異なる視点から読むと、同人数の通常読解より広い欠陥カバレッジを達成すると実証しました。`/intent-validate` は四観点の静的チェックとして、各観点で「破綻条件が記述されているか」を read-only で確認します（対話的監査は別途のレビューに委ね、新たな自律ループは作りません）。
- **Requirements Smells** — Femmer らの研究に基づき、曖昧語・主観語・比較級・弱い語・未定義代名詞といった表層欠陥を機械的に拾います。これは「書かれ方」を直す二次防衛線で、「そもそも書かれていない」を補う完全性スキーマとは別の層です。
- **トレース** — NASA SWE-059 や Mäder & Egyed（双方向トレースが保守作業を有意に速く・正確にする）に基づき、上流（親意図）に加え下流（realized-by / verified-by）への最小十分なリンクを張ります。Gotel & Finkelstein が指摘した **pre-RS トレーサビリティ**（意図→要求の変換が最初の切断点）を踏まえ、parent_intents 欠落を検査します。

漏れを見つける検査だけでは、全体の進み方を把握できません。そのため `/intent-overview` と `/intent-status` は、packet の `state`、export の履歴、delta の状態を照合し、工程を次の5つに分けて縦に表示します。

- 反映済み
- 現在の作業
- 未着手
- 書き戻し待ち
- 統合済み

特に、実装は完了していても、学びが意図へ書き戻されていない packet を工程上のずれとして示します。表示自体が状態を推測することはなく、packet、export-log、deltas を正本として読み取るだけです。

逆に、packet を作らずに仕様や実装へ進んだ可能性も `/intent-status` が調べます。`.kiro/specs/` に進行中または完了済みの仕様があるのに、対応する packet や delta が見つからない場合、「起草されていない実装の可能性」として候補を示します。

export から下流の仕様作成まで進んだ場合は、生成された機能名を export 記録へ追記するため、packet と仕様の対応をたどれます。古い記録や途中で止まった export では、packet 名も補助的に照合します。それでも対応を確認できなければ、断定せず警告だけを出します。enforcement が export 後の書き戻し漏れを扱うのに対し、この検査は export 前の起草漏れを扱います。

5信号は「反映の進捗」を映しますが、それだけでは「いま **どの packet が・どの工程に居て・この後どの工程を通るか**」が読めません。そこでレールの各行には、信号に続けて `[現在の工程 → 次に通る工程]` を併記します。これは新たな観測・推論ではなく、すでに進捗軸で読み取る packet `state` を、固定パイプライン（discover → compass → packets → export → 実装 → verify → writeback）上の位置として読み替えた射影にすぎません。これにより「今ここ」は packet 単位の指し示しから、**工程単位での現在地と残工程の連なり**へと解像度が上がります。あわせて、各スキルの出力は「読み手（人間開発者）が最初に掴むべき結論＝今どこ・次に何をするか」を筆頭に置き、内部規律（派生・正本ではない旨・検査 ID 等）は詳細へ退避する構成に統一しています。出力の読みやすさは、トレーサビリティの可視化が運用に乗るための前提条件です。

### 責任範囲を分けて仕様を読む

観点別レビューは、職種の人格や架空の会議を再現しない。同じ仕様を、製品を決める責任、条件付きで進行を整える責任、利用体験を設計する責任から別々に読む。これは Perspective-Based Reading（PBR）が、異なる責任範囲から読むことで一つの読み方では見落とす欠陥を補う考え方を、実装前の対話へ移したものである。職種名は固定された登場人物ではなく、案件に必要な判断範囲を見つける手がかりに留める。

AIが担当者の代わりに答えるときは、結論だけでなく根拠の強さを保つ必要がある。人が確かめた「確認済みの事実」、材料から導いたが未承認の「根拠付きの推測」、材料が足りない「未確認」を分ければ、もっともらしい補完を決定と取り違えず、あとから訂正できる。案件に関係しない論点は「非該当」として閉じ、質問数を増やさない。

複数の責任範囲が異なる判断を求めたら、食い違いは分けたまま根拠と未確認事項を示し、人の判断へ戻す。早く一案にまとめることより、何を選び何を失うかが見えることを優先するためである。一方、体験設計のフレームは考えを整理する任意の整理手段であり、必要な観点を拾うレビューとは責任が別である。フレームを採用しなくても、接点、失敗、裏側の支援という確認事項は残る。

### 縦の深度 — 可視の設計規則を固定し、規則の内側を委譲する

縦は「どこまで深く決め・どこで止め・上下をどう繋ぐか」です。指針は **「人間は可視の設計規則（visible design rules）を先に固定し、規則の内側の局所探索はエージェントに委譲する」** という二層化で、Parnas の情報隠蔽・Baldwin & Clark の visible rules / hidden modules・Sullivan らの **Real Options** が支えます。

- **投与量（前倒し / 遅延）** — 不可逆・外部契約・品質トレードオフ・受入オラクルに効く決定は前倒しで固定し、可逆で局所的・探索可能な決定は「未定（遅延中・再訪条件付き）」としてエージェントに委ねます。Real Options の語彙では、遅延してよいのは「可逆で情報価値が増える決定」だけで、選択肢が消える前（last responsible moment）には固定します。
- **分解の終端** — 「テストが書ける（testability）」だけでは不十分で、**誤った実装を落とせるオラクルがある（discriminative testability）**まで降ります（SWE-bench Verified / UTBoost が、弱いテストでは誤実装が通ることを実証）。粒度は「一 packet = 一 concern」を停止条件とし、how の完全指定はせず what + constraints + oracle に留めます。

### 既定値を出さない — アンカリングの回避

ツールがスロットに「妥当な既定値」を提示すると意思決定は速くなりますが、Tversky & Kahneman の**アンカリング**、Johnson & Goldstein の**デフォルト効果**が示すとおり、提示値が「推奨・標準」として無批判に採用され、本当の要求から乖離します。Zhou ら（LLM 協働開発のバイアス研究）は、開発者アクションの約半数が認知バイアスの影響下にあり、その多くが LLM との相互作用に起因することを報告しています。そこで intent-planner は**既定値を埋めず、未決定を理由付きで明示保持**します。さらに高コストな決定（認可・整合性・エラー意味論・後方互換）に限り、Mussweiler らの **consider-the-opposite**（その選択が不適切になりうる条件を能動的に挙げさせる）で確証バイアスを緩和します。全スロットに摩擦を課さないのは、意思決定疲労を避けるためです。

## 保管構造 — 1単位1ファイルと「現役 / 履歴」の物理分離

packet を `.intent/packets/active/` の個別ファイルとして保持し、完了後は削除せず `archive/` へ移す構造は、要求管理と文書工学の複数の知見に対応します。

- **廃止要求は削除せず保存する** — Wnuk らの実証研究（2013）は、廃れた要求（obsolete software requirements）が現場の深刻な問題でありながら、回答者の7割超が処理プロセスを持たないことを示し、削除ではなく「obsolete とタグ付けして保存し将来の意思決定に再利用する」ことを推奨しています。done / superseded の packet を archive へ移して**消さない**のはこの実践です。過去の判断の監査と、書き戻し漏れの突合（export-log × 残存下書き × deltas）の材料にもなります。
- **要求は状態を持つオブジェクト** — ISO/IEC/IEEE 29148 や SEBoK は、要求に status・rationale・owner などの管理属性を持たせることを標準としています。packet ファイルの frontmatter（packet_id / name / state / spec_refs …）はその最小形で、`name` を export-log・cc-sdd 下書き・deltas を貫く照合キーの正本に固定することで、トレーサビリティ研究が指摘する**リンクの腐敗（link decay）**に備えています。
- **テキスト小単位 + バージョン管理** — Doorstop や、Ericsson の大規模アジャイル開発で使われた T-Reqs は、要求を Git 管理下の小さなテキスト単位として扱うことで、行単位の差分レビュー・並行編集・履歴追跡を成立させました。1 packet = 1 ファイルは同じ設計です。単一ファイルへの全件蓄積は、実務ツールの調査（OpenSpec / Backlog.md / ADR 等）でも主流ではありません。
- **canonical と生成ビューの分離** — docs-as-code の実証研究（Cadavid ら）は、機械可読な正本から索引やビューを自動生成する構成が single source of truth と保守コストの両面で有効なことを示しています。`packets/index.md` が「読み専用の生成物」であり、各コマンドが index + 対象 packet だけを読むのはこの構成です。
- **設計根拠は放置すると劣化する** — Grudin は設計根拠の記録が散逸・検索不能になりやすいことを早くから指摘し、Burge らは根拠の保守には変化の伝播が必要だと論じました。覆された Decision Rules を6欄のまま `compass-archive.md` へ退避し、**常時読み込まれる compass には現役の判断基準だけを残す**運用は、この劣化への対処です（ADR の superseded 慣行を、毎回全文を読む LLM という読者に合わせて物理分離まで進めたもの）。
- **追記型の記録は1単位1ファイルに分ける** — deltas / export-log / drift-log / milestones / compass-archive を単一ファイルの末尾に追加し続けると、並行作業が同じ場所を書き換えて衝突し、ファイルも読みにくくなります。そこで、packet に関する記録は packet 単位、出来事に関する記録は日付と短い名前の単位で、別々のファイルに保存します。中央の連番は使いません。異なる作業は異なるファイルを更新するため、同じ末尾を同時に編集する問題を避けられます。履歴は Git と `archive/` に残し、進行中の内容だけを現在の表示にまとめます。分散した記録は `/intent-overview` が進行中と保管済みに分けて再生成するため、新しい正本を増やす必要はありません。status / improve / writeback / overview は、分割形式があればそちらを読み、なければ従来の単一ファイルを読みます。移行の前後で集計や判定は変わりません。順序に意味がある記録には、順番を失う `merge=union` を使いません。
- **intent-tree / intent-compass 本体も、現役の内容と履歴に分ける** — `intent-tree.md` と `intent-compass.md` の本体は、機能を追加するたびに分析や候補が追記され、intent-planner 自身ではそれぞれ約8万トークンまで増えました。実装直前に関係する packet と判断基準だけを読むには、本体を軽く保つ必要があります。そこで、完結した機能の履歴を `intent-tree.history.md` と `compass-history.md` へ移し、本体には現役の骨格だけを残します。覆された Decision Rules を保存する `compass-archive.md` とは役割が異なります。移動するときは文面、番号、意味を変えません。`status: active | archived` で現役か履歴かを示し、grep で検索できる状態を保ちます。[損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす)で述べたとおり、正本は人にも AI にも読めるテキストのままとし、データベースは導入しません。
- **書き戻した学びは、実装時に読まれる場所へ置く** — `/intent-writeback` が実装から読み取った暗黙の挙動（implicit-behavior）などは、以前は intent-tree の L3 に追記していました。しかし、実装時には tree 全体を読まないため、そこに書いた学びが参照されないまま残ることがあります。そこで、学びの反映先を、関係する packet の `## Expected Behavior` に変更します。packet は実装直前に必ず読むため、学びが次の実装へ届きます。packet が `archive/` へ移れば、学びも一緒に履歴として残ります。変更するのは反映先だけで、delta への記録、人の承認、正本への反映という2段階の手順は維持します。
- **履歴へ移せない compass の項目は、領域で絞って読む** — 履歴へ移せるのは、番号を持たず、ほかの項目から参照されない記録です。一方、`intent-compass.md` の Anti-direction は番号を持ち、多くが現役の Invariant や Decision Rule から参照されています。intent-planner では214件中115件がこの状態でした。これらを移すと参照先がなくなるため、各項目へ `[領域: 配布]` のような領域タグを付けます。複数領域に共通する項目には `[領域: always]` を付け、どの領域を読む場合にも含めます。実装時は「対象領域と `always`」だけを grep し、必要な項目に絞って読みます。タグの追加では本文、番号、参照先を変えないため、既存の参照を壊しません。タグがない従来形式では、全文を読む動作へ戻ります。
- **新しい保存形式への移行は任意** — compass は、新しく作る項目を1項目ずつ別のファイルに保存します。一方、従来の単一ファイルも引き続き読み取ります。既存プロジェクトのデータをインストーラーが自動で移動したり、上書きしたりすることはありません。

> **まだ検証できていないこと**: 現役の項目を少なく保つとレビューの質が上がる、という直接的な証拠は、要求工学の研究ではまだ十分ではありません。この設計は、情報過多が意思決定を難しくすることや、廃止された要求を放置すると管理しにくくなることから導いた仮説です。実際の運用を通じて検証する必要があります。

## 大規模開発で意図を維持する — 領域別の管理責任（federated governance）

領域別の管理責任は、案件や参加者が増えても意図を保守できるようにするための中核設計です。補助的な運用規則ではありません。意図の正本は一つに保ったまま、誰が判断するか、どの範囲を処理するかを領域ごとに分けます。

意図の正本である compass と tree は、規模が大きくなるほど2つの問題に直面します。1つは、複数のセッションが同じ正本へ同時に書き込むと変更が重なりやすいこと。もう1つは、validate や improve の実行コストが記号の総数に応じて増えることです。このリポジトリでも、一人の開発者が複数の AI セッションを並行して使うだけで compass の記号が800件を超え、毎回すべてを走査する運用では、必要な頻度で保守することが難しくなっていました。

先行研究からも、ゴールモデルを1本の巨大な木として育て続ける方法には限界があることが分かります。i* のスケーラビリティに関する拡張マッピング研究（2016）は126本の文献を分析し、複雑さとスケーラビリティの管理を重要な未解決課題として挙げています。KAOS についても、大規模に適用するとモデルが複雑になり、完全な状態を保ちにくいことが報告されています。

一方、Linux カーネルのメンテナ階層、Python の PEP における決定者への委任、Google のモノレポにおける可視性ルールは、いずれも共通の形を取っています。**管理対象は一つに保ち、誰がどこまで判断するかを分ける**という形です。今回の調査では、意図の正本そのものを領域ごとに別々に持つことで成功した事例は確認できませんでした。

intent-planner では、この考え方を Data Mesh の **federated computational governance**（Dehghani）を参考に取り入れています。中央には全領域に共通する最小限の規律だけを置き、それ以外の判断は領域ごとに委ねます。具体的には、compass の `always` タグが付いた判断基準だけを全領域共通とし、各記号の領域タグに応じて管理責任と処理対象を分けます。

領域ごとに別々の正本を作らない理由は3つあります。

- 1回の grep で「対象領域と `always`」をまとめて検索できることが、必要な情報だけをその場で渡す仕組みの土台だからです。
- 複数領域にまたがる変更を、複数の保存場所の同期問題に変えずに済むからです。
- 外部連携を片方向に保ち、双方向同期の負担を持たないという intent-planner の方針に合うからです。

仕組みは次の4つです。どれも既存の仕組みを領域単位で使えるようにするもので、新しい階層を増やすものではありません。

- **対象領域を絞った実行**: 案件の領域が分かる場合、validate と improve は「その領域と `always`」の記号だけを読みます。ただし、古くなった記録や使われていない記録の検出には全体の確認が必要なので、その検査は絞り込みません。
- **領域情報と担当宣言の分離**: 領域の名前と説明は、Git で共有する用語として管理します。誰が現在その領域を扱っているかという担当宣言は、Git で共有しないローカル情報として分けます。記号が属する領域は、各記号の領域タグだけを正本とし、担当宣言側に重複した一覧は持ちません。
- **書き込み時の注意喚起**: 新しい記号を作るときは、案件の文脈から領域を推定し、利用者に1問だけ確認します。同じ領域を別のセッションが担当中であれば、その事実を知らせます。書き込みを禁止する仕組みではなく、並行作業をどう進めるかの判断は人に残します。
- **`always` 登録前の確認**: 新しい記号を全領域共通の `always` として登録する前に、本当にすべての領域へ適用するかを1問だけ確認します。共通規律が増えすぎると、領域で絞り込む効果が薄れるためです。

利用者から寄せられた「データベースのようにトランザクションで管理したい」という要望にも、実際のデータベースを導入せずに対応しています。依存を増やさず、grep と git diff で読める状態を保つため、次のように既存の仕組みを対応させています。

- レコード: 1単位を1ファイルに保存し、同時編集が重なりにくくする
- トランザクション: Git のコミットで、複数ファイルにまたがる変更と履歴をまとめる
- ビュー: 正本から index を再生成する
- 絞り込み: 領域タグを grep する

compass と intent-tree の案件記録には、どちらもこの1単位1ファイルの保存形式を適用済みです。

> **実装済みの範囲と、まだ検証していないこと**: この設計は、一人の開発者が複数の AI セッションを並行して使う運用と、800件を超えた記号の走査コストに対応するものです。人間のチームにおける担当管理は、宣言の形式と読み取り方を定めた段階に留めています。また、対象領域を絞ることで保守の実行頻度が実際に上がるか、並行セッションによる変更の重なりがどの程度減るかは、今後の運用で測る必要があります。保存形式の移行が完了していることと、運用上の効果が実証済みであることは区別しています。

## 横断変更を束ねる — ジャーニー（複数 packet にまたがる1つの変更）

packet は1つずつ実装に渡せる垂直な作業単位ですが、実際の案件はしばしば複数の packet に分かれます。骨格を先に置き、接続を後から足し、締めに文書を揃える、という工程です。このとき「工程の順序」「複数の packet が共同で守る契約と統合時の検査」「案件全体としての完了の判定」は、どの packet 単体にも属しません。intent-planner 自身の運用では、この束ねが計画ファイル（`.intent/packets/plan.md`）の手書き散文として繰り返し自然発生していました。需要は実在する一方で、手書き散文には2つの問題があります。並行セッションが同一節を取り合って衝突すること、そして読み手のコマンドが構造化して読めないことです。大規模開発の実務でも、複数の変更単位にまたがる仕事は専任の束ね単位で扱うのが実証された形です（Google が Large-Scale Changes と呼ぶ運用が代表例。ただし本節の射程はその主対象とは異なります＝末尾の注記）。

そこで、複数 packet にまたがる1つの変更を束ねる単位を**ジャーニー**として一級化し、`.intent/packets/journeys/` 配下の1ジャーニー=1ファイル（git 追跡・チーム共有）を正本にしています。設計は新しい記法の発明ではなく、packet で実証済みの型 — frontmatter スキーマ・1単位1ファイル・`archive/` への退避 — の再適用です。

- **必須の frontmatter は7キーだけ**（識別子・名前・lifecycle・構成 packet の列挙・作成/更新時刻・一行要約）です。工程計画・共有契約・統合時の検査・後回しの記録は本文の自由記述で持ち、見出しの形式を固定しません。記入が重いと使われなくなり、手書き散文へ回帰するためです。
- **進捗の状態を持ちません**。ジャーニーの進捗・完了は、構成 packet の `state`（per-packet frontmatter が正本）から読み手が毎回導出します。同じ情報を二か所に持つと食い違いが構造的に生まれるためで、[進捗の表現](#進捗の表現--状態は持つが状態機械は持たない)と同じ立場です。frontmatter に持つ状態は、人が閉じる `lifecycle: active | archived` だけです。
- **参照は journey→packet の一方向だけです**。packet 側の frontmatter（12キー固定）にジャーニーのキーを足しません。packet からジャーニーへの逆引きは `journeys/` の grep で行います。対応関係の正本を二重に持たないための向きの固定です。
- **完了は機械が判定しません**。「構成 packet がすべて done ＋ 統合時の検査が green」を人が確認して `lifecycle: archived` を記入し、`archive/<年>/` へ移します（削除しません）。経過日数などの機械閾値で自動で閉じることはしません。
- **数値スコア・日付の見積もり・進捗率を持ち込みません**。工程は順序と依存だけで表します。

書き手は `/intent-packets` です。案件が複数の packet に分かれるときだけ利用者に1問で提案し、承認されたときだけ起案します。単発 packet の案件には作りません（数合わせの儀式にしない）。読み手（`/intent-status` / `/intent-overview` / `/intent-validate`）は read-only で読みます。status はジャーニーごとの現在地を毎回導出し、overview のロードマップは構成 packet をジャーニー単位でも束ね、validate は複数 packet の共有契約（統合時の検査）の担い手漏れをジャーニー正本からも照合します。いずれも既定の出力を変えない純粋な追加で、`journeys/` が無い・空のリポジトリではすべてのコマンドが従来どおり動きます（計画ファイル経路の読み取りは恒久に残します）。

> **正直な注記**: 一級化したのは、機能追加のまとまりという実在した型だけです。記号の一斉改名のような機械的な全量横断変更（Large-Scale Changes の本来の主対象）には、実例が出るまで仕組みを広げていません。少数の実例に仕組みを合わせすぎないための歯止めです。また、将来のチームでの分担・レビュー委譲は「束ね単位が正本化されていることが足場になる」という見込みの段階です。チーム分担に関する作り込みは置き場と記入形式までに留めており、チーム運用での効果はまだ検証していません。

## Drift-watch — 外れきる前に捕まえる軽いフック

drift-watch は、intent を立てた後の各工程で実装が意図から逸れていく現象を、外れきる前に捕まえるための軽量な層です。enforcement と同じく `.intent/` の任意レイヤーであり、discover / export / improve に薄いフックを差して逸脱の兆候を観察します。

- **drift / erosion への接続** — Perry & Wolf（1992）の drift / erosion（[「なぜ『spec の手前』」](#なぜspec-の手前に1段挟むのか--問題空間と解空間)節で導入済み）は、設計者の意図したアーキテクチャから実装が徐々に逸れていく現象でした。これを intent-planner の文脈に置き直すと、intent を立てた後で spec 工程が進み実装が積み上がるほど、横断的な意図から離れていく現象として捉えられます。drift-watch は、その逸脱を**外れきる前に**捕まえる軽いフックを工程に差す層です。
- **fitness function への接続** — Ford ら（『進化的アーキテクチャ』）の **fitness function**（アーキテクチャ特性を継続的に検査する関数）の発想を借りています。ただし機械的な計測ではなく、LLM 文脈での**意味的な照合**として軽量化したものです。決定的な違いは、drift-watch は**停止しない**点です。fitness function が自動ゲートとして CI を止めうるのに対し、drift-watch は gate を持たず、誤検知を前提として観察と記録に留まります。
- **vibe architecting への接続** — Konrad ら "Architecture Without Architects"（arXiv:2604.04990, 2026）は、**vibe architecting** を "architecture shaped by natural-language prompts rather than by deliberate, recorded design"（自然言語プロンプトに形作られる、熟慮され記録された設計によらないアーキテクチャ）と定義します。論文は5つのメカニズム — model selection / task decomposition / default configuration / scaffolding and autonomous generation / integration protocols — を挙げています。drift-watch はこれを、AI エージェント協働における drift の**発生機序**として位置づけ、どのメカニズムから逸脱が生じうるかの見取り図に使います。
- **outcome 対称記録の誠実さ** — drift-log は逸脱の outcome を、効いた系（prevented / caught）と効かなかった系（missed / false-positive / not-applicable）の**両側**に対称に列挙します。これは**確証バイアスの構造的な回避**です。Popper の反証可能性に接続し、「効かなかった瞬間」= 反証の機会を捨てません。Ries の仮説駆動にも接続し、missed を記録することで設計仮説そのものを検証にかけます。したがって `missed=0` は「効いた証拠」ではなく「**記録漏れの疑い**」として読みます。
- **/intent-improve との分業** — drift-watch は逸脱を**記録するだけ**です。是正は `/intent-improve`（既存の5分類）の責務であり、記録と是正を混ぜません。writeback にはフックを差しません。観察する層と、直す層を分けておくことで、記録が「直す前提」に汚染されることを防ぎます。

正直な注記を2つ:

1. **論文は microservice 過剰分割を例示していない。** 同梱 seed の `microservice-over-split` は、我々の実プロジェクトでの実観測です。これを Konrad らの task decomposition / default configuration メカニズムの下に**我々が位置づけた**近接報告にすぎず、論文自身が報告した発見ではありません。論文の発見と現場の観測を混同しないでください。
2. **逸脱しやすい場面の事前チェックの有効性は未実証。** 「着手前に逸脱の型を名指しすると逸脱が減る」ことの直接の実証は薄く、周辺研究と近接報告からの推論です。drift-log 自身が、この設計仮説を検証する装置として設計されています（だから missed を記録します）。

### packet の範囲を超えた指示を検知する（packet-scope-overflow）

export のあと、利用者が packet の `## Scope` を超える実装を依頼することがあります。たとえば、フロントエンドだけを対象にした packet へ、バックエンドの実装を追加する場合です。そのまま進むと、新しい領域で必要になる認可、データ整合性、トランザクション境界、冪等性などの約束が、cc-sdd の成果物にないまま実装されるおそれがあります。

対策は2段階です。

1. CLAUDE.md や AGENTS.md に、範囲を超える場合は `/intent-packets` へ戻る規則を記します。これは AI に判断のきっかけを与えますが、強制はしません。
2. `drift-watch: on` の場合は、実装指示と packet の `## Scope` / `## Non-scope` を意味で照合します。範囲を超えていれば、`mechanism: packet-scope-overflow` として drift-log に記録します。コード差分は読みません。

この検査は、compass にある全体共通の Invariant との照合とは目的が異なります。ここで見るのは、packet の範囲を広げたことで、その packet 固有の約束が欠ける危険です。また、`outcome: missed` を記録することで、規約文書だけでは防げなかった件数も観測できます。`missed=0` であっても、ただちに有効だったとは判断せず、記録漏れがないかを確認します。これは[損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす)で述べた、文書と実装の間で失われた情報を見つける仕組みの1つです。

### 効果を検証できなかった機能を撤去した例（context-cost-cues）

drift-watch には以前、文脈を大量に消費しそうな進め方を知らせる `context-cost-cues` がありましたが、2026年7月10日に撤去しました。読み取り専用の検査では実際のトークン消費量を測れず、効果を検証できなかったためです。測れない値をログへ混ぜると集計が推測に依存し、何を過剰と感じるかという個人差を記録すれば、プライバシーにも踏み込みます。

文脈の消費を抑える問題は、必要な領域だけを読む、入口の文書を薄くする、仕様へ必要な情報だけを渡す、といった構造で対処しています。効果を測れない警告を増やすより、最初から不要な情報を読ませない方を選びました。ただし、`context-cost-cues` から得た「気づきを与えても強制しない」「測れないものをログにしない」という判断基準は残しています。

### 制約の候補を、必要な場面で提示する（constraint-starters）

compass の判断基準は、影響の洗い出しや失敗の想定から作ります。この方法だけでは、「SQL は必ずプレースホルダで組む」といった分野ごとの定石を、毎回一から思い出す必要があります。constraint-starters は、この作り方を置き換えず、文脈に合う定石を候補として補います。

参照するのは、出典付きの同梱カタログ `.intent/constraint-starters.md` と、利用者が育てる個人台帳 `.intent/constraint-library.md` です。どちらも読み取り専用で照合し、候補を compass へ自動転記しません。採用するかどうかは利用者が決め、個人台帳への追加も手動または明示的な承認がある場合だけ行います。制約の蓄積はこのプロジェクト内に閉じ、別のプロジェクトへ自動共有しません。

同じ仕組みは、案件の種類に応じた質問候補にも使います。`.intent/question-packs.md` から役立ちそうな質問群を提示しますが、採否は利用者に委ねます。該当する候補がなければ何も表示せず、既存の質問手順を置き換えません。制約と質問では蓄積する内容が違うため、カタログも分けています。

個人台帳は、記録するだけでは役に立ちません。以前は `/intent-compass` のときにしか参照しなかったため、実装方法に関する制約が、実際に必要となる実装段階まで届かない問題がありました。そこで制約を、方針を決めるときに役立つものと、実装手段が決まってから役立つものに読み分けます。後者は compass ではなく、packet の起草や実装前の判断時にも候補として提示します。このときも、[決定前に仮説を既存の記録と照合する](#決定前に仮説を既存の記録と照合する)手順の証拠源として使うだけで、新しい工程は増やしません。

同梱カタログも、compass だけでなく次の場面で照合します。

- packet を起草するとき
- データモデルを設計するとき
- 実装を始めるとき
- discover で意図を整理するとき
- validate で採用漏れを確認するとき

同じ定石を繰り返し表示しないよう、案件ごとに採用・否認・保留を記録します。一度判断した定石は、同じ案件の後続工程では再提示しません。ただし、案件の目的や前提が変わった場合は再び候補に含めます。その判断ができるよう、否認時には当時の文脈を1行残します。「一定の日数が経ったら再提示する」といった機械的な条件は使いません。

この採否記録は、チームで共有する正本ではなく、案件ごとのローカルな記録です。既定では判断結果だけを残します。利用者が任意の計測を有効にした場合に限り、いつ、どの定石を、どの工程で提示したかも記録し、あとから役立ち方を振り返れるようにします。

## 造語の管理 — ユビキタス言語を保ち、AI が増やす新語を減らす

使い込むほど、AI コーディングエージェントは「正規語彙（ubiquitous language＝そのプロジェクトで合意された正しい用語の集合）に無い新造の語＝**造語**」を増やしがちです。語彙が分裂すると、intent-planner の核である「意図の擦り合わせ」そのものが崩れます。造語管理は、これを**予防・検出・正規語彙の管理・言い換え**の4軸で扱い、造語を減らす層です。

- **既存の用語作法との境界** — intent-planner は当初、術語を英語のまま保ち初出に一行説明を添える「用語の作法」を持っていました（読みやすさと検索性のため、訳語へ置換しない）。造語管理はその作法が**引いた境界の外側**＝「AI が増やす造語そのものを管理する」を扱います。両者は地続きで、用語作法（初出説明済みの正当な新語）は造語検出の**除外条件**として再利用されます。
- **正規語彙の台帳（β）** — 「これが正しい用語」という母集合を、軽量な canonical 台帳 `.intent/glossary.md`（正規語＋表記ゆれ・同義語＋一行説明）として持ちます。訳語への一括置換はせず（可読性を損なわないため）、人が編集する canonical で、skill は read のみ・自動改変しません。台帳を明示することで、「既出語＝正規」とみなす雑音を避け、誤検出を抑えます。
- **登録済みの語彙を平易に保つ（予防2本）** — 造語の流入を検出するだけでは、台帳に登録された公認用語そのものが専門語のまま増え続けるのを止められません（登録されるほど公式の専門語が増える向きの穴）。そこで①台帳の一行説明を「初見の人に通じる普通の言葉」で書く規律（定義の中に未説明の登録語・英語術語を連ねない）、②新語を正式採用に諮る**前に**「既存の普通の言葉の組み合わせで言えないか」を先に確かめ、言えるなら登録しない関門、の予防2本を持ちます。検査は足しません（検査の名前がまた1つ公認用語を増やす自己言及を避ける）。既存の識別子の改名もしません（文書間の参照が壊れるため・平易にするのは説明文と、これから登録される語）。
- **普通の言葉に狭い意味を載せた語は初出で開く** — 造語検出は「台帳に無い新語」を差集合で拾う設計のため、字面が普通の言葉（「代行」「配布」等）に道具や AI の側が固有の狭い意味を載せた語は原理的に素通りします（2026年7月、このプロジェクト自身の対話で「代行＝答える人がいない観点を AI が仮回答で肩代わりすること」「配布＝正本を使える場所へ届ける2段の経路」が通じなかった実例で確認）。これは語の**置換では直しません**（置換は新しい語を増やすだけで、一般語への過剰な訳語狩りは読みやすさを壊します）。残して、利用者へ向く出力の**対話・文書ごとの初出で一行の普通の言葉の言い換えを添えます**——識別子の扱いと同じです。判定は「その狭い意味を載せたのは誰か」で行い、世間の意味のまま使われている一般語・既存技術用語には何も付けません。この観点は出力直前の点検に第4項として入り、discover の観点の確認（本人/代行）の問い文言には言い換えの実文が最初から書き込まれ、症例は台帳へ言い換え例付きで蓄積されます。
- **正確さが土台・平易さは手段** — ユーザーへ向く文章（質問・報告・読める生成文書）と新しく書く内部記録は、比喩や基準のない曖昧な言い方（「かなり」等）だけで意味を渡さず、意味が一意に読める記述で書きます（比喩を使うなら直後に正確な言い直しを併記。確立された専門用語を無理に普通の言葉へ開くと指す概念が特定できず逆に曖昧になるため、用語は残して初出に説明を添えます）。平易さはこの正確さを保ったまま易しく書くための手段であり、易しさのために意味を粗くしません。この原則は出力直前の点検・記録の書式・下流 spec へ渡す下書きの言葉の規律まで一貫して敷かれます。
- **検出は read-only・gate を持たない** — `/intent-validate` が台帳に無い語を「造語の疑い」として read-only で名指しし、正規語への言い換え案を添えます。判定は意味的（LLM 文脈）で機械検査スクリプトには寄せません。drift-watch と同じく**誤検知を前提に候補提示へ留め、停止しません**（gate を持たず既定 off の温度）。固有名詞・既存の英語技術用語・初出一行説明済みの正当な新語は除外します。
- **言い換えは提示まで** — 検出した造語を正規語へ寄せる更新案は**提示**するに留め、intent-tree / compass / packets / 台帳のいずれも自動では書き換えません。反映は人が承認してからの別アクションです（read-only 規律）。
- **場面としての予防** — 「造語が増えやすい場面」を drift-watch の型カタログに seed として持ち、着手前の逸脱しやすい場面の事前チェックでも名指しできます（drift-watch が on のときのみ・既定 off では何もしません）。

### 実際の利用で見つかった3つの問題（2026年7月）

実際のプロジェクトで、生成された設計文書の用語をそのまま使った質問が、エンジニアにも伝わらない事例がありました。そこから、従来の設計に次の3つの問題があると分かりました。

- **長い作業では、最初の指示が効きにくくなる** — 会話の入口には「普通の言葉で話す」という規則がありましたが、後の実装段階で質問を作るときには守られませんでした。入口の指示を増やすだけでは解決しません。そこで、下流の仕様作成ツールへ渡す下書きにも言葉の規則を含め、さらに利用者へ質問を出す直前にも文章を点検します。生成時の予防と、生成後の検査を組み合わせます。
- **語だけを見ても、一般語か内輪語かは決められない** — 英語圏で一般的な `footgun` や `hydrate`、一般語と同じ名前を持つ製品の Go や Zen のように、語そのものだけでは分類できない例があります。そこで、チーム共通語は「台帳に承認済みで登録された語」、未承認の独自用語の疑いは「台帳にない語、または暫定の語」、一般語は「想定する読み手の基礎語彙」として判断します。否認済みの語も台帳に残し、同じ語が後から再び作られたときに気づけるようにします。また、AI が承認を取ること自体を目的にしないよう、登録や承認済みへの変更は、1語ずつ説明を確認した場合に限ります。複数語の一括承認は行いません。
- **台帳との照合だけでは、一般語の内輪転用を見逃す** — 実在する2つのリポジトリを手作業で調べたところ、独自用語の主な形は台帳にない新語ではなく、「配線する」「drain する」「settle する」のような一般語の内輪転用でした。これは単純な辞書照合では見つけられません。そのため、台帳にない語、一般語の比喩的な転用、使用箇所から定義へたどり着けない語の3つを調べます。

さらにその後の実走（2026年7月）で、同じ「最初の指示が効きにくくなる」穴が質問だけでなく**報告**（進捗・完了・確認事項の提示）にもあると分かりました。内部の整理メモの文章がそのまま報告に貼られ、整理番号や作業単位名が文の主役になり、利用者に意味が通らなかったのです。そこで質問と同じ「出す直前の点検」を、すべてのコマンドの報告工程にも置きました。内部メモは読み手に通じる言葉へ言い直し、整理番号は「何を・なぜ」が単体で通じる一文の後ろに参照として添えます。平易化のために番号を削ることはしません（報告から記録へ辿れなくなるのは、平易さと引き換えにできない損失のため）。この点検も、通じなかった実例をテスト用の判別例文として固定する事後の検査と対になっています。

### すでに独自用語が増えたプロジェクトを整える（term-drift）

予防と検出だけでは、すでに独自用語が増えたプロジェクトを整えられません。そこで、用語を見直す機能は独立したツール [term-drift](https://github.com/ijust/term-drift) として提供しています。intent-planner を使っていないプロジェクトでも利用できるようにするためです。

term-drift は、文書の走査、LLM による検出・分類・言い換え案の作成、判断済みの置換の機械的な適用と再検査を繰り返します。一般語の内輪転用は辞書照合だけでは見つけにくいため、候補の判断には LLM を使います。通常は人が意味の同じ箇所ごとに確認し、利用者が対象リポジトリ、現在の点検、特定の語など明確な範囲を任せた場合だけ、エージェントが低リスクな言い換えを判断できます。意味が確定できない箇所や、法務・セキュリティ・公開 API・実行時の挙動に関わる箇所は人へ戻します。適用は Git 管理下で同じ結果を再現できる機械処理として行い、外部へデータを送信せず、秘密ファイルを走査しません。

intent-planner から term-drift への依存は一方向に保ちます。term-drift 0.3.3 は、intent-planner がバージョンを固定して依存する npm パッケージとして標準導入されます。通常の CLI は、対象リポジトリとエージェントが決まったあと、term-drift の公式 installer または公式 update を呼び出します。`./.term-drift/` とエージェント専用スキルは term-drift 側が配置し、intent-planner はその内容を複製しません。旧オプション `--with-term-drift` も既存スクリプトとの互換性のため受け付けますが、指定しなくても配置されます。`--yes` が省略するのは、従来どおりルート文書への追記確認だけです。

導入状態は読み取り専用で確認し、`not-installed`、`ready`、`inconsistent` に分けます。`inconsistent` はさらに、安全に不足分だけを追加できる `additive-compatible`、検証済みの 0.2.3 / 0.2.5 / 0.3.0 など公式 update を試せる `update-attemptable`、自動処理しない `blocked` に分けます。`not-installed` なら install、`update-attemptable` なら update を行い、`ready` なら何もしません。`install-failed` はその回の処理結果として、現在の配置状態とは分けて表示します。公式処理が拒否または異常終了した場合も、同じ方法で処理後の状態を確認します。intent-planner が独自に巻き戻したり、term-drift 所有ファイルを修復・上書き・削除したりすることはありません。未知の版や 0.3.3 より新しい将来版にも自動追随しません。

term-drift 0.3.3 の新しい適用辞書は、判断主体を `human-approved`（人の承認）または `delegated-agent`（明示された範囲でのエージェント判断）として記録し、判断日時と委任範囲も残します。これにより、確認回数を減らしても、誰がどの範囲を判断したかを後から追えます。以前の辞書形式も読み取り可能ですが、判断主体は不明として報告されます。さらに、未ステージの差分がある追跡済み文書でも、現在の内容に置換元が一意に一致する場合は、対象外の差分を保ったまま適用して警告します。これらの判断規則、辞書検証、適用処理は term-drift の rules・skill・CLI が担当します。intent-planner は意味仕様を複製せず、公開されるファイル一式の互換性だけを検査します。

責務は三段に分けます。日常の予防はルート規約・質問と報告の直前点検・export 注入で続けます。project-local な term-drift 配置物がない環境だけ、`/intent-validate` の軽い `coinage-suspect` を互換性のために残します。配置物が見つかったら、validate は用語の意味や version/hash/skill の互換性を判定せず、通常 installer の `--dry-run` health 確認へ案内します。`ready` と確認された後だけ、term-drift 専用 skill が本格点検の入口になります。これにより、health の正本を installer inspector 一つに、詳細な用語点検の入口を term-drift 専用 skill 一つに保てます。つまり、普段の予防、意図文書の構造検査、専門の用語点検を、それぞれ別の役割として扱います。

これは、[Drift-watch](#drift-watch--外れきる前に捕まえる軽いフック) の「大きく外れる前に見つけるが、作業は止めない」という考え方を、用語にも適用したものです。

## 結論だけでなく、見直しに必要な根拠を残す

Intent には、「○○方式を採用する」のような結論だけが記録され、その結論に至った理由、制約、前提、トレードオフが抜けることがあります。結論は根拠から導き直せますが、結論だけを見ても元の根拠は復元できません。根拠が残っていなければ、あとで結論を否定する事実が見つかっても、何を手がかりに見直すべきか分からなくなります。

これは記憶圧縮の研究が「もろい記憶（brittle memory）」と呼ぶ失敗と構造的に同じです（Kwon, *Reclaim Evaluation: A Lossy Memory Is Worse Than an Empty One*, 2026）。会話やセッションを要約して持ち越すとき、既定では**結論（takeaway）を残し、それを導いた source（working＝根拠・入力）を捨て**ます。ところが結論だけ残った記憶は、訂正が来ても直す対象が無く、むしろ**空の記憶より悪い** — 空なら「分からない」と棄権するのに対し、結論だけの記憶は古い誤りを自信を持って言い続けるからです（誤った結論が attractor になる）。処方箋は単純で、**再計算できる source を残し、再導出できる結論を捨てる**（source-first）。同じ予算でも訂正可能性が回復します。

intent-planner は、新しい仕組みを増やさず、既存の3つの工程で根拠を扱います。

- **discover（予防・最上流）** — 意図を最初に言語化する局面で、結論に対し「それを導いた根拠（理由・制約・前提・トレードオフ）」を併走させるよう read-only で促し、根拠が辿れないものは推測で埋めず Open Questions へ逃がします（「推測は Open Questions へ」という既存規律を強める方向）。
- **validate（事後検出）** — `/intent-validate` に `groundless-conclusion` 検査軸を足します。結論だけで根拠が辿れない意図を「根拠なき結論の疑い」として read-only で名指しし、各指摘に**訂正可能性の観点**（その結論が否定する事実が来たとき根拠から再評価できるか）を添えます。これは造語検出（`coinage-suspect`）と**同型**で、判定は意味的（機械検査・必須フィールドの有無に寄せない）・候補提示まで・誤検知前提で停止しません。
- **writeback（昇格時）** — 学びを canonical へ昇格する局面で、**結論だけ昇格して根拠を捨てる**のを止め、根拠を併走させます。論文の completeness signal（「元 N 項目中 k 項目を保持」）の精神は、Intent の根拠が離散的に数えにくいことを踏まえ、昇格の節目だけ**質的なフラグ**（根拠が併走しているか）で薄く添えます（数では持ちません）。

制限は、ほかの読み取り専用検査（`coinage-suspect` / drift-watch）と同じです。問題の検出も根拠の候補も提示するだけで、正本（intent-tree / compass / packets）を自動では書き換えません。特に、**AI に根拠を自動で補わせない**ことが重要です。AI が作った根拠は、結論を後から正当化するための作り話になるおそれがあります。根拠を記入するのは、人が内容を確認したあとです。専用の必須欄は増やさず、Assumptions、Open Questions、L3 の設計意図、Decision Rule の Why / Consequences など、既存の場所に記録します。

## 決定前に、仮説を既存の記録と照合する

前節（結論と根拠）が扱うのは「既に下した**結論**に、それを導いた**根拠（履歴）**を残す」という事後・保存の問題でした。ここで扱うのは、その**逆向き**の問題です — まだ確定していない**仮説（暫定の確信）**に、それを**検証/反証する証拠**を当てる、事前・検証の問題。同じ「根拠」でも、片方は結論の rationale を**保存**し、もう片方は仮説の evidence を**探索**する。並立する別の軸です。

着想は世界モデルの研究から来ています（Cai ほか, *Beyond Next-Observation Prediction: Agent-Authored World Modeling*, 2026）。LLM エージェントの世界モデル訓練は従来「次の観測を予測する」ことを目標にしますが、それは**訓練ターゲットを「環境が何を返したか」で決めてしまう**誤りだ、というのが論文の主張です。提案手法 **Self-Probing** は逆に、エージェント自身に「今わかっていること（確信）」と「次の行動を変えうる未解決の問い」を言語化させ、その問いを起点に過去の記録から証拠を引いて、**意思決定に効く**ダイナミクスをターゲットにします。論文が確かめた重要な経験則は、**問いを少数（n=3 程度）に絞ると効き、増やしても頭打ち**ということ — 少数精鋭でないと意思決定支援にならない、という観察です。

intent-planner はこれを、訓練手法としてではなく（重みは更新しません）、**`/intent-packets` の決定地点で人間に意思決定支援を出す**形に翻訳します。packet を切る・decision slot を埋めるとき、エージェントは「map をコピーして出力先を変えるだけ」のような暫定の確信で素通りしようとします。そこで `rules/decision-probe.md` が、(1) 自分の仮説と問いを言語化し、(2) 問いを起点に `.intent/`（compass の Invariant/Decision Rule・glossary・過去 deltas・関連 packets＝**既に貯まった証拠の台帳**）から証拠を pull し、(3) 確信と矛盾する証拠を**第一に** read-only で名指しします（「あなたの確信は INV9 と出力先閉じ込めに抵触する」）。これは [損失付き射影](#損失付き射影--仕様意図コードは互いに情報を落とす) の落差検出を、**実装の前ではなく決定の瞬間**へ前倒しした一手です。

この確認を有効にするには、対象を絞る必要があります。論文で問いを3つ程度に絞った場合に効果が高かったことと同じ考え方です。対象は、あとから覆すと影響が大きい決定に限定し、`.intent/` に実際の証拠がある問いだけを扱います。こうすることで、記録が少ない新規リポジトリで何も見つからない確認、誤った確信の追認、すべての決定に質問が増える状態を避けます。残るのは、既存の記録に判断材料があるのに、それを見ずに進みそうな決定だけです。

この確認を通らなかった仮説は、あとで `/intent-validate` の `unverified-hypothesis` が検出します。`groundless-conclusion` が「結論の理由をたどれない」ことを見るのに対し、こちらは「仮説を支える証拠がない」ことを見ます。いずれも読み取り専用で候補を提示するだけです。正本を自動で変更せず、証拠が見つからなくても export や実装を止めません。

## Enforcement — 手続きの実行だけを検査する

enforcement が保証するのは「writeback という**手続きが実行されたこと**」だけで、書き戻した内容の正しさではありません。これはプロセス品質とプロダクト品質の古典的な区別で、lint や CI ゲートと同じ設計です: 機械的に安く検査できる範囲に検査を限定し、内容の正しさは `/intent-improve` と人間レビューに委ねます。

staleness 検知の誤検知を認めたうえで既定を off にしているのも、検査理論の定石に沿っています。偽陽性の多いゲートは「ゲートを常に迂回する習慣」（`--no-verify` の常用）を生み、検査自体を無効化するためです。

## 進捗の表現 — 状態は持つが、状態機械は持たない

packet には、現在の進行段階を示す **state** を記録します。ただし、その state を自動で進める **state machine** は持ちません。この違いが、進捗管理の基本です。

- **state を記録する** — 現在の進行段階を draft / ready / implementing / verifying / done のいずれかで表します。これは、要求に状態属性を持たせる ISO/IEC/IEEE 29148 の考え方を簡略化したものです。
- **state machine で動かす** — 遷移規則や条件を持つ実行機構が、状態を自動で変えます。intent-planner はこの機構を持ちません。

進行段階は AI の自己申告だけで決めず、人の確認や intent-validate / drift-watch の検査結果を根拠に記入します。`state=done` にするには、`## Evidence` に確定した検証結果が必要です。テストが通っただけで自動的に done へ変えるのではなく、人または検査が確認し、その結果を記録したうえで done にします。

つまり、「状態機械を持たない」は「状態を記録しない」という意味ではありません。frontmatter の `state`、export-log、drift-log などに進行状況は残します。持たないのは、その状態を遷移規則や条件に従って自動で変える実行エンジンです。

state を次へ進める役割は、intent-planner の外に置けます。人が `/intent-status` を見て次の操作を依頼しても、`/loop` のような外部の実行ループが繰り返しても構いません。どちらの場合も、次に何を作るかは Intent Tree、Compass、packets と status が示す「次の一手」によって決まります。自動進行を製品に内蔵しないことで、人が任意の時点でレビューできる軽さを保ちながら、意図駆動の開発を実現します。

図にすると、駆動エンジンは製品の外側（破線の枠）にあり、製品（実線の枠）は state を記録し「次の一手」を指すだけです。

```
        ┌─────────── 外側のループ（駆動エンジン）───────────┐
        │     人手のレビュー  ／  /loop 等のハーネス        │
        │                                                  │
        │   次の一手を拾って実行 ──┐          ┌── 学びを戻す │
        └──────────────────────────┼──────────┼────────────┘
                                    ▼          │
   ┌──────────────────── intent-planner（製品）─────────────────┐
   │                                                            │
   │   Intent ──> /intent-status ──> 「次の一手」を1つ指す        │
   │  (tree/compass/packets)              │                     │
   │      ▲                               ▼                     │
   │      └──── /intent-writeback <── 実装（export 先で進む）     │
   │                                                            │
   │   製品が持つもの: state の記録（frontmatter・log）＋次の一手  │
   │   製品が持たないもの: 遷移を自律実行する状態機械             │
   └────────────────────────────────────────────────────────────┘
```

遷移（次の packet へ進む・state を `done` にする）を**実行**するのは外側のループ＝人または `/loop` です。製品は state を記録し、Intent から「次に何をすべきか」を指すところまでを担います。この分担が「状態は持つが状態機械は持たない」の実体であり、駆動を外へ出せること自体が IDD を軽量に成立させています。

状態の**ストレージ**も同じ規律に立ちます。state を frontmatter（YAML）のスキーマ — キー固定・値域固定・未確定キーは空値保持 — で持つのは、実質「スキーマ付きレコード」であり、DB を導入せずに state を規律する形です。grep / git diff / 人の可読性を保ち、依存ゼロ原則（INV2）を守ります。「state を正式に持つ」とは DB 化することではなく、スキーマと制約で規律することだ、という立場です。

### 進捗は単一の％にしない — 完成度と流れを分ける

進捗を1つの総合値にすると、性質の違う情報が見えなくなります。そこで `/intent-overview` は、進捗を次の3軸に分けて示します。

- **意図の安定度** — 目的や判断基準がどこまで定まっているか
- **実現の完了度** — 実装がどこまで終わっているか
- **証拠の確定度** — 完了や成果を裏付ける証拠がどこまで揃っているか

この分け方なら、「実装は進んでいるが証拠が未確定」「証拠は揃ったが上位の意図と合っていない」といった状態を、そのまま表せます。

`/intent-overview` は、指定された範囲について、コードから対応する意図をたどれない領域も一覧にできます。packet の対象範囲、Invariant の影響先、コミットと意図の対応を調べ、どの記録からも意図を確認できない領域を根拠付きで示します。意図が記録されていないこと自体を誤りとは断定せず、見直す場所を決める材料として扱います。

必要なときだけ生成する表示には、ほかにも次のものがあります。

- **判断待ち一覧** — 未回答の Open Questions、承認待ちの delta、追跡中の警告を、出所へのリンク付きでまとめます。
- **ロードマップ表示** — 現役の packet の順序、状態、依存による停止、保留を、日付なしで1枚にまとめます。
- **Mermaid 図** — 意図の全体像と、`depends_on` による作業の前後関係を GitHub や VS Code で描画できる形にします。
- **担当状況** — `.intent/assignments/` の宣言から、誰がどの packet を実装中か、同じ packet へ複数の担当が付いていないかを示します。注意は出しますが、作業は止めません。
- **引き継ぎ文書** — 新しいセッションへ移るときに、現在地、残作業、正本への参照、注意点、リポジトリの状態、確認方法を短くまとめます。

技術的な依存関係だけでは、「まずログイン、次にリマインダー」のような人が決めた優先順を表せない場合があります。その場合は、`.intent/packets/plan.md` に任意の工程計画を書けます。見出しは `Phase`、`step`、`期` など自由で、入れ子にもできます。番号の順を優先順とし、同じ番号は並行して着手できます。ツールは上から順に候補を選び、完了済み、依存が未解決、ほかのセッションが着手中の packet を飛ばします。工程計画と `depends_on` が矛盾する場合は、技術的な依存関係を優先します。順序の強制や点数化はしません。

引き継ぎ文書は `.intent/` に保存された状態から生成します。会話履歴の要約や自動圧縮は実行環境の役割であり、intent-planner は会話ログやトークン数を扱いません。引き継ぎ文書は生成時点を明記し、正本の方が新しければ正本を優先するよう案内します。未コミットの変更や並行セッションなど、その時点だけの情報を含められるよう、Git で追跡しない `.intent/handoff/` に置きます。

これらはすべて、正本を変更せずに生成する一時的な表示です。日付、ガントチャート、ベロシティ、総合進捗率は持たず、intent-planner をプロジェクト管理ツールへ広げない方針を守ります。

軸を分けるもう1つの理由は、Goodhart の法則と Campbell の法則への配慮です。どちらも、代理の指標を目標にすると、本来の目的からずれる危険を指摘しています。AI エージェントの研究でも、不完全な報酬指標を最適化しすぎると本来の性能が下がる現象が、reward hacking や overoptimization として報告されています。

これをソフトウェアの進捗へ当てはめる部分は、測定と管理に関する一般理論からの推論です。「進捗速度を目標にすると必ず指標が操作される」と直接示した定番の一次文献があるわけではありません。それでも、単一の進捗値だけを目標にすると、簡単な作業を増やして数値を高く見せる動機が生まれます。そのため、少なくとも証拠の確定度は独立して確認し、AI の自己申告だけで完了を判断しません。

## AI に進捗・完了を委ねるときの落とし穴 — 自己申告を信じない

AI エージェントに工程・完了の管理を委ねると、人間同士の開発では目立たなかった固有の失敗様式が現れます。intent-planner の進捗・完了の設計は、これらへの防御として組まれています。

- **進捗のでっち上げ・完了の過大申告（hallucinated progress）** — AI エージェントは、未実装や不具合が残っていても完了と報告することがあります。Cao らの手続きを考慮した評価（2026）では、τ-bench 上で GPT-5、Kimi-K2-Thinking、Mistral-Large-3 を調べたところ、通常は成功と判定される結果の約27〜78%に手続き違反が含まれていました。ここでいう27〜78%は、6つの観点で手続きを検査した場合に成功扱いから外れた割合で、モデルと課題に依存する値です。intent-planner では、`state=done` を `## Evidence` に記録した検査結果や人の確認に基づいて決めます。検査 ID は証拠そのものではなく、どの検査で確かめたかを示す参照です。
- **状態をファイルへ保存しても、正しさまでは保証されない** — 意図や決定を会話の中だけに置くと、セッションが変わったときに失われます。`.intent/*.md` へ保存すれば、長期保存、共有、参照はできます。ただし、内容が正しいか、最新か、正本が1つに定まっているかは別の問題です。そこで、書き込み可能な正本と、そこから再生成する読み取り用の表示を分け、人の承認を設けます。`/intent-overview` が正本を変えずに表示を再生成するのは、この考え方に基づきます。
- **重要な節目は、人または独立した検査で確認する** — 完了、採用、仕様全体の更新は、AI の報告だけで通しません。writeback では、delta への記録、人の承認、正本への反映という順序を守ります。正本を書き換える discover、compass、packets、writeback、improve、export は、利用者が明示的に起動した場合だけ実行します。一方、正本を変えない status、validate、overview、from-spec、to-spec は、文脈に応じて起動できます。自動起動を読み取りと派生物の生成に限ることで、人の承認なしに正本を書き換えない原則を守ります。

> **数値の読み方**: この節で参照した AI エージェント評価は、2024年以降に増えた新しい研究です。27〜78%という値は、τ-bench で6つの手続き観点を適用した場合の条件付きの結果であり、課題やモデルによって変わります。「あらゆる成功の27〜78%に問題がある」という一般則ではありません。intent-planner が採用するのは、特定の数値や手法の優位性ではなく、AI の自己申告だけを完了の根拠にしないという原則です。

## モード — 状況に応じた方法の組み替え

単一の手順を万能とせず、状況に応じて手順を組み替える発想は、要求工学で **situational method engineering** と呼ばれる領域に対応します。各モードにも個別の理論的背景があります。

- **behavior-unknown** — Michael Feathers（『レガシーコード改善ガイド』）の**仕様化テスト（characterization test）**: 仕様が失われたシステムでは、現在の観測可能な振る舞いを記録し、それを暫定的な仕様として固定します。「観測できる振る舞いから意図を逆算する」はこの文書版です。
- **refactor** — Fowler の behavior-preserving と、Perry & Wolf の drift 是正。「正しく動くが設計意図からズレる」変更を防ぐには、保存すべき意図を先に言語化しておく必要があります。
- **feature-growth** — 既存システムへの変更が及ぼす影響の分析（impact analysis）。追加単位の分解は、既存の invariant を壊さない範囲で切ることが中心になります。

### モードで使う分析・分解手法

モードは状況を分類するだけでなく、discover（意図を探る段階）と packets（実装へ渡す作業単位に分ける段階）で使う分析・分解手法を組み替えます。ここでいう手法は、外部の方法論をそのまま再現するものではありません。対話とリポジトリの読み取りだけで、実装前の意図計画に使える部分を軽量化しています。

**Example Mapping** は、Matt Wynne が提案した、ストーリーをルール・具体例・疑問・別に切り出す項目へ整理する対話手法です。具体例は受入テストの土台になり、疑問は「まだ理解できていないこと」を可視化します。intent-planner では物理的なカードや会議を必須にせず、L2/L3 の能力を「ルール・例・疑問・切り出し」に展開します。例は packet の Expected Behavior と Validation、疑問は Open Questions、切り出しは先送りの候補へ流します。これは認識を揃えるという元の狙いを保ちながら、AI と人の非同期な対話にも使える形へ移したものです。

`feature-growth` の **Impact Analysis** は、Bohner & Arnold の Change Impact Analysis（変更影響分析）を基礎にします。この分野では、変更によって起こりうる結果を特定し、変更のために修正が必要な範囲を見積もります。intent-planner は静的な依存解析ツールを内蔵せず、変更の起点から既存の境界・契約・データフローを読み、影響候補を一段ずつたどります。したがって出力は網羅性を保証した依存グラフではなく、後の段階で接合箇所を決められるだけの根拠を添えた影響候補です。

`refactor` の **Migration Slicing** は、Ola Ellnestam と Daniel Brolund の Mikado Method を分解前の整理に使います。目標から「その変更を安全に行うため、先に何が真である必要があるか」を逆算し、前提を Mikado Graph（前提グラフ）として並べ、前提のない末端項目（葉）から着手します。元の手法では、試しに変更して失敗したらすぐ戻す実践も重視しますが、intent-planner は計画段階ではコードを変更しません。この考え方を、各スライスの Rollback と、振る舞いが保たれたかを確かめる検証点として残します。そこから差分を、振る舞いを保ち、単独で検証・取り消しできる単位へ切る部分は、Fowler のリファクタリングと小さな変更単位の考え方を組み合わせたものです。

`feature-growth` の **Additive Slicing** は、既存の単一手法をそのまま採用したものではなく、段階的な変更パターンを intent-planner が組み合わせた設計です。まず Branch by Abstraction、Feathers の差し替え箇所（seam）、Parallel Change（expand–migrate–contract）を使い、既存の振る舞いを変えずに新実装を差し込める接合面を作ります。次に、既存コードからまだ到達されない新機能を追加し、最後に接合面を通して有効化します。大きすぎる候補は Mike Cohn の SPIDR（Spike / Paths / Interfaces / Data / Rules）を分割の手がかりにします。つなぎ込みには Pete Hodgson の Release Toggles を使い、リリースとコード配置を分けたうえで、初期状態では無効にする範囲と切り替え機構の削除条件を計画します。「接合面の確立 → 追加 → つなぎ込み」という三段の順序そのものは、これらの手法を安全な新機能追加向けに組み合わせた intent-planner の設計判断です。

### モードの保存先 — 個人の作業状態と、チームの設定を分ける

モードに関する情報には、性質の異なる2種類があります。1つは、選んだ mode、purpose、designer-questions など、開発者やセッションごとに変わる作業状態です。もう1つは、Enforcement / Drift-watch のように、チーム全員で共有する設定です。両方を同じファイルに書くと、並行するセッションや複数の開発者が互いの作業状態を上書きしてしまいます。

作業の進め方は Git で追跡しないローカルファイルに、チームの設定は Git で追跡する `.intent/mode.md` に分けて保存します。これにより、個人の作業状態は共有履歴に混ざらず、チームの設定変更はコミットを通じて全員に届きます。

ただし、1人が同じリポジトリで複数のセッションや worktree を並行して使うと、ローカルファイルが1つだけでは、後から実行した `/intent-discover` が先のセッションの設定を上書きします。そこで、`/intent-discover` の実行ごとに `.intent/discovery/<スラッグ>-<rand>/mode.md` を作り、セッションごとの状態を分けて保存します。末尾のランダムな4文字により、中央で連番を管理しなくても同名のディレクトリを避けられます。後続の compass / packets などは、discover が出力したディレクトリ名を引き継いで `mode.md` を読みます。従来の `.intent/mode.local.md` も互換性のため引き続き読み取れます。

## cc-sdd 連携 — 仕様駆動開発と EARS

**仕様駆動開発（spec-driven development）**は、AI エージェントに requirements → design → tasks の段階的な成果物を作らせ、人間が各段階でレビュー・承認してから次へ進む方式です。AWS の Kiro（2025）が広め、[cc-sdd](https://github.com/gotalab/cc-sdd) はその Claude Code / Codex 向け実装です。cc-sdd の requirements が用いる **EARS（Easy Approach to Requirements Syntax）** は、Rolls-Royce の Alistair Mavin らが提案した（RE'09）、自然言語要求の曖昧さを少数の構文テンプレートで抑える記法です。

intent-planner が spec そのものを生成せず「下書きまで」に留めるのは、冒頭で述べた問題空間と解空間の責務分離です。1つのツールが両方を担うと、解の都合（実装のしやすさ）が問題定義の側へ逆流しやすくなります。

export 先は cc-sdd に限りません。[OpenSpec](https://github.com/Fission-AI/OpenSpec) には `/intent-export-openspec` で、Why / What Changes / Impact を含む proposal の下書きと、ADDED / MODIFIED / REMOVED、Requirement / Scenario のヒントを作り、`/opsx:propose` へ渡します。[GitHub Spec Kit](https://github.com/github/spec-kit) には `/intent-export-speckit` で仕様の下書きを作ります。

どの export でも、入力は1つの packet と関係する compass に限定し、書き戻し漏れの検査、export の記録、下流ツールの起動までという共通の流れを使います。ツールごとに異なるのは、情報の対応付けと出力形式だけです。intent-planner の担当が「下書きまで」であることは変わりません。

**出口は、案件の種類と導入済みのツールから選びます。** `/intent-packets` は cc-sdd を常に勧めるのではなく、`format` の指定、選択中の mode、下流ツールの導入状況を確認します。cc-sdd は `.kiro/`、OpenSpec はリポジトリ直下の `openspec/`、Spec Kit はリポジトリ直下の `.specify/` の有無で判断します。確認は読み取り専用です。

`format` に `cc-sdd` / `openspec` / `speckit` / `to-spec` / `direct` が指定されていれば、それを優先します。指定がなければ、案件の種類と導入状況から候補を示します。1つに決められない場合は、複数の候補を並べて人に選んでもらいます。この判定規則は `intent-packets/rules/export-route.md` にまとめ、各 export スキルから共通して参照します。

候補は導入済みのツールを先に並べ、未導入のツールには「導入が必要」と添えます。ただし、未導入だから候補から除外したり、export を止めたりはしません。インストールの代行も行いません。複数のツールが導入済みなら、ツール間の優先順位は付けず、人が選びます。導入状況はディレクトリの有無だけで確認し、複雑な自動診断は行いません。

各 export スキルは、必要なディレクトリが見つからない場合に警告しますが、処理は止めません。`direct` を選んだ場合は、その記録を実装後の `/intent-writeback` が使い、仕様作成ツールを経由しない案件でも対象の packet を特定します。記録がない場合は、`spec_refs`、export-log、`state=done` から推測します。

## 外部連携の姿勢 — 片方向2本に留め、双方向同期を背負わない

Notion / Jira / Slack などとの連携は、2つの片方向の操作で行います。外部文書をエージェントが読んで intent に渡す操作と、intent が作った Markdown を人またはエージェントが外部ツールへ貼る操作です。この方法なら、intent-planner に追加機能を導入せず、常駐処理や自動送信も増やさずに連携できます。

一方、双方向の自動同期は行いません。自動同期を始めると、外部ツールと intent のどちらを正本とするか、両方が同時に編集されたときにどう解決するかを、intent-planner が継続的に管理する必要があります。これは、意図を Markdown で軽量に管理し、人が承認した内容だけを書き戻す現在の方針と合いません。そのため、双方向同期は将来の予定にも含めません。具体的な連携方法は [docs/integration.md](integration.md) で説明します。外部サービスへ貼り付ける前には、機密情報が含まれていないか確認するよう案内します。

## 設計姿勢 — 理論はフローに埋め込む

intent-planner は要求工学を**教える**ツールではありません。経験のある設計者・要求アナリストなら自然に問うこと — 成功をどう計測するか、最初に E2E を貫くか、何が観測されたら仮説を棄却するか — を、フロー上の質問と検査として埋め込み、理論を知らなくても流れに沿えば同じ問いを通過するように作られています。このドキュメントは、その埋め込まれた問いの出典を示すものです。
