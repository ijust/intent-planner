# 外部ツールとの連携（Notion / Jira / Slack など）

このドキュメントでは、intent-planner と Notion、Jira、Slack などの外部ツールを連携させる方法を説明します。README は「何ができるか」という全体像に絞っているため、具体的な手順と設計上の理由はこちらにまとめています。

> 各機能の使い方は [docs/guide.md](guide.md)、設計の背景は [docs/theory.md](theory.md) を参照してください。

## 前提：追加インストールは要りません

intent-planner 自体は外部サービスへ接続しません。常駐プロセスはなく、外部へのデータ送信も行いません。外部ツールとの連携には、**ふだん使っているエージェント（Claude Code / Codex / Gemini CLI）の読み書き機能**を使います。たとえば MCP 経由でエージェントが対象ツールを読める状態なら、そのまま連携できます。

MCP サーバの導入や認証など、外部ツール側の接続設定は、各ツールとエージェントの公式ドキュメントに従ってください。外部ツールの仕様は変わるため、ここでは特定のバージョンに依存しない連携方法だけを示します。

## 2つの片方向の流れとして連携する

intent と外部ツールの間に自動同期は設けず、次の2つの片方向の操作として扱います。どちらも、人またはエージェントが内容を読み、渡すか貼り付けるだけです。

### 1. 外 → intent（外部の文書を読んで intent に渡す）

外部ツールにある PRD・issue・議事録などをエージェントに読ませ、その内容を intent の入口に渡します。

- **Notion の PRD から意図を起こす**: 「この Notion ページの PRD を読んで、`/intent-from-spec` に渡して意図候補と沈黙ギャップを出して」とエージェントに頼みます。エージェントがページ本文を読み、そのまま from-spec の入力にします。
- **Jira / GitHub の issue から始める**: 「この issue を読んで `/intent-discover` の材料にして」と頼み、課題の背景を intent Tree の出発点にします。
- 外部文書は、意図を整理するための**読み取り材料**として渡します。この操作で、intent 側から外部ツールへ内容を書き戻すことはありません。

### 2. intent → 外（intent の成果物を外部に貼る）

intent がまとめた Markdown の成果物を、人またはエージェントが外部ツールへ貼ります。

- **一枚もの・報告を Notion / 共有ドキュメントへ**: `/intent-to-spec` で作った読み手向けの文書（`.intent/nl-spec/` の Markdown）を、そのまま Notion ページや共有ドキュメントに貼ります。
- **俯瞰ビューを共有する**: `/intent-overview` で作った俯瞰（`.intent/overview/` の Markdown）を、チームの資料として貼ります。
- **Slack へ進捗を報告する**: `/intent-status` の要約や overview の一部を Slack に貼って、いまの状態と次の一手を共有します。エージェントに「この overview を Slack 用に短くまとめて」と頼んでもよいです。
- ここでも intent は Markdown を出すだけで、貼る操作は人かエージェントが行います。

## 貼る前の確認（機密の取り扱い）

外部サービスへ内容を貼る前に、**機密情報や個人情報が含まれていないかを確認してください**。一度外部サービスへ渡した内容は、あとで削除しても、バックアップや検索索引に残ることがあります。安全かどうか判断できない場合は、貼り付けないでください。

## なぜ双方向の書き込み同期をやらないのか

「Notion の PRD を書き換えたら intent 側も自動で更新され、逆に intent を直したら Notion も書き換わる」——このような**双方向の自動同期は、恒久的にやりません**（将来やる予定も置きません）。理由は次のとおりです。

- **出所（どれが正か）の管理を製品が背負わないため**: 双方向同期は「外部ツールと intent のどちらの版が正しいか」を常に判定し続ける責務を生みます。intent-planner は意図を Markdown の射影として持つ軽量なレイヤーであり、複数ツール間の正本管理という重い責務を持ちません。
- **衝突解決を製品が背負わないため**: 両側を同時に編集したときの衝突をどう解くか（どちらを勝たせるか・マージするか）は、自動化すると意図せぬ上書きを生みます。intent-planner は「人が承認した分だけ反映する」を基本規律にしており、自動の書き戻し・自動同期はこの規律と相容れません。
- **2つの片方向の操作で必要な連携を実現できるため**: Notion 上の PRD を intent に渡し、intent の成果物を Notion に戻す流れは、「読んで渡す」と「出力を貼る」の組み合わせで実現できます。

つまり、intent-planner は双方向の自動同期を行いません。代わりに、「外部文書を読んで intent に渡す」と「intent の成果物を外部へ貼る」という2つの操作を使います。
