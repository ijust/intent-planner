# Journey Plan（ジャーニーの起案・更新）

複数 packet 案件で、ジャーニー（複数の packet にまたがる1つの変更を束ねる単位）を `.intent/packets/journeys/` へ起案・更新する手順。`/intent-packets` の Step 4（優先順位の提示）で使う。スキーマ・置き場の規約の正本は `.intent/packets/journeys/README.md`、読み手の契約は `skills/CONTRACT.md` の「ジャーニーの読み取り契約」（このファイルへ複製しない）。

## 発火条件（すべて満たすときだけ・満たさなければ沈黙して従来どおり）

- **今回の案件の packet が2つ以上**である（新規起案が2つ以上、または既存ジャーニーの構成へ packet を足す更新）。**単発 packet 案件では提案しない**（数合わせ・儀式化をしない）。
- `.intent/packets/journeys/README.md` が存在する（無い旧環境では発火せず、従来どおり plan.md への記録だけを行う＝後方互換・INV103）。
- designer-questions / mode の値に依存しない（複数 packet かどうかだけで判定する）。

## 手順

1. **利用者に1問で提案する**: 「この案件は複数の packet に分かれるので、束ねるジャーニーを `.intent/packets/journeys/` に起案しますか」（承認／不要／後で確認 を選べる形・回答を強制しない）。**不要・保留なら何も作らず従来どおり進む**（plan.md への記録は従来のまま）。
2. **承認されたら起案する**: `journeys/<スラッグ>.md` を新規作成する。
   - `journey_id` は `jny-<YYYYMMDD>-<スラッグ>-<rand>`（日付部はシェルで取得・`<rand>` は `[a-z0-9]` 4文字をシェルで生成・packet ID 規則と同型）。スラッグは `name` から `rules/packet-format.md` のスラッグ規則で導出する。
   - frontmatter は **7キー固定**（`journey_id` / `name` / `lifecycle: active` / `packets`＝構成 packet の packet_id 列挙 / `created_at` / `updated_at`＝作成時は同値 / `summary`）。**これ以外の必須キーを足さない**。
   - 本文は自由記述。工程計画（順序・依存）・共有契約（複数 packet が守る契約と統合時の検査）・後回しの記録など、plan.md のジャーニー節に書いてきた内容をここへ書く。
   - 同名スラッグのファイルが既にあるときは、黙って上書きせず「別の案件か・この案件の更新か」を利用者に確認する。
3. **plan.md への併記はしない（2026-07-17 の移行で終了）**: これは**承認されてジャーニーを起案した後**の書き先の定めであり、手順1で不要・保留を選んだ（ジャーニーを作らない）場合に plan.md へ従来どおり記録することとは矛盾しない。工程計画・共有契約の一次の置き場はジャーニー正本で、plan.md のジャーニー節は参照スタブ（派生）になった。他の rule が plan.md へ書くと定めているもの（Walking Skeleton の確認記録・Recommended First Packet 等）は従来どおり（この rule はジャーニー本文の書き先だけを定める）。plan.md 経路の読み取りは恒久フォールバックとして残る（INV103）。

4. **既存ジャーニーの更新**: 構成 packet の追加・工程の変更があれば、該当ファイルの `packets` 列挙と本文を更新し、`updated_at` を打刻する（**内容変更を伴わない再実行では打刻しない**＝冪等）。
5. **閉じるのは人**: `lifecycle: archived` は「構成 packet がすべて done ＋ 統合時の検査が green」を人が確認した宣言でのみ記入し、ファイルを `journeys/archive/<年>/` へ移す（削除しない）。機械が自動で閉じない・経過日数などの機械閾値を持たない。

## 書いてはいけないもの（読み手の導出を壊さない）

- **進捗の state を frontmatter に書かない**（`state:` キー・進捗率・done 数などを持たせない。進捗は読み手が構成 packet の `state` から毎回導出する＝DR200）。
- **packet 側の frontmatter（12キー固定）へジャーニーのキーを足さない**（参照は journey→packet の一方向のみ＝DR203。逆引きは journeys/ の grep で足りる）。
- 数値スコア・日付コミット・ベロシティを持ち込まない（工程は順序と依存だけで表す・INV81）。
- 「ジャーニー」以外の新語（フェーズ・エピック等）をここから増やさない。

## 温度

- これは提案であって gate ではない。ジャーニーを作らない選択はいつでも有効で、作らなくても packets 工程は従来どおり完了する（INV103）。
- 実装コードを変更しない（これは計画の記録手順であり、実装の道具ではない）。
