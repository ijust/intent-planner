# Walking Skeleton Check

最優先 packet が主要なユーザージャーニーを端から端まで貫く（walking skeleton である）かを確認する手順。`/intent-packets` の優先順位提示（Step 4）で、`.intent/mode.md` の designer-questions が `on` のときのみ使う。対話はすべて利用者への確認として行う（確認の手段は SKILL.md の規約に従う）。

## 適用条件

- `.intent/mode.md` の `designer-questions` を読む。`on` のときのみ、Step 4 の優先順位提示に本手順を適用する。
- **designer-questions が on と記録されていない（off・未記録）ときは適用せず、既存 Step 4 の挙動を変えない**（未記録の場合の告知は `.intent/mode.md` の規約に従う）。
- **purpose は参照しない。**

## 手順

1. **E2E 判定**
   - 最優先 packet の Scope と Expected Behavior を読み、主要なユーザージャーニーを端から端まで（入力 → 処理 → 観測可能な出力）貫くかを判定する。
   - 判定の根拠は packet の記述に求める。途中の層で止まる packet（処理だけ・UI だけ等）は「貫かない」と判定する。
   - **「主要なユーザージャーニー」は、利用者がいま実際に行っている仕事の置き換えとして読む**。技術の層を端から端まで通っていても、利用者の仕事が何一つ置き換わらない骨格は「貫く」と数えない。intent-tree の Current Drift（いま何が手作業・回り道になっているか）を読み、**この骨格が完成したとき利用者のどの手作業が消えるか**を判定の根拠にする。
     - 例（架空）: 「入力欄を全部埋めて出力する」経路が技術的に通っていても、利用者の手作業（別のツールへ資料を貼り、書き写す）がそのまま残るなら、置き換えは起きていない。「資料を投入すると下書きが出る」まで通って初めて、その手作業が消える。
     - **Current Drift が tree に書かれていない案件では、この基準による判定をスキップし、従来の貫通判定だけを行う**（書かれていないものを推測で補わない。スキップした旨は判定根拠に明記する）。
     - 非アプリの案件（文書・調査・ツール等）で「利用者の仕事の置き換え」が定義しにくいときも、従来の貫通判定へフォールバックし、断定しない。

2. **判定結果と根拠を提示して確認する**
   - 判定結果（貫く / 貫かない）とその根拠を利用者に提示し、確認する。
   - 根拠は平易な言葉で提示する: この packet（cc-sdd に渡す作業単位）が何を作り、完了後に何がどこまで動くかを述べ、欄名を知らなくても判定根拠が読めるようにする。
   - **除去する Current Drift を併せて示す**: この骨格が完成したとき利用者のどの手作業が消えるかを一言で述べる。**最も重要な Current Drift を1つも除去しない骨格には、その旨を warn として添える**（「技術的には通りますが、利用者の手作業は何も置き換わりません」）。警告であって停止ではない。

3. **是正案を提示する（「貫かない」と判定した場合・最重要 Current Drift を除去しない場合）**
   - 最初の packet を walking skeleton にする案を、次のいずれかで提示する:
     - **並べ替え案**: E2E を貫く packet が最優先になるよう優先順位を並べ替える。
     - **統合案**: 複数 packet の Scope を統合し、E2E を貫く packet を作る。
   - 利用者が walking skeleton 化を意図的に見送る場合は、それも選択として受け付ける（黙って落とさず記録する。手順 4）。**最重要 Current Drift を除去しない骨格を承知で選ぶことも、理由付きの選択として受け付ける**（新しい是正の枠は作らず、既存の見送り記録に乗せる）。

4. **確認結果を記録する**
   - `.intent/packets/plan.md` の「Walking Skeleton」セクションに記録する: **最優先 packet**（packet 名）/ **E2E 判定**（貫く / 貫かない）/ **除去する Current Drift**（この骨格が完成したとき消える利用者の手作業。判定をスキップした場合はその旨）/ **確認結果**（利用者の確認内容）。
   - 意図的な見送りは、その理由とともに Deferred セクションへも記録する。
   - **旧 scaffold への非破壊追記**: plan.md に「Walking Skeleton」セクションが無ければ、既存の記録内容を保持したままセクションを追記してから記録する。**「除去する Current Drift」を持たない過去の記録は「未記入」として読み、遡って警告しない**（後方互換）。

5. **早期に見せられる中間物を案内する（放っておくと最後まで途中経過が見えないのを防ぐ）**
   - packets の工程が一段落したこの地点は、**利用者が実装前に途中経過を見られる最初の機会**である。実装に入る前に、案件の種別に応じて次のいずれかを見せることを1〜2行で案内する（新しい成果物は作らず、既存の器を案内するだけ）:
     - **アプリ・動くものが作れる案件**: 最初の骨格（walking skeleton）が動いたら、その時点で利用者に見せる。
     - **非アプリ・計画段階の案件**（文書・調査・ツール等）: Intent Tree の俯瞰ビュー（`/intent-overview` が生成する `.intent/overview/`）や工程計画（`.intent/packets/plan.md`）を見せる。
   - これは**案内であって強制ではない**。利用者が「今は要らない」と言えばそれで良い。既存の「次の一手」「乗り換えの促し」と同じ出力位置に複数の促しを並べない（1つに絞る・うるささを避ける）。
   - **見せた（または見せる約束をした）ら、その事実を plan.md の「Walking Skeleton」セクションへ1行記録する**: `早期提示: <何を・いつ>`（例: `早期提示: 最初の骨格が動いた時点でプレビューを見せる（2026-01-01）`）。新しいファイル・新しい節は作らず、既存の Walking Skeleton 節に1行足す。この記録の有無を export の水際（`intent-export-cc-sdd` の preflight）が読み、記録が無ければ一言 warn する（対になる）。
   - **designer-questions が off・plan.md に Walking Skeleton 節が無い環境では本手順を発火しない**（適用条件に従う。器が無いところで案内・記録を強制しない＝後方互換）。

## 規律

- 確認結果は intent-validate が参照する。記録を省略しない。
- コードを変更しない。
