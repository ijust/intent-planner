# db-design 射影出力形式（machine-diffable・暫定契約）

`intent-db-design` skill が、読み取った三層素材（意図 / invariant / 既存スキーマ）を DB 設計の叩き台へ写像する際の**出力形式の正本**。SKILL.md は手順と報告形式のみを持ち、「どの構造・どの見出し・どの表で出力し、どこへ書くか」は本ルールを参照する。トレース付与・inferred / unverified 標識の規律は `db-design-fabrication-guard` が担い（本ルールは形式の枠を定め、各記述の帰属注記がその枠に乗ることを前提とする）、対象特定と三層読み取りは `db-design-input` が担う。

本ルールが定める出力形式は、後続の検査（落差検出）が実装スキーマ（migration/DDL）と**項目単位で diff** できる構造を満たす。出力は `.intent/db-design/<packetスラッグ>/` 配下の派生物に閉じ、canonical（intent-tree / compass / packets）・既存スキーマ・export 下書き（`.intent/cc-sdd/` / `.intent/openspec/`）を一切変更しない。

## 形式の位置づけ（暫定契約・凍結しない）

この出力形式は「凍結された契約」ではなく、**最小限の突合可能性を満たす暫定形式**である。実際に突合する側（intent-validate の落差検出）はまだ別 spec で未設計のため、突合される側の形式を一方的に確定するリスクがある。

- 本形式は「後続 validate が項目単位で diff できる」という**検査可能性の床**を満たすところまでを責務とし、突合ロジック自体は持たない。
- 後続の落差検出 spec の設計時に、テーブル / カラム / 制約 / インデックスが実装スキーマと項目単位で diff 可能かを実地で検証し、不足なら本形式を改訂する（凍結しない）。
- したがって本形式は**安定した突合単位の見出し / 表構造**を提供することに集中し、意味判断（正規化妥当性・欠落インデックス・命名一貫性等）は構造に埋め込まない。意味判断は後続の検査オラクルへ委ねる（INV2 / A1・機械検査に寄せすぎない）。

## 出力ファイルと構造

対象 packet 1つにつき、`.intent/db-design/<packetスラッグ>/db-design.md` を1ファイル出力する（スラッグの導出は下記「出力レイアウト」）。構造は次の machine-diffable な Markdown とする。

```markdown
---
source_packet: "<対象 packet 名>"
generated_at: <ISO 8601>
projection_sources: [packet, compass-invariant, existing-schema]
---

# DB 設計（叩き台）— <packet 名>

> 派生物。`/intent-db-design` が生成。Git 非追跡。これは設計の叩き台であって要件ではない。

## テーブル: <table_name>   <!-- 突合単位＝見出し -->
- **由来**: <packet 由来 | invariant 由来 | 既存スキーマ由来 | inferred>
- **カラム**:
  | カラム | 型 | 制約 | 由来 |
  |---|---|---|---|
  | id | <型> | PK | inferred |
  | <列> | <型> | NOT NULL, FK→<参照> | packet 4.1 |
- **インデックス**: <列・種別・由来>
- **命名規則**: <スネーク / キャメル等・由来>

## inferred / unverified 一覧   <!-- 確認対象 -->
- テーブル <x> の <カラム y>：射影元に根拠なし（inferred）
- テーブル <z>：既存スキーマを同定しきれず未確認（unverified・実在しうる）
```

### 各要素の規律

- **frontmatter**: `source_packet`（対象 packet 名・正本キー）/ `generated_at`（ISO 8601・シェルで取得）/ `projection_sources`（射影に用いた層）を持つ。状態は frontmatter のスキーマ規律で持ち、永続ストア（DB 等）を導入しない（INV2）。
- **`## テーブル: <name>` 見出し**: テーブル1つ＝1見出し。これが**突合単位**であり、後続 validate はこの見出しを実装スキーマのテーブルと項目単位で対応づける。テーブルごとに見出しを分け、複数テーブルを1見出しに束ねない。
- **カラム表**: `| カラム | 型 | 制約 | 由来 |` の4列固定。1行＝1カラム＝**突合単位**。`由来` 列が各カラムの射影元帰属（packet / invariant / 既存スキーマ / inferred / unverified）を持つ（帰属の付与規律は `db-design-fabrication-guard`）。
- **インデックス**: 列・種別（unique / btree 等）・由来を識別可能な形で列挙する。後続 validate がインデックス単位で diff できるよう、どの列に対するどの種別かを明示する。
- **命名規則**: テーブル / カラムの命名規約（スネーク / キャメル等）と由来を明示する。
- **inferred / unverified 一覧**: 出力末尾に「確認対象」として一覧化する。`inferred`（射影元に根拠が無いことを確認したうえでの推論）と `unverified`（既存スキーマを同定しきれず未確認・実在しうる）を区別して並べる（両者を一括しない・区別の正本は `db-design-fabrication-guard`）。

## 検査可能性の床（後続 validate が突合できる構造）

- テーブル名・カラム・制約・インデックス・命名規則が**構造化された見出し / 表で識別**でき、実装スキーマ（migration/DDL）と項目単位で突合できることを満たす。
- 突合単位は次の2つ: テーブル＝`## テーブル:` 見出し、カラム＝カラム表の行。後続 validate はこの見出し / 行を実装スキーマと diff する。
- 構造に埋め込むのは「突合可能な識別子と帰属」までであり、正規化妥当性・欠落インデックス・N+1 誘発・制約漏れ・命名一貫性などの**意味判断は構造に落とさない**（INV2）。それらは後続の検査オラクルが担う。

## 出力レイアウト（スラッグ規則と衝突規則）

### スラッグ規則（決定的）

packet 名から出力先ディレクトリ名（スラッグ）を以下の順で**決定的に**導出する。同じ packet 名は常に同じスラッグになる。この規則は `packet-format.md` および export 系（`map-cc-sdd` / `map-openspec`）のスラッグ規則と**同一**である（出力先導出を揃える）。本節は `packet-format.md` の同名節の**同文の複製**であり、変更する場合は両方を同時に改訂する。

1. NFC 正規化する。
2. 前後の空白を trim する。
3. ASCII 大文字を小文字にする。
4. 空白とパスに危険な文字（`/ \ : * ? " < > |`）を `-` に置換する。
5. 連続する `-` を1つに圧縮する。
6. 先頭・末尾の `-` を除去する。

- 非 ASCII 文字（日本語等）はそのまま保持する。
- 結果が空文字列になる場合はスラッグを `unnamed-packet` とし、その旨を利用者に告知する。

### 衝突規則

- スラッグが既存ディレクトリと一致し、かつそのディレクトリの `db-design.md`（frontmatter の `source_packet`）が指す packet が**異なる** packet 名を指す場合のみ衝突とする。`-2` から始まる連番を付与して別名を割り当て、packet 名 → ディレクトリ名の対応を利用者に告知する。**黙って上書きしない**。
- **同一** packet 名を指す場合は衝突ではなく再生成であり、同じディレクトリの叩き台をその場で更新する（同ディレクトリ更新）。

## 不変条件（read-only・出力境界）

- 出力は `.intent/db-design/<packetスラッグ>/` 配下の派生物に閉じる。canonical（intent-tree / compass / packets）・既存スキーマ・export 下書きを一切変更しない。
- 出力を `.intent/cc-sdd/` および `.intent/openspec/`（export 物）に書き込まない（叩き台は要件ではない・export に混ぜない）。
- 他 packet のディレクトリへは書き込まない（書き込み先は対象 packet のスラッグ配下のみ）。
- 本形式は暫定契約であり凍結しない。後続の落差検出が突合できないと判明したときは本形式を改訂する（一方的に確定しない）。
