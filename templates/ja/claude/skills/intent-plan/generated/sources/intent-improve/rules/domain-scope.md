# Domain Scope（領域スコープ実行）

`intent-improve` が compass の Invariant / Decision Rule / Anti-direction を読むとき、案件の領域に合うものと横断（always）だけを部分ロードする規則。全記号の全量走査が実行コストを記号総数に比例させる問題（規模とともに実行頻度が現実に負ける）を、読む側の JIT（INV47・compass-category-tag-grep-filter）を維持ループへ広げて解く。DB・embedding を入れず grep + インラインタグだけで実効化する（DR71・INV2）。federated-governance / INV101 / C-fed2。

これは既存の3軸評価（completeness / correctness / coherence）と5分類のロジックを**一切変えない**。変えるのは「compass のどの記号を読み込んで照合するか」の範囲だけである。

## 発火条件（opt-in・無指定なら従来動作）

領域スコープは次のいずれかで対象領域が定まるときだけ発火する。定まらないときは従来どおり全量を読む（後方互換・O3・INV101 恒久フォールバック）。

- **利用者の明示指定**: 引数で領域名が渡された（例: `並行` / `語彙` / `配布`）。
- **案件文脈からの推定**: 引き継がれた発行ディレクトリ、または対象範囲に含まれる packet の `parent_intents` / Safety が名指す記号から、案件の領域を1つ読み取れる。推定で領域を定めたときは**黙って確定せず、推定した領域を報告に明示する**（`status` / `first-packet` の既存の「推定＋確認」の型に揃える。推測は推測と分かる形で出す）。

領域が定まらない・タグの無い旧 scaffold（`.intent/compass/index.md` も `[領域: ...]` タグも無い）では、この規則を発火させず従来の全量読みにフォールバックする。

## 読み方（grep + インラインタグ・INV47 の pull 規律をそのまま使う）

1. 対象領域が定まったら、compass を全文ロードせず、**案件の領域タグと `always` タグを一緒に**引く:
   ```
   grep -nE '\[領域: (<案件の領域>|always)\]' .intent/intent-compass.md
   ```
   分割収納 `.intent/compass/`（INV80）があるときは、`index.md` から `[領域: <案件の領域>]` と `[領域: always]` の行を引き、当たった記号ファイルの `## Law` を読む（無ければ上記の旧本体 grep のまま＝DR133 恒久フォールバック）。
2. `always` を**必ず一緒に引く**。複数領域に効く横断 Invariant（INV2 / INV9 / A1 等）を領域フィルタで落とすと drift になる（Anti-direction 226）。grep パターン・タグ書式は変えない（INV47 の共有契約）。

## 全体走査が本質の照合は絞らない（最重要・検出力を落とさない）

**coherence 軸の safety net としての役割**——packet に紐づかない drift・死蔵規律・前提崩れを拾う照合——は、「動かないもの・差分に現れないもの」を見るため全体走査が本質である。領域スコープを一律適用してこれを絞ると検出力が落ちる。次は領域指定時も**全記号を対象にする**:

- Decision Rules の `Revisit when` 成立の検出（coherence 軸）。
- 陳腐化・死蔵規律の検出（`/intent-validate` の `compass-rule-decay` 相当を improve が拾う場合）。

領域スコープが絞ってよいのは、対象領域の実装・成果物と compass 記号の**個別の整合照合**（この領域の invariant に実装が違反していないか等）である。「この領域の照合は領域内で全量と同一に効き、全体横断の safety net は全記号を見る」の二層を保つ。

## タグ欠落記号を黙って読み飛ばさない

分割収納内・compass 本文内に領域タグ（`[領域: ...]` / frontmatter の `area:` 行）を欠いた記号があるとき、**黙って読み飛ばさない**。安全側で always 扱いに含めて読むか、欠落として件数を報告するかは実装裁量だが、いずれでも欠落記号が報告に現れる（領域スコープを口実に検出漏れが静かに起きない）ことを保つ。

## 報告に読み込み対象数を明示する

領域スコープで実行したときは、報告の冒頭に**読み込んだ記号数と全記号数**を明示する（例:「並行 + always = 41 記号 / 全 829 を対象」）。全体走査に留めた軸があれば、その旨も明示する（例:「coherence の safety net は全記号を走査」）。無指定で全量読みしたときは従来どおり（対象数の明示は不要）。

## 温度

領域スコープは opt-in の実行コスト削減であり、gate ではない（INV101・止めない・奪わない）。領域が定まらなければ黙って全量へフォールバックし、利用者に領域指定を強制しない。補助スクリプトを足さず grep + インラインタグだけで実効化する（DR71）。
