# omission recap と昇格案内（読んだ範囲を要約し・人手コピーでの昇格を案内する）

`intent-from-code` skill が、抽出（extract-code-intent）を終えたあとに、**何を読み・何が抽出でき・何が沈黙し・何が読めなかったか**を要約して利用者に提示し、承認した候補を人手コピーで昇格するための正本。SKILL.md は手順と報告形式のみを持ち、「読んだ範囲をどう要約し、未承認候補をどう保持し、承認候補をどう昇格案内するか」は本ルールを参照する。本ルールは新たな抽出・カテゴリを定義せず、extract-code-intent が起こした候補と read-scope が確定した範囲を**そのまま要約**する。観測は Read / Glob / Grep に限り、対象コード・canonical な `.intent/*.md`・入力に使った既存文書を一切変更しない（書き込みは `.intent/code-ingest/` 配下のみ）。

## なぜ recap が要るか（「沈黙の沈黙」を避ける）

本ルールの目的を冒頭に明記する。これは隣の extract-code-intent が「各候補を起こす」のに対し、本ルールは「**読んだ範囲そのものと、埋まらなかった箇所を一覧として要約する**」役割を負う。

- 抽出できた候補を個別に並べるだけでは、AI が**どの範囲まで読み・どのカテゴリまで照合したのか**が利用者から見えない。読んだ範囲と検査したカテゴリの枠が不可視だと、利用者は「もうコードの意図は網羅された」と早合点して自分の探索を止めてしまう。これが「**沈黙の沈黙**」（AI が完全性を匂わせたことで人間が探索を止める不作為エラー・自動化バイアス）である。
- これを避けるため、recap は**読んだ範囲と照合したカテゴリの枠を明示的に提示する**。「どの範囲を読み、どのカテゴリ（目的・成果・能力・設計意図・不変条件・沈黙ギャップ）で照合し、そのうちどれが埋まり・どれが沈黙し・どこが読めなかったか」を一覧化し、抽出の輪郭そのものを利用者に見せる。
- 沈黙だけでなく**読めなかった箇所**（範囲外・既定除外・観測から意図を起こせなかった箇所）も明示する。推測で埋めず、不在を不在として残す。網羅を主張しないこと自体が recap の責務である。

## recap が要約する内容

extract-code-intent と read-scope の出力を受けて、次を観測可能な形で一覧化する。いずれも既に起こした候補・確定した範囲をそのまま引き、再抽出・再定義しない。

1. **読んだ範囲（何を読んだか）** — read-scope が確定した対象範囲（ディレクトリ・モジュール）と、既定除外（依存物・生成物）で読まなかった箇所を提示し、抽出の輪郭を見せる。
2. **抽出できたカテゴリ（コードから意図候補を起こせた項目）** — 候補を起こせたカテゴリを示し、recap が「沈黙だけを拾った」のではないことを観測可能にする。
3. **沈黙したカテゴリ（コードは示すが意図の記述が無く埋まらなかった箇所）** — extract-code-intent が「沈黙ギャップ」として起こした箇所と、そもそも候補が起きなかったカテゴリを一覧化する。意図的か偶発か仕分けできず Open Questions へ送った候補も、区別できる形で併記する。
4. **読めなかった箇所（不在）** — 範囲外・既定除外・観測から意図を起こせず判断を保留した箇所を、推測で埋めずに不在として明示する。

いずれの候補も全項目が inferred（推測 = Assumptions）であり、利用者の承認まで暫定である（INV65）。recap は再確認を促すための提示であって、確定ではない。

## 昇格は人手コピー（機械ハンドオフを持たない）

ここが本機能の境界の要であり、明示しなければ hidden shared ownership（所有権の重なり）を生む。

- code-ingest の出力は**派生（derived）・再生成可能・git 非追跡**であり、canonical の正本ではない。code-ingest は派生領域に「**転記しやすい形**」で承認候補を書き出すだけである。
- canonical への昇格は**利用者が承認した候補を手で discover / compass の対話に持ち込む人手コピー**で行う。intent-from-code が discover / compass を**呼び出す（自動起動する）ことはなく**、discover / compass が code-ingest の出力を**自動で読み込むこともない**。両者の間に自動連携（機械ハンドオフ）は存在しない。
- したがって recap は「承認した候補を、利用者自身が discover / compass の対話へ転記してください」という**案内**を、staging 出力の**末尾**に書く。recap 自身が昇格を実行することはない。転記先（intent-tree の Assumptions / compass の各ブロック）へ 1:1 で写せる見出し・粒度で書き出すことで、人手転記を容易にする（昇格の継ぎ目を人手に保つ）。
- この分離により、派生（code-ingest）と canonical（intent-tree / compass）の二重正本を作らず、所有権の重なりを避ける。承認は人の明示的な行為に限る（自動昇格を持たない・INV65）。

## staging 末尾に書く昇格案内の骨子

出力の末尾に、次を人が読んで実行できる手順として置く（機械ハンドオフを持たない前提を崩さない）。

- この出力は全項目が推測（Assumptions）であり、承認するまで canonical には載らないこと。
- 承認した候補を、見出しに添えた転記先（例: 目的候補 → intent-tree L0 Assumptions、不変条件候補 → compass Invariants）へ、利用者が **`/intent-discover` / `/intent-compass` の対話に手で持ち込んで**昇格すること。
- このスキルは discover / compass を自動で起動せず、discover / compass もこの出力を自動で読み込まないこと（昇格は人手コピーが唯一の経路）。
- 承認しなかった候補は破棄せず、次回の再確認のために staging（`.intent/code-ingest/`）に Assumptions のまま残ること（再実行時は全置換で作り直される）。

## 出力の扱い

- すべての出力は派生・再生成可能であり正本ではない。canonical な intent-tree / compass へは書き戻さない（昇格は承認後に人手コピー）。
- 本ルールは抽出の正本（extract-code-intent.md）や範囲統制の正本（read-scope.md）を一切変更せず、それらの出力を要約するだけである。抽出・範囲統制・機微情報の伏せ字を自前で実行しない（各正本に委ねる）。
