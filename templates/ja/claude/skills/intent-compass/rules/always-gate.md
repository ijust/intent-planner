# Always Gate（always 登録の一問確認）

compass の新記号（INV / DR / Anti）を `always`（全領域横断）で登録する前に、「本当に全領域に効くか・効かない領域はどこか」を一問だけ確認する関門。federated-governance / C-fed3 · INV101 / Anti-543。

`always` は領域スコープ実行（improve / validate）が**常に一緒に読む**「連邦の憲法」であり、薄いほど領域スコープの節約が効く。関門が無いと新記号が惰性で always に積まれ、節約効果が漸減する（中央の憲法は少数に保つのが連邦化の要）。

## 発火条件（always を選ぼうとするときだけ）

`rules/domain-write.md`（P-fed3・書き込み側の area 導出）で、新記号の `area` として **`always` を選ぼうとしたときだけ**この関門を発火する。領域タグ（`並行` / `語彙` 等の具体領域）を選ぶ起案では発火しない（area 確認の一問で閉じる）。

**二重質問にしない（domain-write と一体・最重要）**: area 導出の確認と always 関門を同じ起案で二度聞かない。domain-write の area 確認で `always` に倒れそうなときにこの関門の一問へ繋ぎ、具体領域なら domain-write の一問で閉じる。利用者から見て**一問**に畳む（area は何か → always なら「本当に全領域か」まで一続き）。

`.intent/domains/` が無い repo・領域タグの無い旧 scaffold では発火せず従来どおり（後方互換・恒久フォールバック・INV101/DR133 同型）。

## 一問の中身

`always` を選ぼうとしたとき、次を一問で確認する:

> この記号は本当に全領域に効きますか。効かない領域があれば挙げてください。

- 効かない領域が挙がったら、その領域を除いた具体的な領域タグ（1つ以上）を**提案**する（例:「では `並行` タグはいかがですか」）。採否は利用者が決める。
- 「全領域に効く」と確定したら `always` で登録する。**利用者の選択が最終**（下記の温度参照）。

## 温度（gate にしない・一問まで）

これは名前が「関門」でも、書き込みを**止める gate ではない**。利用者が `always` を選んだら従う（拒否しない・再質問を連射しない・INV101/INV91 同型）。関門が課すのは**一問の摩擦だけ**で、always 選択が2問以上の摩擦になったら違反。

- 既存の `always` 記号（現状 82 件）には触れない。振り直しは運用の中で漸進する（本関門は新規起案時のみ・利用者確定 2026-07-15）。
- always 件数の上限・閾値・機械判定を持たない（INV2）。「多すぎる」の判断は人に残す。
- 補助スクリプトを足さず、起案フローの一問だけで実効化する（DR71 同型）。
