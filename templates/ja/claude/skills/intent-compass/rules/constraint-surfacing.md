# Constraint Surfacing（制約叩き台の候補提示）

同梱のドメイン定石カタログ（`.intent/constraint-starters.md`）を read-only で照合し、Anti-direction / Invariants の**叩き台候補**を提示する手順。`/intent-compass` の Compass 構築フェーズで、Anti-direction / Invariant の**導出の前段**として使う。利用者が自分で育てた制約（`.intent/constraint-library.md`）があれば同様に候補へ加える。

これは C2（局所最適の抑止＝Anti-direction/Invariant を compass に明示する器）の導出（影響リスト→Invariant 化・プレモータム→Anti-direction）を**置き換えない補助**である。既存の導出はそのまま行い、その前段に「文脈に合う定石の叩き台候補」を差し込むだけ。

## 規律（守ること）

- **提示は read-only の叩き台に留める。** 合致した定石を Anti-direction / Invariants へ**自動で書き込まない**。compass への転記は、利用者が採否を選んでから人の手で行う（利用者への確認は、このスキルが持つ確認手段で行う）。
- **押し付けない。** 文脈に合わない定石を提示しない。候補は絞り、停止や強制を伴わない。当てはまりが弱ければ黙る（誤検知より黙る側に倒す＝叩き台機能の信頼を保つ）。
- **照合は意味的に行う。** 定石の「適合する状況」を案件文脈に照らす読解で判断する。機械的な文字列スコアリング・正規表現一致に寄せない。
- **既存導出を置き換えない・二重化しない。** 影響リスト→Invariant 化・プレモータム→Anti-direction の手順はそのまま。本手順は候補の供給を前段に足すだけ。
- **カタログ不在なら沈黙する。** `.intent/constraint-starters.md` が不在のときは照合をスキップしてその旨を告げる（停止しない）。`.intent/constraint-library.md` も同様（不在ならスキップ）。
- **どのログにも記録しない。** 提示は read-only の助言に留め、記録を持たない。

## 手順

1. **カタログを読む（領域インデックスから関係領域だけ pull する）**
   - まず `.intent/constraint-starters.md`（親カタログ）の**領域インデックス**を read-only で読む。本ファイルは分割されており、定石の本体は `.intent/constraint-starters/<領域>.md` にある。
   - 領域インデックスの各行（領域・ファイル・含む定石）を、いま書こうとしている compass の案件（題材・ドメイン・触る境界）と照らし、**関係しそうな領域ファイルだけ**を read-only で読む（全領域を常時ロードしない＝最小コストの pull 規律）。関係が判然としないときは候補となる領域を広めに読んでよいが、無関係領域は読まない。
   - あれば `.intent/constraint-library.md`（利用者が育てた制約）も read-only で読む。各ファイルから定石（`## id:` 単位）を取得する。
   - 親カタログ・領域ファイル・台帳のいずれも不在ならスキップして告げる（停止しない）。**後方互換**: 領域インデックスが無い（旧 scaffold の単一ファイル）ときは、従来どおり `.intent/constraint-starters.md` 全体を読む。
   - **既存 compass を参照するときも領域タグで部分ロードする（INV47・DR71）**: 手順4の既存導出で「同じ Anti-direction / Invariant が既に compass にあるか」を確かめるために `.intent/intent-compass.md` を読むときは、全文をロードせず、案件の領域タグ + `[領域: always]`（横断規律）だけを `grep -nE '\[領域: (<案件の領域>|always)\]' .intent/intent-compass.md` で引いて当たった見出し・項目だけを読む（compass-category-tag-grep-filter）。`always` を必ず一緒に引くのは横断 Invariant を領域フィルタで落とさないため（落とすと drift＝Anti-direction 226）。タグ未付与の項目が残っていても従来どおり全文読みにフォールバックできる（後方互換）。DB・embedding を入れず grep + インラインタグだけで実効化する（補助スクリプトを足さない）。分割収納 `.intent/compass/`（INV80）に該当記号があれば `index.md` → 該当ファイルの `## Law` を先に読む（無ければ本項の旧本体 grep のまま・DR133）。

2. **各定石の「適合する状況」を案件文脈と照合する**
   - 各定石の `適合する状況` を、いま書こうとしている compass の案件（題材・ドメイン・触る境界）に照らす。`適合する状況` は強い判定条件ではなく手がかりであり、当てはまりが弱ければその定石は出さない。
   - 照合に使うのは案件の文脈のみ。コード差分・実行時メトリクスは読まない。
   - **意図向きか手段向きかを読み分ける（個人台帳 `.intent/constraint-library.md` 由来の制約のみ）**: 各制約の `適合する状況` を「方針決め向き（意図ベース＝機能の意図を決める局面で効く）」か「実装向き（手段ベース＝SKILL 編集・DB 設計など実装の手段の局面で効く）」かに読み分ける。意図向きは従来どおりこの compass フェーズで候補提示し、手段向きは compass では出さず実装フェーズの発火（手順6）へ送る。読み分けは `適合する状況` の文章から意味的に行い（「いつ・何のときに効くか」が書かれていれば手がかりにする）、新しい記入項目は増やさない（既存 `適合する状況` の書き方で吸収する＝最小性）。同梱カタログ（starters）由来の定石はこの読み分けの対象外で従来どおり扱う。

3. **合致する定石を叩き台候補として提示する（書き込まない）**
   - **採否記録の器を先に読み、採否済みは再提示しない（INV57・DR84）**: 候補提示の前に、引き継がれた発行ディレクトリの `constraint-ledger.md`（`.intent/discovery/<スラッグ>-<rand>/constraint-ledger.md`・無ければ沈黙）を read する。同一発行系列で既に採否（採用/否認/保留）が付いた定石は再提示しない（ただし案件の目的・文脈が否認時の文脈一行から変わったと意味照合で読めるときは、否認済みも候補へ戻してよい・機械条件を持たない・INV2）。
   - 合致した定石の `叩き台`（Anti-direction 候補・Invariant 候補）を、利用者に候補として提示する。例:「この案件は `<id>`（<name>）に当てはまるかもしれません — 叩き台として <Anti-direction 候補> / <Invariant 候補> はいかがですか（採否はお任せします）」。
   - **compass へ自動で書き込まない。** 採用するかは利用者が判断し、採用したものだけを人の手で Anti-direction / Invariants へ取り込む。
   - **採否を器へ記録する（INV57・DR84）**: 提示に利用者が採否を付けたら、器へ1行追記する（`| 定石id | 局面=compass | 採否 | 文脈一行 | 日付 |`・否認は文脈一行必須）。器・発行ディレクトリの不在時は記録をスキップする（停止しない）。記録の詳細規約は `.intent/discovery/README.md` の「定石の採否記録」を正とする。これは提示ロジックを変えず記録先を1つ足すだけ。
   - 非該当の定石は提示しない。候補が多すぎないよう絞る。

4. **既存の導出へ進む**
   - 候補提示の後、既存の Anti-direction / Invariant 導出（影響リスト→Invariant 化・プレモータム→Anti-direction）を通常どおり行う。提示した候補のうち利用者が採用したものは、その導出の中で人の手で取り込まれる。

5. **採用した制約を個人台帳へ蓄積するか確実に問う（叩き台採用の直後・read-only 提示・人が採否）**
   - **発火タイミングを明示する。** ③で提示した叩き台のうち利用者が「これは自分の定番だ」と**採用した**制約があれば、その**採用の直後に**（compass への取り込みを終えてから別件へ移る前に）「この制約を個人台帳 `.intent/constraint-library.md` へ残しますか？」を**必ず一度 read-only で問う**。採用したのに蓄積を問わず素通りしない（この問いの実効化が本手順の要——個人台帳は「読む経路はあるが貯まる経路が働いていない」状態に陥りやすい）。
   - **記入スキーマの下書きを見せる。** 問うときは、採用した制約を個人台帳の確定スキーマ（`## id:` / name / 領域 / 適合する状況 / 制約 / 由来）へ写した**下書き候補**を提示する。`由来` 欄には「どの案件・どの叩き台 id（同梱カタログ由来なら其の id）から採用したか」を下書きで埋める。利用者はこの下書きを見て採否・修正する。次の案件で同梱カタログと並んで候補に出て、repo 内で再利用される。
   - **既に台帳にある制約は再提示しない（重複排除）。** `.intent/constraint-library.md` に同じ `id`（または実質同じ制約）が既載なら、その制約の蓄積は問わない（毎回同じ制約を並べて煩わせない＝叩き台機能の信頼を保つ・誤検知より黙る側）。
   - **自動で台帳に追記しない。** 追記は利用者が手動で行うか、利用者の明示承認のもとでのみ行う（自動的な蓄積は行わない＝read-only の歯止め）。**採用しなければ何も問わず何も追記しない。** ③までの提示の挙動は変えない（本手順は採用後の蓄積の問いを確実にするだけ）。
   - **蓄積はこのプロジェクト内にのみ閉じる。** 追記先は当該プロジェクトの `.intent/` 配下の内側のみとし、プロジェクトをまたいで制約を共有・永続する仕組みは提供しない（横断蓄積は案内しない）。
   - 個人台帳 `.intent/constraint-library.md` が不在のときは、蓄積をスキップしてその旨を告げる（停止しない＝後方互換）。
   - **蓄積下書きの「適合する状況」に「いつ効くか」を書く（書く側と読む側で目印を揃える）**: 採用した制約が手段向き（実装フェーズで効く）なら、下書きの `適合する状況` に「いつ・何のときに効くか（例: SKILL 本文を編集する改修のとき）」を書き添える規約とする。これは手順2の意図 vs 手段の読み分けと手順6の実装フェーズ発火が機能するための目印で、貯める側（本手順5）と読む側（手順2・6）で目印の書き方を揃える（齟齬を作らない）。新しい記入項目は増やさず既存 `適合する状況` の文章で吸収する（最小性）。

6. **手段向きの制約は実装フェーズでも発火させる（個人台帳のみ・read-only 候補提示）**
   - 手順2で「実装向き（手段ベース）」と読み分けた個人台帳の制約は、compass フェーズでは死蔵しやすい（案件文脈が「機能の意図」で手段ベースの制約とマッチしないため）。これらは**実装フェーズ**（`/intent-packets` の packet 起草・decision slot を埋める局面）でも read・意味照合・候補提示する。
   - 宿主は `/intent-packets` の `rules/decision-probe.md`（決定地点で `.intent/` から証拠を pull する手順）に相乗させる（新しい発火点を別に作らず、既存の証拠 pull に個人台帳を証拠源の1つとして加える＝最小コスト）。詳細は `intent-packets/rules/decision-probe.md` を参照。
   - 発火点を実装フェーズへ広げても本手順の歯止めをすべて継承する: 候補提示まで（compass も packet も library も自動で書き込まない）・当てはまりが弱ければ黙る（無関係な制約を出さない・台帳全件を読まない）・意味照合（機械スコアリングしない）・repo 内の `.intent/constraint-library.md` のみ読む（外部証拠源を読まない）・どのログにも記録しない。
   - 個人台帳が不在のとき、または手段向きと読み分けた制約が無いときは、何も発火しない（沈黙＝後方互換）。

## discover との関係

- discover の逸脱しやすい場面の事前チェックレーン（`drift-terrain.md`）でも、`drift-watch: on` のとき同じカタログを薄く照合して早期に気づかせる。本手順（compass）が主接点であり、discover は補助。
## 問いの平易さ点検（出力直前・共通）

利用者へ問い・確認を出す直前に、次の3点を点検する（通らなければ平易に書き直してから出す。書き直しで問いの意味・選択肢は変えない）:

1. **単体で通じるか**: その問いだけを読んで初見の読み手に意味が通るか。直前に読んだ内部文書（compass・packet・rules 等）の語彙をそのまま転写していないか。
2. **詰め込みすぎていないか**: 未説明の専門用語が1つの問いに3つ以上並んでいたら詰め込みすぎの合図＝分割するか言い換える。
3. **識別子に言い換えを添えたか**: コマンド名・記号・packet 名などの識別子を出すなら、初出に一行の普通の言葉の言い換えを添える。

この点検は生成時の予防であり、事後検査（`/intent-validate` の造語検査）と対で働く（予防だけ・検査だけにしない）。
