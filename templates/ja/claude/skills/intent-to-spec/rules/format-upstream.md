# format 写像：上流向け（why 前面）

`intent-to-spec` skill が、source scope 解釈が読み取った三層の素材を、**上流向けの自然言語 Spec**（why が先に立つ RFP / 提案書ふうの体裁）へ組み立てるための写像規則。SKILL.md は手順と報告形式のみを持ち、「どの素材をどの順で・どの見出しに配置するか」は本ルールを参照する。

## 責務の境界（写像であって読み取り・照合ではない）

- 本ルールは **format 写像** である。素材の構成・順序・見出しへの割り当てだけを担う。
- 射影元（intent-tree / compass / packets / steering）を**読み直さない**。独自のパーサ・スキーマを持たず、source scope 解釈が受け渡した素材（どの層・どの見出し・どの packet 由来かを保ったもの）をそのまま並べ替える。
- 各記述を射影元へ辿れる形にすること、射影元に無い記述を inferred として標識すること、不変則を省略・改変せず保持することは**捏造抑制ルールの責務**であり、本ルールは行わない。本ルールは canonical 由来の素材と inferred 由来の素材（Assumptions / Open Questions）を**区別したまま**受け取り、その区別を崩さずに配置することだけを守る。
- 中間 format（上流と統合の中間）は、本ルールの「why 前面」の度合いを統合仕様書側へ寄せたものとして扱う。独立した別ルールを持たない。

## 基本姿勢（why を前面に置く）

上流向け Spec は、**何を作るか（what）より、なぜそれを作るか（why）と何を守るか（制約）を先に読ませる**体裁である。読み手は実装者ではなく、方向と妥当性を判断する立場（上流の意思決定者）を想定する。したがって：

- 目的・不変則・判断基準（why と制約）を文書の**前面**に置く。
- 個別の requirements・受入条件のような実装に近い詳細は、why と制約の**後**に、それらを支える材料として置く。
- 流暢さのために素材に無い背景・動機を足さない（補完が要る箇所は捏造抑制ルールが inferred として扱うのであり、本ルールが本文に溶かし込んではならない）。
- 読み手が非当事者（顧客・経営など repo を読まない意思決定者）と判別されるときは `rules/reader-vocabulary.md` に従い、記号（INV/DR）を台帳で言い換え・台帳に無い識別子（packet id・スキル名等）を本文から省く（当事者向けのときは従来どおり記号を辿るための参照として残す・開き方の本体は同ルールに一本化する）。

## 構成（上から下へ：why → 制約 → 判断基準 → 個別要求）

source scope 解釈が受け渡した三層の素材を、次の順で配置する。各セクションには、その素材がどの層・どの見出し由来かが後段の捏造抑制ルールで辿れるよう、由来を保ったまま渡す。

| 順 | セクション | 何を置くか | 主な素材の由来 |
|---|---|---|---|
| 0 | 意図の階層マップ図（冒頭・素材があるとき） | 案件の意図の階層（L0–L4）を Mermaid 図で示し、読み手に全体像を先に掴ませる | intent-tree の L0–L4 の見出し階層（後述「意図の階層マップ図」） |
| 1 | 目的（なぜ作るか） | この Spec が向かう方向と、それが解く課題 | Intent の North Star（compass）＋ intent-tree の L0–L1（why の上位） |
| 2 | 守るべき不変則・制約 | 逸脱してはならない不変則と、避ける方向 | compass の Invariants / Anti-direction（＋指定時のみ steering 制約） |
| 3 | 判断基準 | 設計・実装で迷ったときの判断のよりどころ | compass の Decision Rules |
| 3.5 | 主要な設計判断（判断素材があるときだけ） | なぜ今の形に決まったか（主要な問い→選択肢→採否と決め手→見直し条件）を体系立ててまとめる | 後述「主要な設計判断」の判断素材（packet の Decisions・compass DR の Annex・supersede 系譜・deltas 昇格） |
| 4 | 個別の要求 | 上記の why と制約のもとで満たすべき個別要求 | 横断 requirements（packets の個別要求・依存・証拠） |
| 5 | 前提・未確定（あれば） | 確定した上記と**混ぜずに**別枠で置く前提・未解決の問い | intent-tree の Assumptions / Open Questions（inferred 由来） |

- **セクション 0（意図の階層マップ図）は冒頭に置き、素材（intent-tree の L0–L4）が読み取れたときだけ出す**。全体像を先に掴ませる目的で目的（1）より前に置く。素材が無い／intent-tree 不在のときは図を省き理由を明示する（空図を出さない）。詳細は後述「意図の階層マップ図」。
- セクション 1–4 は canonical 由来の素材で構成する。順序を入れ替えない（why → 制約 → 判断基準 → 個別要求の順を上流向けの体裁の核とする）。
- **セクション 3.5（主要な設計判断）は判断素材があるときだけ立てる**。判断基準（3）の後・個別要求（4）の前に置く（「なぜこの形か」を要求の手前で読ませる）。素材が無ければ節を立てない（空見出しを作らない）。詳細は後述「主要な設計判断」。
- セクション 5 は inferred 由来（Assumptions / Open Questions）の素材を、canonical の本文へ溶かさず**末尾の別枠**に置く。素材が無ければセクションごと省く（空見出しを作らない）。
- 該当する素材が無い層はセクションを省く（無い情報を埋めない）。ただし目的（1）と不変則・制約（2）は上流向け体裁の要であり、素材が読み取れているなら必ず置く。

## 意図の階層マップ図（冒頭・全体像を先に掴ませる）

案件の意図の階層（intent-tree の L0–L4）を冒頭に **Mermaid 図** で同梱し、読み手が本文に入る前に全体像を掴めるようにする。理論的裏付け＝言葉に図を添えると言葉だけより理解が上がる（マルチメディア原則）・全体から詳細へ段階的に見せる（C4 モデルの発想）。

- **描画規約は overview の `rules/mermaid-tree.md` を参照する（DR187・複製しない）**: ノード ID の導出・ラベルの禁止文字（`(` `)` `[` `]` `"` `/` を入れない）・孤立ノードを作らない（L0 以外は親エッジを持たせる）・図の直後にテキスト階層を常に併記・intent-tree が空/不在なら図を省いて理由を明示、といった描画の規約は同ルールが単一正本である。本ルールはその規約に従って L0–L4 を図にすることだけを定め、規約本文を複製しない。
- **素材の射影に限る（INV100）**: 図のノードは intent-tree の見出しから射影し、素材に無いノードを作らない。図に描いた階層関係も素材（見出しの入れ子）へ辿れる形にする（捏造抑制ルールの照合対象）。
- **範囲に従う**: source scope が部分指定（サブツリー）のときは、その範囲の L0–L4 だけを図にする。
- **素材が無いときは図を出さない**: intent-tree が未記入/不在のときは冒頭図を省き、「意図の階層が未記入のため図を省略（`/intent-discover` で起票できる）」と理由を明示する（空図・推測ノードを出さない）。
- 図の描画は読み手の閲覧環境（Mermaid 対応の Markdown ビューア）に委ね、実行時に描画・変換ツールを呼ばない（INV2）。テキスト階層の併記により、図が描画されない環境でも通読が止まらない。

## 主要な設計判断（判断・議論の過程を体系立ててまとめる）

これまでに積み上がった判断・議論の過程（source-scope が判断素材として読み取ったもの）を、**全部列挙せず体系立ててまとめて**射影する節。「なぜ今の形に決まったか」を読み手が辿れるようにするのが目的で、内容の厚みは判断過程を素材から拾って出す（本文を圧縮しない・数値スコアを入れない）。

### 3層で段階的に開示する（索引 → 要約 → 詳細）

一般的な設計判断の提示は、実務で確立した3層構造に従う（大量の判断を全部見せない）:

1. **索引層**: 何を決めたかの1行リスト（主要な問いの見出しだけ）。まず全体像を掴ませる。
2. **要約層**: 各主要判断を1件ずつ「問い → 検討した選択肢 → 採否と決め手 →（重要なら）見直し条件」でまとめる。ADR の形（Context/Decision/Consequences/Revisit）に対応する。
3. **詳細層**: 代替案の詳しい比較・帰結・supersede の変遷は、詳細度が「詳細」のときだけ後段に開く。

- 詳細度に従う: 概要=索引層のみ／標準=索引＋要約層／詳細=＋詳細層（source-scope の詳細度の読み分けと対応する）。
- 主系列を1本に線形化する（案件の主要な問いの列）。supersede の変遷や横断する判断は補助として後段に置き、主系列に混ぜない。

### 何を「主要」として要約層に出すか（選定基準）

要約層に出すのは、packet の `## Decisions` のうち **Human-fixed / 前倒し固定した判断**（前倒し5基準＝不可逆・複数モジュール波及・受入オラクル・セキュリティ床・複数 packet 拘束のいずれかに該当）。Agent-discretion / 可逆・局所の判断は索引層の1行に留めるか省く。これは設計判断の要約で確立した「残す判断」の基準（変更コストが高い・波及が広い・リスクが大きい・後で再燃する）と同型であり、新しい選別基準を発明しない。

### まとめ方の境界（要約であって創作でない）

- 要点化は素材にある判断の **削除（冗長・無関係を落とす）と一般化（複数の判断を、それが束ねる実際の判断へ辿れる形で束ねる）** に限る。素材に無い判断・採否理由を**合成して作らない**（新しい判断の創作は捏造＝INV73/捏造抑制ルールの照合対象）。
- 各判断は素材（どの packet の Decisions・どの compass DR・どの delta）へ辿れる形にする（トレース保持は捏造抑制ルールの責務）。

## 詳細度（output depth）ごとの厚み

同じ上流向け体裁の中で、確定した詳細度（概要 / 標準 / 詳細）に応じて**各セクションの厚み**を変える。詳細度は体裁と直交する軸であり（DR111）、新しい format を増やすものではない——構成（why → 制約 → 判断基準 → 個別要求）は3段とも共通で、各セクションにどこまで素材を書き下すかだけが変わる。

- **概要**: 目的（1）・不変則（2）を中心に、方向と守るべき核だけを短く。個別要求（4）は主要な要求名の列挙に留める。
- **標準**: 概要に加え、判断基準（3）と個別要求（4）に「何をどう満たすか」の要点（受入の観点まで）を添える。
- **詳細**: 標準に加え、個別要求（4）を packet 本文由来の素材（Scope の範囲・守るべき制約・受入基準の観点）まで書き下し、各記述を射影元へ辿れる形にする（トレースの密度が上がる）。

厚くするのは**素材を書き下す密度**であって、素材に無い背景・詳細を足すことではない（INV73）。素材が読み取れていない層は、どの詳細度でも「未記入／未観測」と明示して省く。

**主要な設計判断（3.5）も詳細度に従う**: 概要=判断の1行索引のみ／標準=主要判断の要約（問い→採否→決め手）／詳細=＋代替案比較・帰結・supersede 変遷。判断素材が読み取れていない案件では節を立てない（素材の射影であって、判断の創作で厚く見せない）。

## 不変条件

- 射影元を読み直さない・変更しない（写像のみ。読み取りは source scope 解釈、書き込みは SKILL.md の派生 Write の責務）。
- why（目的）と制約（不変則・判断基準）を、個別要求より**前**に置く順序を崩さない。
- canonical 由来の素材と inferred 由来の素材を混在させない（区別を保ったまま配置し、標識そのものは捏造抑制ルールに委ねる）。
- 素材に無い背景・動機・詳細を、文章を整えるために足さない（詳細度を上げても同じ＝素材で深くする・INV73）。
- 主要な設計判断（3.5）は判断素材があるときだけ立て、判断基準（3）の後・個別要求（4）の前に置く（順序を崩さない）。要点化は素材にある判断の削除・一般化に限り、素材に無い判断・採否理由を合成して作らない（要約であって創作でない・INV73）。読みやすさの数値スコアは導入しない。
- 意図の階層マップ図（0）は素材（intent-tree の L0–L4）があるときだけ冒頭に置き、描画規約は overview の `mermaid-tree.md` を参照する（複製しない・DR187）。図のノードは素材から射影し、素材に無いノードを作らない（INV100）。素材が無ければ図を省き理由を明示する（空図を出さない）。図生成の共通枠組みを新設しない（各体裁への個別追記・Anti-537）。
