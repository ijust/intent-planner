# format 写像：定例報告（結論先行）

`intent-to-spec` skill が、source scope 解釈が読み取った三層の素材を、**定例報告**（週次・隔週などの節目で「進んだこと・決まったこと・判断待ち」を関係者へ伝える報告文書）へ組み立てるための写像規則。SKILL.md は手順と報告形式のみを持ち、「どの素材をどの順で・どの見出しに配置するか」は本ルールを参照する。

## 責務の境界（写像であって読み取り・照合ではない）

- 本ルールは **format 写像** である。素材の構成・順序・見出しへの割り当てだけを担う。
- 射影元（intent-tree / compass / packets / steering）を**読み直さない**。source scope 解釈が受け渡した素材（どの層・どの見出し・どの packet 由来かを保ったもの）をそのまま並べ替える。
- トレース付与・inferred 標識・不変則保持は捏造抑制ルールの責務であり、本ルールは行わない。canonical 由来と inferred 由来（Assumptions / Open Questions）は区別したまま配置する。

## 読み手と設計制約（結論先行・BLUF）

読み手はこの取り組みを追う関係者（チーム内外・意思決定者を含む）を想定する。定例報告は「今回の節目で何が動いたか」を素早く掴めることが第一なので、**結論先行（BLUF）** を構成の核とする（DR93 の `bluf-message`）。

- **各節の要点を冒頭に置く。** 「進んだこと・決まったこと・判断待ち」の3本柱それぞれで、結論を先に書き、詳細・経緯は後に置く。
- **1出力1目的（種別分離）。** 報告は状況の伝達に徹し、手順書・設計詳細を混ぜない（DR93 の `doc-type-separation`）。
- **予定表・ベロシティを載せない（INV62）。** 日付コミットの予定や速度指標は出さず、状態遷移と前向きの順序までに留める。
- **語彙は読み手に合わせる（A33）。** 開発者向けの報告なら内部識別子（DR/packet id 等）を残してよいが、初出では普通の言葉の言い換えを添える。非開発者が混じる報告では識別子を省くか言い換える。

## 構成（3本柱：進んだこと → 決まったこと → 判断待ち）

source scope 解釈が受け渡した素材を、次の3節で配置する。

| 順 | セクション | 何を置くか | 主な素材の由来 |
|---|---|---|---|
| 1 | 進んだこと | 前回以降に進んだ作業（state 遷移・done になった単位） | packets の state / updated_at（最近 done / verifying になったもの） |
| 2 | 決まったこと | 新たに確定した判断・決定 | 新規の Decision Rules（compass）／packet の確定した決定スロット |
| 3 | 判断待ち | いま人の判断を待っている問い | intent-tree / compass / packets の未回答 Open Questions ＋ 承認待ち delta |

- 各節とも要点（結論）を先頭に置く（背景から入らない）。
- 該当する素材が無い節は「該当なし」と明示して残す（節ごと消さない＝「今回は決定なし」も報告価値がある）。
- 判断待ち（3）の各項目には、どのファイルのどの問いかの出所を添える（読み手が原典へ辿れるように）。
- inferred 由来（Assumptions / Open Questions）を canonical の確定事項と混ぜない。

## 不変条件

- 射影元を読み直さない・変更しない（写像のみ）。
- 各節で結論を先頭に置く順序を崩さない（BLUF）。
- 予定表・ベロシティを載せない（INV62）。
- 素材が無い節は「該当なし」と明示し、推測で埋めない。
- canonical 由来と inferred 由来を混在させない（標識は捏造抑制ルールに委ねる）。
