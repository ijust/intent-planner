# Glossary（正規語彙の軽量台帳・β）

> このプロジェクトで合意された正規語彙（ubiquitous language）の母集合を1か所にまとめる軽量な canonical 台帳です。`/intent-validate` が「台帳に無い語＝造語の疑い」を read-only で名指しする際の母集合として読みます。**編集の主体は人です。人が1語ずつ内容を確かめて承認した登録・置換に限り、ツールによる代筆（承認済みの行の書き足し・承認済みの言い換えの適用）を認めます。承認なしの自動改変はしません。**

## この台帳の使い方

- **用途は「正規語＋表記ゆれ/同義語の集約」に限定します。** 用語を訳語へ一括置換するための辞書ではありません（読みやすさ・discoverability を損なう訳語置換はしません）。
- **これは網羅ではありません。** 正規語が増えたら行を追記して育てる前提のファイルです。新しい語を正式採用したら、その正規語と表記ゆれ・一行説明をここに登録してください。
- **一行説明は、初見の人に通じる普通の言葉で書きます。** 定義の中に、説明していない登録語や英語の専門用語を連ねないでください。迷ったら「何をするものか」を主語と述語で書きます（例:「〜を指摘する検査。ファイルは書き換えない」）。普通の言葉で説明できない語は、その語を登録すること自体を見直してください。
- **canonical（編集の主体は人）です。** 造語管理 skill はこの台帳を勝手に書き換えません。登録・言い換えの反映は、人が承認してからの別アクションです（人が承認した内容に限り、ツールが代わりに行を書き足すことはできます）。
- **登録・昇格は1語ずつです。** 語の登録と状態の昇格（暫定→承認済み）は、人がその語の一行説明を読んで「初見の人に通じるか」を確かめたうえでの1語ずつの個別承認だけで成立します。複数語をまとめた一括登録・確認を省いた流れ作業の承認・承認なしの状態の昇格は、登録として成立しません。
- **否認済みの語は削除せず残します。** 「この語は採用しない」と決めた語は、行を消すのではなく状態を「否認済み」にして残します。残しておくことで、同じ語が後からまた発明されたときに「一度否認された語の再発明」として気づけます。
- 「正規語」列に無い語が intent 成果物に現れたとき、検出はそれを「造語の疑い」として候補提示します。固有名詞・既存の英語用語・初出一行説明済みの正当な新語は除外されます。

## 記入スキーマ（最小3項目＋任意2項目）

先頭3項目（正規語・別表記・一行説明）が最小の記入単位です。後ろ2項目（状態・言い換え例）は任意で、3項目だけの行や旧来の3列の台帳もそのまま有効です（列の追加だけの拡張で、既存の行の書き換えや一括の移行は不要です）。

- **状態**は「この語がチームでどこまで合意されたか」を表す3値です — `承認済み`（チームで合意済みの共通語）／`暫定`（登録した人の暫定の語。チームの合意はまだ）／`否認済み`（採用しないと決めた語。行は削除せず残します）。**状態が書かれていない行（旧来の3列の行を含む）は「暫定」として読みます**（チーム合意済みとは扱いません。承認済みへの昇格は、人が1語ずつ確認して状態を書いたときだけ成立します）。
- **言い換え例**は、その語を知らない読み手に向けて開くときの言い方の例です（例: 「JIT 供給」→「必要なときに必要な分だけ渡す方式」）。質問や文書がその語を初見の読み手に向けるとき、そのまま使える形で書きます。

| 正規語 | 別表記・同義語 | 一行説明 | 状態 | 言い換え例 |
|---|---|---|---|---|
| ubiquitous language | 正規語彙, ユビキタス言語 | このプロジェクトで「正しい」と合意した用語の全体の集まり。使う言葉が人によってばらばらになると意図の擦り合わせが崩れるので、1か所にまとめておく。 |
| 正規語彙 | canonical vocabulary, ubiquitous language | この台帳に登録された、使ってよい正しい用語のこと。ここに無い語は「新しく作られた語では」と疑われる。 |
| coinage-suspect | 造語の疑い | 台帳に無い語が成果物に現れたとき、「新しく作られた語では」と指摘する検査の名前。指摘するだけで、ファイルは書き換えない。 |
| groundless-conclusion | 根拠なき結論の疑い | 「なぜそう決めたか」の理由（前提・制約・引き換えにしたもの）が辿れない結論を指摘する検査の名前。理由が残っていない結論は後から見直せなくなるため。ファイルは書き換えない。 |
| unverified-hypothesis | 未検証の仮説の疑い | 証拠の裏付けが無いまま「こうだ」と確定されている仮説を指摘する検査の名前。groundless-conclusion が「理由の欠け」を見るのに対し、こちらは「証拠の欠け」を見る（役割が別）。ファイルは書き換えない。 |
| dangling-reference | dangling 参照の疑い, 宙吊り参照 | 文書の中の番号付きの参照（`INV 3` や `DR 5` のような指し先）が、指す先の項目の削除や移動で行き先を失っていないかを指摘する検査の名前。coinage-suspect など他の検査とは見るものが別で、こちらは「参照の行き先が実在するか」だけを見る。ファイルは書き換えない。 |
| glossary | 語彙台帳, 用語集 | 正しい用語とその別表記・一行説明をまとめた、この一覧表のこと。 |
| drift | 逸脱 | 作業が進むうちに、当初の意図から少しずつ離れていってしまうこと。起きやすい場面は drift-patterns という一覧にまとめてある。 |
| packet | パケット | 意図を実装へ渡すときの、ひとまとまりの作業単位。どこから来たかの記録を各ファイルの先頭（frontmatter）に残す。 |
| compass | intent-compass | 判断に迷ったとき立ち返る基準集。「守る約束（Invariants）」と「決め方の記録（Decision Rules）」をまとめたファイル。 |
| intent tree | intent-tree, 意図ツリー | 「何のために・何を・どう」という意図を、目的から作業単位の候補まで階層で書いた正本のファイル。 |
| 工程計画 | work plan | packet を「どのまとまりで・どの順に作るか」という人の計画意図を、`.intent/packets/plan.md` の任意の節に書いたもの。グループの見出しは自分の言葉で自由に書け（Phase・step 等・入れ子可）、並び順が優先順位を表す。ツールはこれを読んで推奨着手順を案内するだけで、順序を強制したり自動で割り当てたりはしない。**ツール自身の工程フェーズ（discover→compass→… という進め方の段）や、packet の `state`（draft/ready/… という進行段階）や、milestones（過去に起きた節目の記録）とは別物**（それらは既存の別の仕組み）。 |
