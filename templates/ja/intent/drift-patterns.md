# Drift Patterns（逸脱の型カタログ）

> `/intent-discover` が地形診断の根拠として読みます（`drift-watch: on` のとき）。利用者が自分の現場で踏んだ逸脱を型として足して育てるファイルです。

## このカタログの読み方

- **これは網羅ではありません。** ここに載っている型は出発点（seed 1件 + 汎用2件）にすぎません。あなたの現場で実際に踏んだ逸脱を型として追記し、育てていく前提のファイルです。
- 各型は集計キー（`id`）で `drift-log.md` と突き合わせます。型を増やすほど、着手前に「踏みやすい地形」を名指しできる範囲が広がります。
- 同梱の汎用2件（`premature-abstraction` / `layer-leak`）は、症状の異なる**弱い手がかり**として置いています。1個の強い代表例（`microservice-over-split`）に思考が固着（fixation）しないよう、あえて毛色を散らしてあります。
- 型に「該当した」ことは逸脱の確定ではありません。地形診断は誤検知前提です。該当したら**先に anti-direction / invariant を書いておく**ことで、外れきる前に手を打ちます。

## 型の書き方

新しい型は次のスキーマで追記してください。`id` は kebab-case の集計キーとして一意にします（`drift-log.md` の `pattern` がこの `id` を参照します）。

```markdown
## id: <kebab-case 集計キー>
- name: <短い名前>
- symptom: <題材・進み方の手がかり。discover が題材と照合する弱い手がかり>
- 先に書かせるもの:
  - Anti-direction: <該当型が疑われたとき compass 化前に書かせる anti-direction 候補>
  - Invariant: <同 invariant 候補。題材依存なら discover が文脈から具体化する>
```

- `symptom` は「これに当てはまったら必ず逸脱」という強い判定条件ではなく、照合の手がかりとして書きます（弱い手がかりにすることで fixation を避けます）。
- 「先に書かせるもの」は、その型が疑われたときに compass 化より前に Intent Tree の Open Questions / anti-direction 候補へ書かせる文面です。題材に依存する部分は discover が文脈から具体化します。

---

## id: microservice-over-split

- name: マイクロサービス過剰分割
- symptom: 単一の意図に対し「分割して疎結合に」という美徳へ過剰適応し、サービス・モジュール・パッケージを必要以上に切り出していく。境界をまたぐ呼び出しや調整コストが増え、当初の意図より複雑・疎結合過剰になっていく進み方。
- 先に書かせるもの:
  - Anti-direction: 「疎結合・分割」を目的化しない。サービス／モジュールを増やす変更は、その分割が解く具体的な問題を1つ名指しできるときだけ行う。
  - Invariant: 当初の意図が要求しない境界（サービス・プロセス・パッケージの分割）を増やさない。分割は意図の境界（L3）に一致するときのみ許す。

## id: premature-abstraction

- name: 早すぎる抽象化
- symptom: まだ用例が1〜2件しかない段階で共通化・汎用基底・設定駆動の枠組みを先に作り始める。「将来こう使うかも」を根拠に、現に存在しない要求のためのレイヤーや拡張点を導入していく進み方。
- 先に書かせるもの:
  - Anti-direction: 現に存在しない将来要求のための抽象化・汎用化を先回りで作らない（rule of three を待つ）。
  - Invariant: 抽象（共通基底・汎用枠組み・設定駆動）の導入は、実在する用例が3件以上あるか、意図が明示的に汎用性を要求するときに限る。

## id: layer-leak

- name: レイヤー越境
- symptom: 本来下層に閉じるべき関心事（ドメインロジック・永続化・外部 I/O）が上層（UI・ハンドラ・表示層）へ滲み出す、あるいはその逆向きに依存が逆流する。各変更は局所的に楽でも、層の責務境界が少しずつ崩れていく進み方。
- 先に書かせるもの:
  - Anti-direction: 目先の都合でドメインロジックや I/O を表示層・ハンドラへ寄せない。層をまたぐ近道をしない。
  - Invariant: 各層の責務境界（依存の向き）を保つ。上層は下層に依存してよいが、下層は上層に依存しない。

## id: coinage-proliferation

- name: 造語が増えやすい地形
- symptom: AI が正規語彙（ubiquitous language）に既にある語を再利用せず、台帳に無い新造の語を次々と充てていく。同じ概念に複数の言い回しが並び、語彙が分裂して意図の擦り合わせ（製品の核）が崩れていく進み方。
- 先に書かせるもの:
  - Anti-direction: 正規語が既にある概念へ新しい語を作らない。やむを得ず新語を導入するときは初出に一行の説明を添える。
  - Invariant: 用いる語は glossary の正規語彙へ辿れるか、初出で一行説明されている。台帳に無い語を説明なしで増やさない。

## id: scope-creep

- name: スコープが拡大しやすい地形
- symptom: export 済みの作業単位（packet）の宣言スコープを超える実装指示が後から来る（例: フロント専用 packet なのにバックエンド・認可・トランザクション境界の実装を足すよう求める）。各追加は局所的に自然でも、新領域でこそ必要になる packet 固有の不変則（認可・データ整合性・トランザクション境界・冪等性）が抜け落ちたまま積み上がる進み方。
- 先に書かせるもの:
  - Anti-direction: export 済み packet の `## Scope` を超える指示を cc-sdd でそのまま押し切らない。新領域は別 packet として intent に戻す。
  - Invariant: 実装は対象 packet の宣言スコープ（`## Scope` / `## Non-scope`）内に留まる。新領域に及ぶなら、その領域の packet 固有不変則（認可・整合性・トランザクション境界・冪等性）を先に立てる。
