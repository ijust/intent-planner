# Constraint Starters（制約の叩き台カタログ）

> `/intent-compass`（主接点）・`/intent-discover`（逸脱しやすい場面の事前チェックレーン）が、Anti-direction / Invariants を書くときの**叩き台候補**として読みます。同梱のドメイン定石を文脈と照合して「このケースはこれでしょ」と候補提示するための母集合です。**提示は read-only の叩き台であり、compass への転記は人が採否を選んでから手動で行います**（AI が自動転記しません）。利用者が自分のよく使う制約は別ファイル `constraint-library.md` で育てます。

## このカタログの読み方

- **これは出発点であって完成形ではありません。** 領域カバレッジを広げて網羅性を高めていきますが、各定石は「必ずこれ」という強制ではなく、文脈に合う**候補**として並べるだけで、採否は利用者が判断します。網羅性を上げても read-only の叩き台・誤検知前提・人手転記という性質は変わりません。
- **このカタログは領域別ファイルに分割されています。** 本ファイル（親カタログ）はスキーマ・読み方・検証規律と**領域インデックス**（下記）を持ち、定石の本体は `constraint-starters/<領域>.md` にあります。`/intent-compass`・`/intent-discover` は、当該案件に関係する領域だけを read-only で pull します（全領域を常時ロードしません＝最小コストの pull 規律）。
- 各定石は `id`（kebab-case）で識別します。`領域` でコード／非コードを区別し、`適合する状況` で「どんな案件のとき効くか」を、`叩き台` で Anti-direction 候補・Invariant 候補を持ちます。**デザイン（UI/UX）はコード・非コードの両方に跨る横断軸**であり、独立した領域値ではなく `領域` は `code | non-code` のいずれかに割り当て、`適合する状況` に「フロントエンドの UI 実装」等の文脈を書いて吸収します。
- **これは補助であって置き換えではありません。** compass の Anti-direction/Invariant 導出（影響リスト→Invariant 化・プレモータム→Anti-direction）はそのままで、その前段に叩き台候補を差し込むだけです。
- **これは静的な文書です。** 利用のたびに外部サービス・オンライン照会・データベースへ問い合わせることはありません。「最新の定石を取得する」目的の外部呼び出しもしません。同梱の静的内容と、後続スライスでの文脈補完に留めます。
- **これは既存の逸脱カタログ（drift-patterns）・気づきカタログ（context-cost-cues）とは別カタログです。** 溜める対象が違う（再利用したい制約定石）ため別ファイルにしており、逸脱ログ（drift-log）の記録・集計には一切触れません。

## 出自・正確性の検証規律（同梱定石の品質）

- **各定石は出典（`出典` フィールド）を必ず持ちます。** 根拠（一次情報の URL 等）と取得日を記し、出自不明の定石を同梱しません。誤った・陳腐化した定石を「このケースはこれでしょ」と押し付けないための歯止めです。
- **正確性レビュー**: 定石を追加・更新するときは、出典が実在し、現在も妥当であることを確認します（推測や捏造の出典を付けません）。
- **更新方針**: 出典が陳腐化した（推奨が変わった・撤回された等）定石は、出典を更新するか、定石自体を見直します。
- **ライセンス**: 出典から引用する場合は、引用元のライセンス・利用条件を尊重します。

## 定石の書き方

新しい定石は次のスキーマで追記してください。`id` は kebab-case として一意にします。`出典` は必須です（出典なしの定石を加えないでください）。

```markdown
## id: <kebab-case 識別キー>
- name: <短い名前>
- 領域: <code | non-code>
- 適合する状況: <どんな案件・題材のときこの定石が効くか。compass/discover が文脈と照合する手がかり>
- 叩き台:
  - Anti-direction: <その状況で書かせたい anti-direction 候補>
  - Invariant: <同 invariant 候補。題材依存なら compass が文脈から具体化する>
- 出典: <根拠（一次情報の URL 等）・取得日。必須>
```

- `適合する状況` は「これに当てはまったら必ずこの定石」という強い判定条件ではなく、照合の手がかりとして書きます（誤検知前提・候補を絞り押し付けません）。
- `叩き台` は compass の Anti-direction/Invariants へ**そのまま入れる確定値ではなく**、人が採否を選ぶ候補です。

## 領域インデックス

定石の本体は領域別ファイルにあります。照合するときは、案件の文脈に合う領域だけを読んでください（全領域を常時ロードしない）。

| 領域 | ファイル | 含む定石（適合する案件） |
|------|----------|--------------------------|
| code / セキュリティ | `constraint-starters/code-security.md` | SQLi・XSS・CSRF・認可（最小権限）・シークレット管理（ユーザー入力・認証・機密の扱いが絡む案件） |
| code / 設計・信頼性 | `constraint-starters/code-reliability.md` | 冪等性・リトライ安全・入力バリデーション・N+1 クエリ回避（書き込み・外部入力・データアクセスが絡む案件） |
| code / フロントエンド・デザイン横断 | `constraint-starters/code-frontend.md` | アクセシビリティ（WCAG）・フォーム UX（UI コンポーネント・画面を実装する案件。デザイン横断軸） |
| code / VCS（バージョン管理）衛生 | `constraint-starters/code-vcs-hygiene.md` | 誤コミットした秘密の履歴除去+ローテーション・コミットメッセージの機密・機密コミットの入口防止（.gitignore/コミット前検知）・公開化前の履歴監査（Git を使う案件全般・公開リポジトリ化する案件。`secrets-no-hardcode` の直書き予防とは射程を分ける Git 操作・履歴経路） |
| non-code / 文書 | `constraint-starters/non-code-document.md` | 発表資料の構成・要件記述の明確化（RFC2119）・ドキュメント種別の分離（資料・仕様文書・README を作る案件） |
| non-code / 伝達 | `constraint-starters/non-code-communication.md` | 結論先行（BLUF）・リリースノート/変更履歴（メール・告知・リリースノートなど読み手を動かす案件） |

> **デザイン（UI/UX）の定石は独立した領域ではなく、code または non-code の領域ファイルに横断的に現れます**（`適合する状況` でフロントエンド/UI 文脈に当てます）。領域とファイルは案件の必要に応じて今後も増やせます。
