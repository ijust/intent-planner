# Question Packs（案件種別の質問パック）

> `/intent-discover` の詰めの局面が、案件種別に合った**詰めの問いの候補**として読みます。**提示は候補まで**で、どのパックを使うかは利用者が選び、採用しても問いへの回答を強制しません（保留・「後で」を毎回選べます）。パックの内容が Anti-direction / Invariants / 台帳へ自動で書き込まれることはありません。**このカタログが無い環境では何も起きません**（従来どおりの詰めに戻ります）。

## このカタログの読み方

- **これは出発点であって完成形ではありません。** 同梱の4種別は seed です。あなたの現場でよく扱う案件種別のパックを追記して育てられます（種別が増えて重くなったら、領域別ファイルへの分割を検討してください）。
- **これは「制約」ではなく「問い」のカタログです。** 同じ静的カタログ＋文脈照合の型でも、constraint-starters（制約の定石＝Anti-direction / Invariant の叩き台）とは溜める対象が違うため、別ファイルにしています。
- **各パックは `id` で識別し、`適合する状況` で「どんな案件のとき効くか」を持ちます。** 照合は弱い手がかりで行い、当てはまりが弱ければ何も提示されません（誤検知より黙る側）。
- **問いは「推論+確認」の既定を置き換えません。** パックの問いを順に全部聞くチェックリストにはしません。効きそうな問いを少数選んで出し、各問に「なぜ聞くか」を添え、毎回「後で／不明／該当なし」を選べます。
- **これは静的な文書です。** 利用のたびに外部サービスへ問い合わせることはありません。

## 出典・由来の規律（ハイブリッド）

- **一次情報のある問い**（構成セオリー・研究に由来するもの）は `出典` に根拠（URL・書名等）と取得日を書きます。捏造した出典を付けません。
- **経験則の問い**は `由来` に「何に基づくか」を正直に書きます（出典の体裁を偽装しません）。
- 追記するときもこの使い分けに従ってください。

## パックの書き方

```markdown
## id: <kebab-case 識別キー>
- name: <短い名前>
- 適合する状況: <どんな案件・題材のときこのパックが効くか（照合の手がかり）>
- 問い:
  - <問い> — なぜ聞くか: <一行>
- 出典/由来: <一次情報なら出典＋取得日／経験則なら由来>
```

## id: proposal-planning

- name: 企画書・提案書
- 適合する状況: 企画書・提案書・稟議など「誰かに承認・賛同してもらう文書」を作る案件。作りたいものはあるが、誰の何の問題かが言葉になっていないとき。
- 問い:
  - 誰の・どんな状況の問題を解くのか — なぜ聞くか: 対象が曖昧な企画は承認者が価値を判定できず、後工程すべてが揺れるため。
  - うまくいったと何で分かるか（観測できる形で） — なぜ聞くか: 成功の判定手段がない企画は、実行後に評価も撤退判断もできないため。
  - 承認者が最初に知りたい結論は何か — なぜ聞くか: 結論を末尾に置いた企画書は忙しい承認者に読まれないため（結論先行）。
  - この企画を「やらない」場合に何が起きるか — なぜ聞くか: 不作為のコストが言えない企画は優先順位の競争に負けるため。
  - 採らなかった代替案はあるか・なぜ採らないか — なぜ聞くか: 代替案の検討跡が無いと「他は考えたのか」の一問で差し戻されるため。
- 出典/由来: 結論先行は Barbara Minto "The Pyramid Principle"（結論先行・根拠で支える構成原則）に基づく。他は本ツールの詰めの型（目的・成功・想定ユーザーの追認、代替案の検討跡）を非コード案件の言葉に言い換えた経験則。

## id: research-summary

- name: 調査まとめ・リサーチノート
- 適合する状況: 調べたことを1本のまとめに落とす案件。集めた情報は多いが、何を結論として言えるかが定まっていないとき。
- 問い:
  - この調査が答えるべき問いは何か（1文で） — なぜ聞くか: 問いが定まらないまとめは情報の羅列になり、読み手が結論を受け取れないため。
  - 何が分かれば調査を「終わり」にできるか — なぜ聞くか: 終了条件がない調査は際限なく広がるため。
  - 情報の確からしさをどう区別するか（一次情報・伝聞・自分の推測） — なぜ聞くか: 出所の混在したまとめは、後から「どこまで信じてよいか」を検証できないため。
  - 結論に反する情報はあったか — なぜ聞くか: 都合の良い情報だけを集めたまとめは、反例1つで崩れるため（確証バイアスの緩和）。
- 出典/由来: 経験則（本ツールの canonical / inferred 分離と consider-the-opposite の詰め観点を、調査文書の言葉に言い換えたもの）。

## id: article-outline

- name: 記事・発信文書の構成
- 適合する状況: 記事・ブログ・登壇資料の原稿など「読み手に届ける文章」の構成を練る案件。書きたいことはあるが、読み手に何が残るかが曖昧なとき。
- 問い:
  - 読者は読み終えたあと、何ができる・何が変わるのか — なぜ聞くか: 読後の変化が言えない記事は「良い話だった」で終わり、行動につながらないため。
  - この記事の読者は誰で、何をすでに知っているか — なぜ聞くか: 前提知識の見積もりを誤ると、易しすぎて退屈か難しすぎて脱落のどちらかになるため。
  - 読者の目的は「学ぶ・作業する・調べる・理解する」のどれか — なぜ聞くか: 目的の違う内容を1本に混ぜると、どの読者にも届かない文書になるため。
  - 最初の3行で読者に何を約束するか — なぜ聞くか: 冒頭で価値が伝わらない文章は、本文がどれだけ良くても読まれないため。
- 出典/由来: 読者の目的4分類は Diátaxis（tutorials / how-to guides / reference / explanation・https://diataxis.fr/・取得 2026-06-26〔constraint-starters の doc-type-separation と同一出典〕）に基づく。他は経験則（読み手起点の構成の型）。

## id: event-planning

- name: イベント・ワークショップ計画
- 適合する状況: 勉強会・ワークショップ・説明会など「当日がある催し」を計画する案件。やることリストはあるが、成立条件が曖昧なとき。
- 問い:
  - 参加者は帰るとき、何を持ち帰っていれば成功か — なぜ聞くか: 主催側の進行ではなく参加者の持ち帰りで設計しないと、滞りなく進んだのに何も残らない会になるため。
  - 当日の「失敗」とは何が起きることか（プレモーテムとして挙げる） — なぜ聞くか: プレモーテムは、思いつきの列挙より失敗要因の検出力が高いため。
  - 最少で何人・何が揃えば成立するか — なぜ聞くか: 成立ラインが無いと、集まりが悪いときに実施・延期の判断ができないため。
  - 当日変更できないもの（会場・機材・締切）はどれか — なぜ聞くか: 不可逆な要素から先に固定しないと、直前の変更が全体を壊すため。
- 出典/由来: プレモーテムは Gary Klein "Performing a Project Premortem"（Harvard Business Review, 2007）に基づく。他は経験則（不可逆の前倒し固定・成立条件の明示）。
