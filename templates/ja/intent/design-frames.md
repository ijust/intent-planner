# デザインフレームの定石

案件に合う既存のデザインフレームを比較するための共有カタログです。各骨格の空欄は検討材料であり、調査や確認をしていない内容を事実として扱わないでください。

## id: persona

- 一般名: ペルソナ
- 適合する状況: 調査で得た行動、目的、困りごとを、設計判断に使える代表的な利用者像としてチームで共有したい場合。
- 下書きの骨格: 次のMarkdown見出しを使う。
  - `### 対象と根拠`
  - `### 行動と利用状況`
  - `### 目的と困りごと`
  - `### 確認が必要な推測`
- 誤用注意: 想像だけで属性を作ったり、利用者全体を一人の人物像で代表させたりしない。調査から分かったことと確認が必要な推測を分ける。
- 出典: Nielsen Norman Group, “Personas: Study Guide”（https://www.nngroup.com/articles/persona/）

## id: jobs-to-be-done

- 一般名: Jobs to be Done（利用者が達成したいこと）
- 適合する状況: 利用者が特定の状況で何を前進させたいのかを捉え、既存機能や属性ではなく達成したいことを軸に選択肢を検討したい場合。
- 下書きの骨格: 次のMarkdown見出しを使う。
  - `### 利用者の状況`
  - `### 達成したいこと`
  - `### 期待する結果`
  - `### 根拠と確認が必要な推測`
- 誤用注意: 解決策や機能名を達成したいこととして書かない。利用者の状況と期待する前進を、調査根拠とともに記述する。
- 出典: Christensen Institute, “Jobs to Be Done Theory”（https://www.christenseninstitute.org/theory/jobs-to-be-done/）

## id: customer-journey-map

- 一般名: カスタマージャーニーマップ
- 適合する状況: 一つの目的に向かう体験を時間の流れに沿って捉え、複数の接点で起きる行動、考え、感情、課題を共有したい場合。
- 下書きの骨格: 次のMarkdown見出しを使う。
  - `### 利用者と目的`
  - `### 段階と接点`
  - `### 行動・考え・感情`
  - `### 課題と根拠`
  - `### 確認が必要な推測`
- 誤用注意: 理想的な成功経路だけを描かない。実際の調査に基づき、行き詰まり、離脱、接点間の不整合も含める。
- 出典: Nielsen Norman Group, “Journey Mapping 101”（https://www.nngroup.com/articles/journey-mapping-101/）

## id: service-blueprint

- 一般名: サービスブループリント
- 適合する状況: 利用者から見える接点と、それを支える担当者の活動、内部処理、支援資源の関係を整理したい場合。
- 下書きの骨格: 次のMarkdown見出しを使う。
  - `### 利用者の行動`
  - `### 利用者から見える接点`
  - `### 利用者から見えない活動`
  - `### 支援する処理と資源`
  - `### 受け渡しと失敗点`
  - `### 根拠と確認が必要な推測`
- 誤用注意: 表側の流れだけで終わらせない。接点を支える裏側の活動、受け渡し、待ち時間、失敗点を対応づける。
- 出典: Nielsen Norman Group, “Service Blueprints: Definition”（https://www.nngroup.com/articles/service-blueprints-definition/）

## id: user-story-mapping

- 一般名: ユーザーストーリーマッピング
- 適合する状況: 利用者が目的を達成する活動の流れに沿ってストーリーを並べ、関係者が全体像と実装範囲を対話したい場合。
- 下書きの骨格: 次のMarkdown見出しを使う。
  - `### 利用者と目的`
  - `### 活動の流れ`
  - `### 活動ごとのストーリー`
  - `### 不確かな点と学ぶこと`
- 誤用注意: 機能一覧を並べ替えただけの表にしない。利用者の活動の流れを背骨にし、詳細は対話と学習に応じて更新する。
- 出典: Jeff Patton & Associates, “Story Mapping”（https://jpattonassociates.com/story-mapping/）
