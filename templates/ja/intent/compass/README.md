# compass 記号の分割収納（canonical・1記号=1ファイル）

`.intent/intent-compass.md`（旧本体）の記号（INV / DR / Anti / C / A）を、1記号=1ファイルで収納する置き場です（INV80・正規化収納）。**この収納が空・不在でも、すべてのスキルは従来どおり旧本体で動きます**（読み手にとって旧経路は恒久フォールバック＝DR133）。

## ファイル規約（1記号=1ファイル）

- ファイル名 = 記号 ID（例: `INV80.md`・`DR131.md`・`Anti-478.md`）。**次番号のファイルを作成すること自体が採番の宣言**です（DR131）。中央の採番カウンタは持ちません。並行セッションが同じ番号を作ったら、git の同名ファイル衝突として顕在化させます（黙って番号をずらさない）。
- frontmatter は最小スキーマ（既存概念の昇格のみ・新しい分類軸を発明しない）:

```markdown
---
id: INV80          # ファイル名と同一の記号 ID
area: 派生         # 旧本体の [領域: <name>] タグの昇格（横断規律は always）
status: active     # active | superseded（superseded の理由・後継は Annex に書く）
---
```

- 本文は二層（law / annex）:

```markdown
# INV80 <題名>

## Law
<規範本文 1〜2文。これだけ読めば従える>

## Annex
<経緯・プレモータム・検査オラクル・影響パス・追補・採番注記>
```

## index.md（派生・手編集しない）

- `index.md` は 1記号=1行の派生キャッシュです: `- <ID> [領域: <area>] <status> — <要旨1行>`
- canonical（この収納の記号ファイル）を変更したスキルが、処理完了時に再生成します（`packets/index.md` と同じ規約）。実体とズレても「再生成で直る派生のズレ」であり、衝突ではありません。

## 読み手の契約（新旧両対応）

- この収納に該当記号があれば `index.md` → 該当ファイルの `## Law` を読みます（記号単位・数KB 以下の pull）。
- 無い記号は従来どおり旧本体 `.intent/intent-compass.md` を領域タグ grep（`[領域: <name>|always]`）で読みます。旧経路は削除しません（DR133）。
