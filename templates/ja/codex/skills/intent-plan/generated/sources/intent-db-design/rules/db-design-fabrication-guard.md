# 捏造抑制：DB 設計記述↔射影元の照合（トレース・inferred/unverified 標識）

`intent-db-design` skill が、射影した DB 設計（テーブル定義・制約・インデックス・命名）が**射影元（対象 packet / compass invariant / 既存スキーマ）を超えて捏造していないか**を照合し、各記述が射影元へ辿れることを保証するための正本。SKILL.md は手順と報告形式のみを持ち、「各記述をどう射影元へ結び、何を `inferred` として標識し、何を `unverified` として標識し、何を確認用に提示するか」は本ルールを参照する。本ルールは生成 DB 設計と射影元の**照合**を担い、射影元（intent-tree / compass / packets / 既存スキーマ）を一切変更しない（読み取りは入力範囲限定、配置は射影の形式仕様、書き込みは SKILL.md の派生 Write の責務）。

DB 設計の load-bearing 課題は**捏造抑制**である。射影元に無いカラム・制約・インデックス・命名規則を、設計として流暢に埋めてしまう confabulation を抑えることにある。DDL や表として読めてしまうがゆえに、根拠の有無は本文を読んだだけでは判別できない。本ルールはその判別を、生成 DB 設計の各記述を射影元へ突き合わせる照合として与える。これにより「なぜこの正規化・キー設計か」を後から検証でき、根拠のない設計を排除できる（R2）。

## 照合の発想は intent-to-spec の fabrication-guard と同型（生成物↔射影元）

- 本照合は `intent-to-spec/rules/fabrication-guard.md`（射影した自然言語 Spec を Intent / packet / 制約へ突き合わせる照合）と**同型**である。あちらは三層（intent-tree / compass / packets）を射影元とするが、本ルールは source に**既存スキーマ（Grep/DDL）**が加わり、target が**DB 設計（テーブル/制約/インデックス/命名）**になる点が違う。
- 照合の根拠は、入力範囲限定（db-design-input rule）が read-only で読み取った**三層の素材そのもの**（対象 packet の意図・Scope / compass の Invariants・Anti-direction / 既存スキーマ・migration）にする。新しい構造を持たず、その由来情報を辿って各記述の根拠を確かめる。
- 照合は**意味的な突き合わせ**であって機械判定ではない（INV2/A1）。`intent-check.mjs` 等の機械検査を呼ばず、各記述と射影元を意味で突き合わせる。疑わしい記述（射影元に根拠が見当たらないのに DB 設計へ溶け込んでいるカラム・制約・インデックス）は拾う側に倒す。
- read-only。canonical な成果物（intent-tree・intent-compass・packets）および既存スキーマを一切変更しない（不可侵）。

## 手順

1. **各記述を射影元へトレースする（R2.1）**
   - 生成 DB 設計の各記述（テーブル定義・カラム・制約・インデックス・命名規則）について、それがどの射影元に由来するか（**対象 packet** の要求・Scope / **compass invariant** / **既存スキーマ** のいずれか）を、辿れる形の「由来」注記として付与する。
   - 注記は、入力範囲限定が read-only で読み取った素材に基づく。新たに射影元を読み直して作るのではなく、読み取った三層を辿って由来を落とす。
   - 1つの記述が複数の射影元に由来するときは、由来をすべて挙げる。トレースの付かない記述を確定として残さない。

2. **inferred と unverified を区別して標識する（R2.2 / R2.3 / R5.3・本ルールの核）**

   射影元への帰属が確認できない記述は、確定（射影元由来）と混在させず、**inferred と unverified の2種を区別して**標識する。両者を一括しない。

   - **`inferred`＝射影元に根拠が無いことを確認したうえでの推論。** 対象 packet・compass invariant・**既存スキーマ（同定済みの範囲）**のいずれにも対応が見当たらないことを確認した記述（流暢さのために補われたカラム・制約・インデックス・前提）を `inferred` として標識する。
   - **`unverified`＝既存スキーマを同定しきれず確認できていない記述。** 既存スキーマの読み取り範囲（ORM・SQL DDL 等の多様な表現）の一部が Grep で同定できなかった場合（R5.3）、その同定不能なスキーマ範囲に関わる記述は、`inferred` と一括せず `unverified`（未同定ゆえ未確認・実在しうる）として標識する。
   - **両者を一括しない理由**: 既存スキーマに**実在するのに**同定漏れで `inferred` 扱いになると、実在するものを「射影元に無い」と誤標識する（捏造抑制と逆方向の誤り）。`inferred`（=根拠なしを確認済み）と `unverified`（=未同定ゆえ要確認だが実在しうる）を分けることで、後続の落差検出が両者を区別して扱える。
   - 判定の落としどころ: ある記述が同定済みスキーマにも packet/invariant にも対応せず、かつ同定不能なスキーマ範囲に**関わらない**なら `inferred`。同定不能なスキーマ範囲に関わるなら `unverified`。どちらとも確信が持てないときは、実在を否定しない側＝`unverified` に倒す（実在するものを誤って捏造扱いしないため）。
   - `inferred`・`unverified` の記述を、確定（射影元由来でトレースの付く記述）と DB 設計本文の中で混在させない。「由来」注記で識別できる別標識として置く。

3. **トレース率 100% を満たす（R2.3）**
   - 生成 DB 設計の**すべての記述**を、`対象 packet` / `compass invariant` / `既存スキーマ` / `inferred` / `unverified` の**いずれかに帰属させる**。帰属不明の記述を残さない。
   - 各テーブル・各カラム・各制約・各インデックス・命名規則のどれにも、由来注記が付く。由来注記が空の記述・トレースの付かない記述を確定として残さない。
   - トレース率 100% は「全記述が上記5区分のいずれかに帰属する」ことで満たす。

4. **inferred / unverified 一覧を確認用に提示する（R2.2）**
   - 射影元を超えて補完した記述（手順2で `inferred` / `unverified` として標識した記述）の**一覧を、利用者が確認できる形で出力末尾に提示する**。
   - 一覧は、どの記述（どのテーブルの・どのカラム/制約/インデックス）が・どの理由で射影元に帰属しないか（`inferred`＝根拠なしを確認 / `unverified`＝未同定ゆえ未確認・実在しうる）を名指しし、利用者がその補完を容れるか退けるか判断できる粒度にする。`inferred` と `unverified` は一覧上でも区別して並べる。
   - 提示は確認のための警告であり、射影を止めるものではない。補完の採否は利用者の判断に委ねる。後続の落差検出は、この一覧の `inferred`（=確認対象）と `unverified`（=未同定ゆえ要確認だが実在しうる）を区別して扱う。

## 不変条件

- 射影元（intent-tree / compass / packets / 既存スキーマ）を読み直して再解釈しない・変更しない（照合のみ。読み取りは入力範囲限定、配置は射影の形式仕様の責務）。read-only・canonical 不可侵。
- すべての記述を `対象 packet` / `compass invariant` / `既存スキーマ` / `inferred` / `unverified` のいずれかに帰属させる（トレース率 100%・帰属不明の記述を残さない）。
- `inferred` と `unverified` を区別する（一括しない）。`inferred`＝射影元に根拠が無いことを確認した推論。`unverified`＝既存スキーマを同定しきれず確認できていない（実在しうる）。実在するものを同定漏れで `inferred` と誤標識しない。
- `inferred` / `unverified` の記述を確定（射影元由来）と混在させない（別標識を崩さない）。
- 補完した箇所（`inferred` / `unverified`）は必ず出力末尾の一覧として確認用に提示する（黙って DB 設計本文へ溶かし込まない）。
- 照合は意味的な突き合わせであって機械判定ではない（INV2/A1）。機械検査ツール（intent-check.mjs 等）を呼ばない。疑わしきは拾う側に倒す。
