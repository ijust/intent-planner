# Question Pack Surfacing（案件種別の質問パックの候補提示）

案件種別ごとの詰め質問カタログ（`.intent/question-packs.md`）を read-only で照合し、効きそうなパックを**候補提示**する手順。`/intent-discover` の詰めの局面（designer-questions の手順から適用される）で使う。制約の定石カタログ（constraint-starters）の照合とは**別カタログ・別対象**（制約の叩き台 vs 問いの定石）であり、互いを置き換えない。

## 規律（守ること）

- **候補提示のみ・採否は人。** 合致したパックを提示するだけで、パックの内容を Anti-direction / Invariants / intent-tree / カタログ・台帳へ自動で書き込まない。
- **当てはまりが弱ければ黙る。** 誤検知より黙る側に倒す（候補は絞る・押し付けない）。
- **カタログ不在なら沈黙してスキップする**（停止しない・何も変わらない＝従来どおりの詰め・後方互換）。
- **「推論+確認」の既定を置き換えない。** パックの問いを順に全部聞くチェックリスト化・ゲート化をしない。採用時も、効きそうな問いを**少数**選んで既存の詰めの流れに混ぜるだけ。
- **問いの歯止め（INV58）を継承する**: まとめて少数（1バッチ最大4問）・各問に「なぜ聞くか」一行・毎回「後で／不明／該当なし」を選べる・値（妥当な既定回答）を先に置かない。
- **照合は意味的に行う**（パックの `適合する状況` を案件文脈に照らす読解。機械スコアリング・スクリプトに寄せない）。
- **どのログにも記録しない。** 例外は採否の器のみ: 引き継がれた発行ディレクトリの `constraint-ledger.md`（あれば）へ、定石の採否記録と同じ書式で1行追記する（`| パックid | 局面=discover | 採否 | 文脈一行 | 日付 |`）。器・発行ディレクトリの不在時は記録をスキップする（停止しない）。

## 手順

1. **カタログを読む**: `.intent/question-packs.md` を read-only で読む。不在ならスキップして沈黙する（従来動作）。
2. **採否済みは再提示しない**: 引き継がれた発行ディレクトリの `constraint-ledger.md`（無ければ沈黙）を読み、同一発行系列で既に採否が付いたパックは再提示しない。
3. **照合する**: 各パックの `適合する状況` を、いま扱っている案件の題材・モード（non-code 等）に照らす。弱い手がかりであり、当てはまりが弱ければ出さない。
4. **候補として提示する**: 合致したパックを1行で提示する（例:「この案件は `<id>`（<name>）の質問パックが効きそうです — 使いますか（採否はお任せします）」）。複数合致しても絞る。
5. **採用されたときだけ問いに使う**: 利用者が採用したパックから効きそうな問いを少数選び、各問に「なぜ聞くか」を添えて、既存の詰めの流れ（推論+確認）の中で使う。不採用・無応答なら何も変えず従来どおり進む。
6. **採否を器へ記録する**: 上記の器の規約に従う（器が無ければスキップ）。
## 問いの平易さ点検（出力直前・共通）

利用者へ問い・確認を出す直前に、次の5点を点検する（通らなければ平易に書き直してから出す。書き直しで問いの意味・選択肢は変えない）:

1. **単体で通じるか**: その問いだけを読んで初見の読み手に意味が通るか。直前に読んだ内部文書（compass・packet・rules 等）の語彙をそのまま転写していないか。
2. **詰め込みすぎていないか**: 未説明の専門用語が1つの問いに3つ以上並んでいたら詰め込みすぎの合図＝分割するか言い換える。
3. **識別子に言い換えを添えたか**: コマンド名・記号・packet 名などの識別子を出すなら、初出に一行の普通の言葉の言い換えを添える。
4. **普通の言葉に狭い意味を載せていないか**: 字面が普通の言葉（例: 代行・配布）でも、このプロジェクト／ツール固有の狭い意味を載せて使っていないか。載せたのが自分（ツール・AI）なら、その対話・文書の初出に一行の普通の言葉の言い換えを添える（世間の意味のまま使われている一般語・既存技術用語には付けない）。
5. **比喩や曖昧な言い方だけで意味を渡していないか**: 土台は正確さ（意味が一意に読める記述。平易さはそれを保ったまま易しく書く手段）。基準のない曖昧な言い方（「かなり」「うまく」等）や比喩単独で意味を渡さない。比喩を使うなら直後に正確な言い直しを必ず併記する（確立された専門用語・世間の意味のままの一般語は無理に普通の言葉へ開かない＝かえって曖昧になる）。

この点検は生成時の予防であり、事後検査（`/intent-validate` の造語検査）と対で働く（予防だけ・検査だけにしない）。
