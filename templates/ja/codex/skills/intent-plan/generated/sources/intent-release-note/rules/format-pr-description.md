# format 写像：pr-description（PR 説明の下書き）

`intent-release-note` skill が、git のコミット履歴と意図照合の結果を、**GitHub の Pull Request 説明欄へそのまま貼り付けられる下書き**へ組み立てるための出力構造の定義。SKILL.md は手順と git 読み取り・intent 照合を担い、「どの素材をどの見出しへ・どの順で置くか」は本ルールを参照する。既存の `format-changelog.md`（開発者向け・種類別）／`format-github-releases.md`（物語＋一覧）／`format-changelog-customer.md`（顧客向け・利用者影響先行）と並ぶ第4の出力形。

## 責務の境界（出力構造の定義であって読み取り・照合ではない）

- 本ルールは **format 写像（出力構造の定義）** である。PR 説明のセクション構成・並び順・各項目の書き出しだけを定める。
- git log を読む・コミットを意図（packet name / parent intent / deltas）と照合する・range を解釈するといった手続きは **SKILL.md の責務**であり、本ルールは行わない。
- **この format は gh を呼ばない**（`gh pr create`・PR 本文の直接更新を含め、GitHub への書き込みを一切しない）。生成物は下書きであり、GitHub への反映（PR 説明欄への貼り付け）は**人が行う**。外部書き込み経路をこの format に持たない。

## 出力先（既存の変更履歴を上書きしない）

- 出力は `.intent/release-note/pr-description.md` へ**全置換**で書く（派生・再生成可能）。
- 既定の `release-note.md`（changelog 系 format の出力先）とはファイルを分け、PR 下書きの生成が変更履歴の生成物を上書きして消さないようにする。

## 読み手と設計制約（結論先行・利用者影響先行）

読み手は**忙しいレビュアーとチームメイト**を想定する。この読み手は、差分の羅列より先に「この PR は何のためか・どこを先に見ればよいか」を知りたい。

- **結論先行（BLUF）**: 判断してほしい点・見どころを下書きの冒頭側に立てる。背景説明から始めない。
- **利用者影響先行**: コミットの羅列でなく「何が変わり、読み手にどう影響するか」を先頭に書く。
- **平易な言葉で書く（A33）**: 識別子（packet 名・packet_id・Invariant / Decision Rule の番号）を出すときは、初出で一行の言い換えを添える。内部設計文書の語彙をそのまま転写しない。

## 構成（4部・上から下へ）

| 順 | セクション | 何を置くか |
|---|---|---|
| 1 | **意図の要約**（なぜこの変更か） | range 内のコミットが支える意図（照合済みの「なぜ」）を数行で要約する。照合できる意図が無いときは推測で盛らず「意図の記録なし」を明示するか、書くなら inferred（推測）標識を付ける |
| 2 | **関係する判断基準** | この変更が守る Invariant / Decision Rule への参照（識別子+一行の言い換え）。pull 規律に従い**関係する分だけ**を挙げる（compass 全文からの網羅列挙をしない） |
| 3 | **レビューの見どころ**（先に見る順） | (a) 人が前倒しで固定した決定（packet の Human-fixed）が実装で守られているかの照合観点 (b) 受入オラクル（packet Validation の判別テスト）とそれをどこで確かめるか (c) Scope / Non-scope 境界（スコープ超過の兆候を見る場所）。各項目は packet / compass の**実在記述への参照付き**（この下書き自身が新しい主張をしない） |
| 4 | **コミット↔意図の対応一覧** | 記録に基づく対応（Intent trailer 由来）と推測による対応（テキスト照合由来）を区別して並べる。**紐づかないコミットは薄い行で残す**（黙って捨てない・推測で埋めない） |

- 該当素材が無いセクションは「無い」ことを一行で明示する（黙って見出しごと消して読み手に探させない）。

## 末尾のツール目印（1行だけ）

- 下書きの末尾に、intent-planner で生成した旨の**控えめな1行**を置く（例: `_この PR 説明は intent-planner の下書きから作成しました。_`）。
- 目印は**末尾の1行だけ**とし、本文の複数箇所に埋めない・広告調にしない。下書きなので、貼り付け前に人が消してよい。

## 機密の線引き（INV67・厳守）

- 外へ出すのは**要約+識別子**まで。packet 本文・compass 本文の段落をそのまま転記しない。
- `.intent/` / `.kiro/` の内部パス・dogfooding の開発文脈・顧客名・障害詳細を下書きに載せない（PR 本文も履歴の一部＝公開されうる）。
- 識別子で辿れる人（リポジトリにアクセスできる人）だけが内部の詳細に到達できる、が情報設計の床。

## 異常系（薄い下書き・エラーにしない）

- Intent trailer もテキスト照合の紐づきも無い range は**エラーにしない**: 対応一覧が薄い行だけの**薄い下書き**を生成し、意図の要約には「意図の記録なし」を明示する（trailer は任意規律であり、無いことは常態）。
- range にコミットが無いときは SKILL の既定（空である旨の明示）に従う。

## 不変条件

- git・canonical を読み直さない・変更しない（読み取り・照合は SKILL.md、書き込みは派生ディレクトリへの SKILL.md の Write の責務）。
- gh を呼ばない（外部書き込みゼロ・貼り付けは人）。
- 要約+識別子の線を越える転記をしない（INV67）。
- 紐づかないコミットを薄い行で残す順序と、記録に基づく対応／推測による対応の区別を崩さない。
- 素材に無い意図を体裁のためにでっち上げない（推測には inferred 標識）。
- 末尾のツール目印は1行だけ（欠落も肥大もさせない）。
