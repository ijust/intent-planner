# First Packet 推薦

最初に着手すべき packet をちょうど1つ、定性的な理由とともに推薦して記録する手順。`/intent-packets` の優先順位提示（Step 4）で常に使う。

## 適用条件

- **常時適用する。** `.intent/mode.md` の designer-questions に依存しない（off・未記録でも適用する）。
- SKILL.md Step 4 では、walking-skeleton の適用の**後**に本手順を適用する（walking-skeleton が適用されなかった場合も、本手順は適用する）。

## 手順

1. **材料を読む**
   - packet 候補と優先順位、（記録されていれば）`.intent/packets/plan.md` の「Walking Skeleton」セクション、`.intent/mode.md` の purpose を読む。

1.5. **工程計画があれば読み、推奨着手順を候補の入力にする（節が無ければ従来どおり）**
   - `.intent/packets/plan.md` に「工程計画」節（グループ見出し＋番号付き列挙）があれば read-only で読み、次の**導出規則**で推奨着手順を組んで、着手 packet 選定の入力にする（DR139）:
     - 節を**上から順に読む**。並び順そのものが優先順位（ラベル・数値は持たない）。**同番号は同順位**（どちらから着手してもよい）。
     - 各 packet について、`state=done`・`depends_on` に done でないものがある（未解決）・`.intent/assignments/` に他セッションの着手宣言がある、のいずれかなら**飛ばす**。
     - 並び順と `depends_on` が矛盾するときは **`depends_on`（技術的前提）が常に勝つ**。
     - 列挙された名前がどの packet とも照合できなければ「見つからない」と明示してその行を飛ばす（推測で紐づけない）。
   - **節が無い・空のときは、この手順を発火せず従来どおり**（後方互換）。工程計画は候補を絞る入力であって、下記手順2の定性観点（リスク低減 / 依存解放 / rollback 容易性 / 学びの大きさ）を**置き換えない**（数値スコアリングを持ち込まない・INV81/INV62）。推奨理由には、工程計画を参照したなら「工程計画で最優先グループの先頭」の旨を定性的に添える。

1.6. **他セッションの割当宣言を必ず読み、推薦に添える（工程計画の有無に関わらず発火・DR163/INV91）**
   - `.intent/assignments/*.md` を read-only で読む（不在・空なら何もせず従来どおり＝後方互換）。各宣言の `phase`（`drafting` | `implementing`・**不在は `implementing` として読む**）と `session` を見る。
   - **自セッションの宣言は自分に対して示さない**（`session` の乱数で自他を区別する。単独利用者に「他人が触っています」と誤って告げない）。
   - 推薦する packet に**他セッションの `implementing` 宣言**があれば、その旨を一言添える（「他セッションが実装中」）。
   - **起草中（`phase: drafting`）の宣言があれば、その発行ディレクトリ（`issue_dir`）で他セッションが意図を詰めている最中だと一言添える**（その案件から packet が起きる可能性がある＝二重着手の芽）。
   - **止めない・拒否しない・順位を機械で決めない**（read-only の助言に徹する・INV91）。着手するかは人が判断する。

2. **最初に着手すべき packet をちょうど1つ選ぶ**
   - 理由は次の定性観点から該当するものを挙げる: **リスク低減** / **依存解放** / **rollback 容易性** / **学びの大きさ**。
   - purpose=poc が記録されている場合は「仮説を最も安く反証できるか」の観点を必ず理由に含める（purpose が未記録・product の場合は purpose を参照しない）。

2.5. **優先順位・トレードオフの許容をユーザーに確認する**
   - 推薦を確定する前に、優先順位とトレードオフの許容をユーザーに確認する: **速度 vs 品質のどちらを優先するか**、**この着手で削ってよい／後回しにしてよいスコープはどこか**。
   - これは追認であり、推薦の上書きを促すものではない。推薦は提案であり、ここで得た利用者の優先順位判断が推薦と食い違う場合は、利用者の判断を優先して推薦・理由を見直す。
   - 「該当なし／不明／後で確認」を選べる形で問い、回答を強制しない。保留・「後で確認」が選ばれた場合は推測で埋めず、当該項目を `.intent/packets/plan.md` の「Open Questions」へ逃がして（無ければ既存内容を保持したまま追記して）推薦を継続する。
   - **確認の宛先はロールレンズに従う**: 引き継がれた発行ディレクトリの `mode.md` に `lens:` 行（ロールレンズ＝案件に必要な観点と本人/代行の別。読み手の契約は CONTRACT.md「スキル間の状態共有」）があれば、この確認を**本人**の観点はその人に向けて問い、**代行**の観点は推論した答えを暫定として提示して追認だけ求める。行が無ければ従来どおり（後方互換）。

3. **Walking Skeleton と整合させる**
   - Walking Skeleton（E2E を貫く最小実装）の決定が記録済みなら、推薦をそれと整合させる。整合しない推薦を行う場合は、その理由を明示する。

4. **推薦を記録する**
   - `.intent/packets/plan.md` の「Recommended First Packet」セクションに記録する: **推薦 packet**（packet 名）/ **理由**（定性観点）/ **Walking Skeleton との整合**（整合 / 整合しない場合はその理由 / Walking Skeleton 未記録）。
   - **旧 scaffold への非破壊追記**: plan.md に「Recommended First Packet」セクションが無ければ、既存の記録内容を保持したままセクションを追記してから記録する。

## 規律

- 数値スコアリング（重み付け合計・点数化）を用いない。
- 推薦は提案であり、利用者の優先順位判断を上書きしない。
- コードを変更しない。
