# Mode Selection

リポジトリの状況から Intent の詰め方モードを推奨するロジック。`/intent-discover` の冒頭で使う。これは拡張点でもある（新モードを足したら、ここに推奨条件を追記する）。

## 手順

1. **利用可能なモードを列挙する**
   - `.intent/modes/*.md` を Glob で探す。各モードの「適合する状況」節を読む。

2. **リポジトリ状況を軽く観察する**
   - 既存コードの有無・規模、テストの有無、README/docs の充実度、`.intent/` の既存成果物の有無。
   - 深く読み込まない。推奨に必要な手がかりだけ。

3. **モードを推奨する**
   - 現在の同梱モードは `standard` / `refactor` / `behavior-unknown` / `feature-growth` の4つ。リポジトリ状況から次の条件で推奨する:
     - 新規・意図が未言語化 → `standard`（既定）
     - 既存大規模・リファクタ対象（既存コード規模が大きい・設計ドリフトがある）→ `refactor`（意図を書かずに作られたコード＝vibe coding も refactor へ。discover で Intent Recovery を併用する）
     - 振る舞いが不明なレガシー（テストなし／少ない・仕様が失われている）→ `behavior-unknown`
     - 既存システムへの**機能追加**（extend / integrate / add-to 系の要望・振る舞いが既知・再設計が目的でない）→ `feature-growth`（使い分け: 既存の構造を変えたいなら `refactor`、既存へ付加したいなら `feature-growth`）
   - どの条件にも明確に当てはまらない場合は `standard` を推奨する。

4. **利用者に確認する**
   - 推奨モードと理由を提示し、`AskUserQuestion` で確認・変更を求める。
   - **モード候補が1つでも複数でも、推奨 → 確認 → 記録の流れは必ず通す**。これにより、モードが増減しても利用者の体験が変わらない。

5. **確定結果を記録する**
   - 確定したモードを `.intent/mode.local.md`（mode 状態のローカル正本・git 非追跡）に書く（mode / selected / reason / definition）。旧環境で `mode.local.md` が存在しない場合の読み取りは、後方互換フォールバック（`mode.local.md` → `mode.md` → standard）で吸収される。Enforcement / Drift-watch の読み取りは引き続き `mode.md` を参照する。

## 新モードを推奨対象に加える

新しい `.intent/modes/<name>.md` を作ったら、上記 3 の推奨指針に「どんなリポジトリ状況で <name> を推奨するか」を追記する。
## 問いの平易さ点検（出力直前・共通）

利用者へ問い・確認を出す直前に、次の5点を点検する（通らなければ平易に書き直してから出す。書き直しで問いの意味・選択肢は変えない）:

1. **単体で通じるか**: その問いだけを読んで初見の読み手に意味が通るか。直前に読んだ内部文書（compass・packet・rules 等）の語彙をそのまま転写していないか。
2. **詰め込みすぎていないか**: 未説明の専門用語が1つの問いに3つ以上並んでいたら詰め込みすぎの合図＝分割するか言い換える。
3. **識別子に言い換えを添えたか**: コマンド名・記号・packet 名などの識別子を出すなら、初出に一行の普通の言葉の言い換えを添える。
4. **普通の言葉に狭い意味を載せていないか**: 字面が普通の言葉（例: 代行・配布）でも、このプロジェクト／ツール固有の狭い意味を載せて使っていないか。載せたのが自分（ツール・AI）なら、その対話・文書の初出に一行の普通の言葉の言い換えを添える（世間の意味のまま使われている一般語・既存技術用語には付けない）。
5. **比喩や曖昧な言い方だけで意味を渡していないか**: 土台は正確さ（意味が一意に読める記述。平易さはそれを保ったまま易しく書く手段）。基準のない曖昧な言い方（「かなり」「うまく」等）や比喩単独で意味を渡さない。比喩を使うなら直後に正確な言い直しを必ず併記する（確立された専門用語・世間の意味のままの一般語は無理に普通の言葉へ開かない＝かえって曖昧になる）。

この点検は生成時の予防であり、事後検査（`/intent-validate` の造語検査）と対で働く（予防だけ・検査だけにしない）。

## 問いの内容点検（出力直前・共通）

利用者へ問い・確認を出す直前に、平易さに加えて次を点検する。この点検は質問数を増やすためではなく、必要な問いだけを適切な時点で出すために使う。

1. **既知事項を聞き直さない**: 指定された資料、今回の発行に属する Intent 成果物、そこから直接参照される必要な文書を、最初の重要な判断に必要な範囲から読む。資料または直前までの回答に答えがあれば聞かない。
2. **探索を広げすぎない**: 全資料の読了を質問開始の条件にしない。答えが見つからない重要な判断を示せる場合だけ参照先を広げ、答えが見つかった時点、または次の資料をその判断に結び付けられない時点で止める。
3. **重要な判断だけを問う**: 答えによって、目的・対象者・範囲・成功条件・使い勝手・守る約束・構成・後戻りしにくい判断のいずれかが変わる問いだけを出す。作業者の興味や念押しだけの問いは出さない。
4. **回答で次の問いを更新する**: 回答から確定事項、撤回された前提、残る未決事項を更新し、その状態から次の問いを作る。回答済み事項を言い換えて聞き直さない。
5. **症状と原因を分ける**: 否定的な反応だけで原因や解決策を確定しない。意図との不一致、新しい資料との矛盾、同じ症状への2回目の対処では、現在の作業範囲の外も含めて原因の層を見直す。単なる文言修正では最上位の目的からやり直さない。
6. **深さの歯止めを保つ**: この点検は `standard` と `deep` の両方に適用する。`deep` は確認する判断の範囲を広げるだけで、既知事項の再質問や無制限な探索を許さない。必要な判断が閉じたら質問を止める。
