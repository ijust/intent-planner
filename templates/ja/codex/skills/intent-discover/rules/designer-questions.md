# Designer Questions（設計者役の詰めの問い）

設計者役の詰めの問い（designer-questions）の要否確認・記録と、on のときに追加する質問群の手順。`/intent-discover` のモード確定後（Step 1）と Tree 更新案の提示時（Step 4）で使う。対話はすべて利用者への確認として行う（確認の手段は SKILL.md の規約に従う）。

**「推論+確認」の既定と deep の例外（A46・INV58）**: 本スキルの既定は「推論して確認する」姿勢であり、機能要件を一つずつ聞き出す全面的な能動質問化はしない（omission recap は最大1往復・consider-the-opposite は高コスト決定限定）。これは**詰めの深さ（question-depth）が standard のときの規律**であって、利用者が明示的に **deep（深掘り）** を選んだときだけ、問う**範囲**を広げる例外を重ねる（既定 standard の挙動・体験は一切変えない）。deep でも歯止め（まとめて少数・「後で／不明／該当なし」を毎回選べる・尋問調にしない）は外さない。deep は問いの範囲を広げるのであって、圧を上げるのではない。この規律は**問いの強度**の軸であり、AI からの**提案供給**（呼び水・複数案の起案＝本ファイル末尾「引き出し・起案レーン」・既定 on・DR196）とは軸が別——提案供給は AI が出す側を増やすだけで、利用者への要求量を増やさない（standard の問いの強度・deep の明示 opt-in は変えない）。

**成果の物さしを受け取ったとき**: リリース後に利用者価値が出たと分かる条件と観測元を、利用者が明示した場合にだけ、該当 L1 の `成果の物さし:` という任意の独立行へ記録する。この情報を追加で推測して埋めない。既存の `計測基準:` から推測したり自動転用しない。守る約束の破損確認を表す `検査オラクル:` とも混ぜない。この規律は designer-questions の on / off にかかわらず適用するが、新しい質問を必須にはしない。

## 手順

### モード確定後（Step 1）

1. **質問をしてよいかをストレートに確認する（トレードオフ明記・DR86-(3)）**
   - `.intent/mode.md` の `designer-questions` を読む。canonical 値は `on` / `off` の2トークン。
   - 未確定なら、**「これから内容を詰める質問をしてよいか」をストレートに問う**。仕組みの内部説明（下記4点）を先頭に置かず、まず**選択のトレードオフを明記して**2択で確認する:
     - **`on`（質問を受ける）= でき上がりの正確性が増す**。意図を先に擦り合わせるので、AI が利用者の思い描いたものと違うものを作ってしまうリスクが下がる。
     - **`off`（質問を飛ばす）= 開発速度が上がる**。ただし、意図の擦り合わせを省くぶん、**AI が利用者の思い描く仕様と異なるものを作ってしまう可能性が残る**。
   - 推奨はしない（どちらのトレードオフを取るかは利用者の判断）。**仕組みの詳細（何を確認するのか）は求められたら出す**: L1（成果）の成功基準の計測可能化 / 最初の packet（作業単位）が walking skeleton（入力から出力まで一通り動く最小実装）になっているかの確認 / ユーザー向け画面がある場合の画面ラフの有無 / 検証（PoC）の場合の仮説と完了判定。
   - 記録済みなら、記録内容（designer-questions / purpose / question-depth）を提示して変更の要否のみを確認する。
   - 利用者が決定を保留したら、推測で埋めず「設計者役の詰めの問いの要否が未確定」を Open Questions に記録して続行する。

2. **確定した designer-questions を記録する**
   - `.intent/mode.md` の `designer-questions` 行に確定したトークンを書く。後続スキル（intent-packets / intent-validate）はこの行を参照する。
   - **旧 scaffold への非破壊追記**: mode.md に designer-questions / purpose 行が無ければ、既存の mode / selected / reason / definition 行を保持したまま不足行を追記する。intent-tree.md に「PoC 実験定義」「画面ラフ参照」セクションが無ければ、既存セクションを保持したまま追記してから記録する。

2.2. **詰めの深さ（question-depth）を確認して記録する（designer-questions=on のときのみ・A46・DR86・INV58）**
   - designer-questions を **on** と確定した直後にのみ、**「さらに深く詰めるか（深掘り＝deep）」を1問で確認する**。off のときはこの手順を発火しない（deep は on の帰結）。
   - 値域は `standard`（既定の「推論+確認」の深さ）／ `deep`（深掘り）の2トークン。トレードオフを添えて聞く: **`deep` にすると、機能の中身・前提・エッジケース・反例・非機能（誰が困ると失敗か）まで能動的に問い、意図の取りこぼしをさらに減らす。ただし確認の往復が増える。** 推奨はしない（利用者の判断）。
   - 確定したトークンを発行ディレクトリの `mode.md` に **`question-depth:` 行**として記録する（**書き手は discover のみ**＝format/designer-questions と同じ一元化・DR26 同型）。旧 scaffold で行が無ければ既存行を保持したまま追記する。**未記載・未知値は standard 扱い**（後方互換・読み手は standard にフォールバック）。
   - 利用者が保留したら standard を既定として続行し（推測で deep にしない）、「question-depth が未確定（standard で続行）」を Open Questions に記録する。

2.3. **引き出し・起案の姿勢（proposals）のオプトアウト記録（designer-questions の値に関わらず有効・DR196・C82）**
   - 呼び水・複数案の起案（本ファイル末尾「引き出し・起案レーン」）は**既定 on** であり、要否の確認質問はしない（毎回の設定質問を増やさない・DR196。体験してから外せる形にする）。**利用者が「例やたたき台・複数案の提案は要らない」と明示したときだけ**、発行ディレクトリの `mode.md` に **`proposals:` 行**として `off` を記録する（**書き手は discover のみ**＝format/question-depth と同じ一元化・DR26 同型）。値域は `on | off` の2トークン。**未記載・行なし・未知値は on 扱い**（既定 on の後方互換。旧 scaffold＝行なしも on）。旧 scaffold で行が無ければ既存行を保持したまま追記する。
   - `off` を記録した発行では、引き出し・起案レーン（呼び水・複数案の起案）を一切発火せず、従来の問い中心の体験へ縮退する。利用者が戻す意思を明示したら行を `on` へ書き換える。

2.4. **ロールレンズ（必要な観点の判定と問いの出し分け・designer-questions の値に関わらず発火・A48・INV60・INV61・DR94・DR95）**
   - 案件の性質と利用者の職種から、この案件に必要な**観点**を推論して利用者に追認する。観点は**固定リストではない（閉じた値域にしない・DR94）**。よくある例は3つ: **製品を決める観点**（誰の問題か・価値・優先順位。いわゆる PdM）／**進行を管理する観点**（見積もり・リスク・依存。いわゆる PjM）／**体験を設計する観点**（導線・接点・使い勝手。いわゆるサービスデザイン）。これに限らず、**その案件の専門領域の観点**（例: 法務・医療・営業・教育など、案件の中身を正しく判断するのに要る職種の目）を案件から推論してよい。UI（利用者向け画面）を含む案件では、**画面を設計する観点**（各画面の目的・情報の優先順位・主要状態・見た目の方向。体験を設計する観点とは別）も候補に挙げてよい（これも例示であり、固定リスト化しない）。複数可・「どれも不要」も可。確認は「該当なし／不明／後で確認」を選べる形で行い、回答を強制しない（誤判定の害は追認1問で止まる）。
   - 追認された観点ごとに「その観点を持つ人がこの案件にいるか」を確認し、**在=本人**／**不在=代行**を確定する。
   - **記録**: 確定結果を発行ディレクトリの `mode.md` に **`lens:` 行**（例: `- **lens**: 製品を決める観点=代行 / 法務の観点=本人`。観点名は普通語の自由記述・旧来の略語表記も正当な値）として記録する（**書き手は本スキルのみ**＝format/question-depth と同じ一元化。読み手の契約は CONTRACT.md「スキル間の状態共有」）。発行ディレクトリが無い旧環境ではスキップし従来動作（未記録=従来動作の後方互換）。**観点を持つ人の在否は git 非追跡の発行ディレクトリにのみ書き、`intent-tree.md` 等の canonical へ書かない**（組織の情報を共有物に載せない）。
   - **出し分け**: 以後の問い（手順 2.5 以降・Step 4・後続スキルの確認）を、**本人**の観点はその人に向けて問い、**代行**の観点は「この観点は代行します」と宣言したうえで AI が観点の問いに自ら暫定回答し、推測（inferred / Assumptions）標識付きで tree 更新案に置いて利用者は承認・修正だけする。保留・未記録の観点は従来どおり。読み手の出し分けは観点の**名前に依存しない**（在/代行の別だけで動く＝観点名が自由記述でも壊れない）。
   - **歯止め**: INV58（まとめて少数・保留可・尋問調にしない）を継承する。ただし**「まとめて少数」に収めることを理由に、案件理解に必要な問いを削らない。1回に収まらないときは回数を分けて全部聞く**（範囲は保ち、圧を上げない）。問い文言は普通の言葉で書く——**床は「その職種の初見の人」に通じること**（DR95）。観点の呼び名は普通語を主にし、職種略語（PdM 等）は括弧の別名に留める。未説明の専門用語を3つ以上連ねない・識別子は初出で一行の言い換えを添える。
   - **デザインフレーム候補への接続**: ロールレンズを確定・記録した直後に、designer-questions の on / off の値に関わらず、`rules/design-frame-surfacing.md` を読み、適用する。このruleが候補提示、人の採否、採用時だけの派生生成までを扱う。
   - **観点の名乗りと常時の気づきへの接続（C81・DR196・INV102）**: ロールレンズを確定・記録した直後から、`rules/role-perspective-review.md` の「名乗りと常時の気づき」節を適用する（designer-questions / question-depth の値に関わらず。`proposals: off` の発行では発火しない＝「引き出し・起案レーン」と同じ縮退・未記載=on）。以後の問い・気づき・起案に観点の名乗り（例:「製品を決める観点から見ています」・代行は「（代行）」を添える）が付き、standard の深さでも各観点から短い気づき・見直し提案が推測標識付きで少数出る（確認論点の全量は deep のみ＝手順 6.6 の接続のまま・問いの量は増やさない）。

2.45. **案件種別の質問パックの照合（designer-questions=on のときのみ・read-only 候補提示）**
   - designer-questions を **on** と確定（または on と記録済みと確認）した後に、`rules/question-pack-surfacing.md` を読み、適用する。案件種別ごとの詰め質問カタログ（`.intent/question-packs.md`）を read-only で照合し、効きそうなパックを候補提示する（採否は人・採用時の問いも INV58 の歯止めの内側・カタログ不在なら沈黙してスキップ＝従来どおり・後方互換）。
   - これは候補の提示までであり、以降の手順（2.5〜）の発火条件・文言を変えない。off のときは発火しない（「質問を飛ばす=速度優先」の選択を尊重する）。

2.5. **目的・成功・想定ユーザーの追認（designer-questions の値に関わらず発火）**
   - 推論で導出した L0 目的・成功の定義を利用者に追認する: `.intent/intent-tree.md` の L0（Product Purpose）と L1（Desired Outcomes）に置いた推論内容を提示し、「この目的・成功の捉え方で合っているか」を確認する。
   - 推論した想定ユーザー・利用文脈（Actor）を利用者に追認する: 「誰がどんな状況でこれを使うと想定しているか」を提示し、合っているかを確認する。
   - 各確認は「該当なし／不明／後で確認」を選べる形で提示し、回答を強制しない。これは推論+確認の哲学に基づく追認であり、機能要件の全面的な能動質問化は**しない**（L2〜L4 を一つずつ聞き出すのではなく、根の目的・成功・想定ユーザーの捉え方が誤っていないかだけを確かめる）。
   - 利用者が追認した内容は当該 L0/L1 直下に canonical として確定する。想定ユーザーは L1 の Actor として L1 直下に記録する。利用者が訂正したら、その訂正内容で canonical を置き換える。
   - 利用者が保留・「後で確認」を選んだ項目は、推測で埋めず `.intent/intent-tree.md` の Open Questions（export までに要回答なら `[export まで]` タグ）または Assumptions（仮の前提として置く場合）へ逃がして続行する。planning は止めない。

2.6. **解収束度の判定と解先当て確認（designer-questions の値に関わらず発火）**
   - 手順 2.5 の直後に、利用者の要望（idea・対象範囲）と構築中の Intent Tree を読み、要望が目指す target architecture（構成）の**解収束度**を判定する。判定は LLM の読解で行い、外部解析ツール・機械的スコアリングには依存しない。
   - 収束/発散/判断不能の3分岐で扱う:
     - **収束**（要望が特定の構成をほぼ一意に含意する。確立パターンに当てはまる）→ 下記の解先当て1問確認へ進む。
     - **発散**（複数の妥当な解が成り立つ）→ 解を先に当てず従来経路（手順7の anchoring 回避を適用し、Impact Analysis の棚卸し・中立な選択肢提示）へ進む。
     - **判断不能** → 推測で確定せず**発散として扱い**、判断できない旨を `.intent/intent-tree.md` の Open Questions に記録して続行する。
   - 確立パターンの例示は数個に留め、網羅列挙しない（例: cron 化・CLI 化・ワンショット化 等。fixation を避け、例は手がかりであって解の空間ではない）。
   - **収束時の解先当て1問確認**:
     - Impact Analysis の棚卸し・中立な選択肢提示より**先に**、推論した構成を「目指す構成はこれですよね」と**1問で**確認する（中立な複数選択肢へ分解しない）。
     - 確認は「該当なし／不明／後で確認」を選べる形で提示し、回答を強制しない。
     - 結果は手順 2.5 と同型の記録規律で扱う: **追認** → 推論した target architecture を `.intent/intent-tree.md` に canonical として記録する（記録先は該当 L3（設計意図）直下、または L1/L2 直下の構成方針として既存セクションを使う。新セクションは作らない）。**訂正** → 訂正された構成で canonical を置き換える。**保留・後で確認** → 推測で確定せず Open Questions（export までに要回答なら `[export まで]` タグ）へ逃がして続行する（planning は止めない）。
     - 解先当ては構成の**確認・記録まで**で、実装・コード生成はしない。

3. **検証性を確認する（designer-questions=on のときのみ）**
   - この開発が「何かを確かめる検証（PoC = `poc`）」か「本番・継続開発（= `product`）」かを利用者に確認し、`.intent/mode.md` の `purpose` 行に記録する。
   - on 確定直後に加え、再実行で designer-questions が on と記録済みかつ purpose が未確定の場合もこの確認を行う。
   - 利用者が決定を保留したら、推測で埋めず「purpose が未確定」を Open Questions に記録して続行する。

4. **PoC の3つの質問（仮説・反証条件・GO/NO-GO。purpose=poc 確定直後のみ）**
   - 次の順で利用者に質問する: 仮説（この PoC で何を確かめるか）→ 反証条件（何が観測できなければ仮説を棄却するか）→ GO/NO-GO 基準（PoC 完了後に先へ進む / やめるを判定する条件）。
   - 回答を `.intent/intent-tree.md` の「PoC 実験定義」へ canonical として記録する。回答できない項目は推測で埋めず Open Questions に記録して続行する。
   - **purpose が poc でないときは、この質問群を発火しない。**

### Tree 更新案の提示時（Step 4・designer-questions=on のみ）

5. **L1 計測基準を確認する**
   - 各 L1 項目について「達成をどう観測・判定するか」を利用者に確認し、該当 L1 項目に `計測基準:` 行として記録する。
   - 基準が定まらない項目は、L1 項目自体は保持したまま「基準未定」を Open Questions に記録する。

6. **画面ラフの有無を確認する**
   - まず L2/L3 にユーザー向け画面が含まれるかを判定する。
   - 該当なら、画面ラフの有無を利用者に確認する: 有 → 参照（ファイルパスまたはリンク）を「画面ラフ参照」に記録する。無 → 作成を推奨する。利用者が見送る場合は、その旨を理由付きで Open Questions または「画面ラフ参照」に記録して続行する。
   - 非該当なら、利用者への確認は行わず、「画面ラフ参照」に「対象外」を**必ず**記録する（intent-validate が推論なしで判定できるようにするため）。
   - ラフ自体の作成・生成はしない（有無の確認・推奨・参照の記録まで）。

6.1. **画面デザインを実装前に詰める対話と下書き（UI 案件かつ deep または画面設計観点の採用時のみ）**
   - 手順 6 の直後に、`rules/screen-design-brief.md` を読み、その発火条件（UI を含む案件・designer-questions=on・question-depth=deep またはロールレンズで「画面を設計する観点」が採用）に従って適用する。発火条件を満たさない案件では質問も下書きも増やさない（rule 側の沈黙規律に委ねる）。

6.5. **tree 版 omission recap で抜け・過剰を確認する**
   - Tree 更新案を確定する前に、推論・収集した L0–L3（目的・成功・想定ユーザー・主要な成果と機能の枝）を短く要約して返し、「抜けているもの・逆に過剰な前提はないか」をユーザーに確認する（LLM の見落とし・ハルシネーションを人間が補正する材料として提示する）。これは機能要件にも適用する omission recap であり、L2〜L4 を一つずつ聞き出すものではない。
   - ユーザーが**抜け**を指摘した場合は、その項目を `.intent/intent-tree.md` の該当セクションへ反映して再提示する: 目的・成功の抜け → L0（Product Purpose）／L1（Desired Outcomes）直下／想定ユーザーの抜け → L1 の Actor として L1 直下／成果・機能の枝の抜け → 該当する L1 直下の枝。
   - ユーザーが**過剰**を指摘した場合は、確認のうえ当該項目を canonical（intent-tree.md の該当セクション）から除去する。確信が持てない場合は除去せず `.intent/intent-tree.md` の Assumptions へ降格する（仮の前提として残す）。除去前に必ずユーザーへ確認する。
   - 各確認は「該当なし／不明／後で確認」を選べる形で提示し、回答を強制しない。保留・「後で確認」を選ばれた論点は推測で埋めず `.intent/intent-tree.md` の Open Questions（export までに要回答なら `[export まで]` タグ）へ逃がして続行する。
   - 再編集は最大1往復に留める（recap で延々と対話しない）。残る論点は `.intent/intent-tree.md` の Open Questions へ逃がす。

6.6. **深掘りの質問群（question-depth=deep のときのみ・A46・INV58）**
   - `mode.md` の `question-depth`（手順 2.2 で記録）を読む。**`deep` のときだけ**この手順を発火する（standard・未記載・未知値・designer-questions=off では発火しない＝既定体験を変えない）。
   - deep では、手順 6.5 の recap の範囲を超えて、主要な機能の枝について次の観点を**能動的に問う**: (1) 中身（その機能が具体的に何をするか）／(2) 前提（成り立つために必要な条件・依存）／(3) エッジケース（境界・異常な入力や状況）／(4) 反例（この方針が不適切になりうる状況）／(5) 非機能（誰が・どんなときに困ると失敗と言えるか）。standard では聞かなかった中身まで踏み込んでよい。
   - 最初に、今回に必要な論点を対話内で一覧にする。各論点は **未確認／回答済み／後で確認／不明／該当なし** のいずれかとして扱い、回答済み・後で確認・不明・該当なしの4つを終端状態とする。この一覧は対話を進めるための判断材料であり、新しい永続形式や台帳は作らない。
   - 最初の一覧を作った直後に、`rules/role-perspective-review.md` を正確に1回だけ読み、適用する。観点ラベル付きの必要論点を上で作った同じ一覧へ追加し、観点ごとの別の質問ループ、別の状態、別の永続形式は作らない。
   - **質問束の歯止め（INV58・厳守）**: (a) 未確認の論点から**まとめて少数**（1バッチ最大4問・一問一答で連射しない）を選ぶ。(b) 各問に**なぜ聞くか**の理由を一行添える（尋問調にしない・好奇心側の温度）。(c) すべての質問束で「後で確認／不明／該当なし／その場で終了」を選べる形にし、回答を強制しない。(d) 2回目以降は、直前の回答から確定・変更したことと、次に確認する理由を1〜3文で質問束の前に示す。
   - **継続と完了**: 未確認の必要論点が残る間は、未確認の論点だけから次の質問束を提示する。質問束の総数や往復回数に固定上限を置かず、未確認がなくなったら深掘りを完了する。たとえば未確認が9件なら **4 + 4 + 1** と分け、2回で打ち切らず3回目まで確認する。一方、全論点が終端したなら不要な次の質問束は出さない。
   - **明示終了**: 利用者が終了を選んだら、新しい質問束は提示しない。終了時に残る未確認の必要論点を `.intent/intent-tree.md` の Open Questions へ記録し、正常に完了する。
   - **再質問と追加の規律**: 終端した論点は言い換えて再質問しない。新しい必要論点は、利用者の回答または既存資料を出所・根拠として示せる場合だけ未確認として追加する。出所・根拠を示せない論点は追加しない。既存論点と同じ意味の論点も追加しない。追加した新しい必要論点は、既存論点と同じ終了条件で扱う。
   - 得られた回答は該当する L1–L3 の枝へ canonical として反映する（手順 6.5 と同じ記録規律）。「後で確認」と不明は推測で埋めず Open Questions（`[export まで]` タグは export までに要回答なら付す）へ記録し、非該当はその対話で終端済みとして扱う。planning は止めない。
   - deep でも**単一の推奨アンカーを置かない**（decision-slots の anchoring 回避規律を継承・DR199）——問いはするが「妥当な既定値」1案だけを先に置いて判断を引きずらない。**複数案の対等提示は可**（「引き出し・起案レーン」に従い、対等・根拠付き・推測標識付きで。推しがあるなら「推し」と明示する）。

7. **制約下の意思決定（④）の posture を確認する**
   - L2/L3 の能力のうち、データの書き換え・外部入力/システム境界・アクセス制御を伴うものを短く要約して提示し、「この機能群で、整合性・冪等性・エラー時の挙動・誰がアクセスできるか、を決めておくべき箇所はどれか」を**まとめて一度だけ**確認する（④の各項目を一つずつ能動質問化しない。omission recap と同じ「抜けていないかの確認」として行う）。製品判断（対象ユーザー・成功指標・スコープ外・代替案＝`intent-packets/rules/decision-slots.md` の product スロット）も同じ一度の確認に含める（こちらも一つずつ能動質問化しない）。
   - これは決定そのものを今ここで埋める手順ではなく、**どの能力に決定スロット（④）を播くべきかの posture を捉える**手順である。**解が発散している設計判断に限り**、単一の推奨アンカー（「妥当な既定値」の1案先出し）を置かない（anchoring bias の回避・DR199。複数案の対等提示は可）。一方、手順 2.6 で**収束**と判定された構成要望には anchoring 回避を適用せず、推論した解を先に当てる（手順 2.6 の解先当てを参照）。スロットの一覧・発火条件は `intent-packets/rules/decision-slots.md` が正本（その anchoring 回避規律は本手順の限定に関わらず不変）。
   - 利用者が「ここは決めておくべき」と指したものは、該当 L3 直下に「決定が要る点（④）」として `.intent/intent-tree.md` に記録する（具体値はまだ書かない。実際のスロット播種と値の確定は intent-packets が packet 単位で行う）。
   - **高コストな決定に限り consider-the-opposite を促す**: 利用者が高コストな決定（認可・整合性・エラー意味論・後方互換）について方針を選んだ場合に限り、その方針が不適切になりうる条件（脅威・エッジケース・適用外の状況）を1つ以上挙げるよう促す（確証バイアスの緩和）。挙がった条件は当該「決定が要る点（④）」に併記して `.intent/intent-tree.md` に記録する。
   - この反証の促しは高コストな決定にのみ課す。全スロットに摩擦を課して対話を重くしない（decision fatigue の回避）。
   - 各確認は「該当なし／不明／後で確認」を選べる形で提示し、回答を強制しない。保留は推測で埋めず Open Questions へ逃がして続行する。

## designer-questions が off のとき

off のとき発火するのは、手順 1〜2 の要否確認と、手順 2.4（ロールレンズ）、手順 2.5（目的・成功・想定ユーザーの追認）、および手順 2.6（解収束度の判定と解先当て確認）のみ。ロールレンズは「問いの宛先と代行」の判定であり、designer-questions の on/off（詰めの問いの量）とは独立の軸のため off でも発火する。手順 2.2（詰めの深さの確認）・手順 2.45（案件種別の質問パックの照合）・手順 3〜7（検証性確認・仮説3質問・L1計測基準・画面ラフ・④ posture）・手順 6.1（画面デザインの詰めと下書き）・手順 6.5（tree版 recap）・手順 6.6（深掘りの質問群）は発火しない。詰めの深さ（question-depth=deep）は designer-questions=on の帰結ゆえ、off では確認も発火もしない（記録が残っていても参照しない）。L0 目的・成功・想定ユーザーと解の形（target architecture）は PoC 専用情報（手順 3〜6）と性質が異なり、product 開発でも意図の根として必要なため、off の最小構成にも含める（解が自明な要望で回り道を防ぐ効果は off でも有効）。purpose の値が残っていても、designer-questions が on と記録されていない限り参照しない。引き出し・起案の姿勢（proposals・手順 2.3）は designer-questions とは**独立の軸**であり、off の発行でも既定 on のまま働く（止めるのは `proposals: off` のオプトアウトだけ・DR196）。

## 引き出し・起案レーン（proposals・既定 on・C79/C80・DR196〜DR199・INV102）

本スキルが利用者へ問い・確認を出すすべての場面（手順 1〜7・Step 4・designer-questions の on / off を問わない）に重ねる姿勢のレーン。発行ディレクトリ `mode.md` の `proposals:` 行が **`off` のときはこのセクション全体を発火しない**（従来の問い中心の体験へ縮退する。未記載・行なし・旧 scaffold は on 扱い＝既定 on・DR196）。このレーンは AI からの**供給**を増やすものであり、利用者への**要求量を増やさない**——質問束の歯止め（INV58: まとめて少数・「後で／不明／該当なし」を毎回選べる・尋問調にしない）と deep の明示 opt-in は1ビットも変えない。

- **呼び水を添える（C79）**: 問いを出すとき、詰まりが予想される箇所と、利用者が実際に詰まった箇所（答えが薄い・沈黙・「分からない」・白紙）には、例・たたき台・「あなたの文脈ならこうではないか」という**呼び水を推測標識付き**（inferred / 推測 の明示）で添える。詰まっている利用者に、呼び水も案も出さず問いだけを繰り返さない（Anti-553 白紙で待つ受け身の禁止）。提示の順序・遠慮の条件は課さない——詰まり・発散・見落としと読んだすべての箇所で積極的に提案してよく、利用者の枠組み・前提への異論も提案としてよい（DR197 提案は自由・確定は人間）。詰まりの判定は LLM の読解で行い、機械条件・閾値を持たない。空振り（実は詰まっていなかった）の害は提案1回で止まる（採否で流せる）。
- **複数案を対等に起案する（C80）**: 発散する判断点（L1 成果の形・解の方向・Open Questions の選択肢 等）では、その判断点で**実質的に異なる選択肢を網羅**し、**対等・根拠付き・推測標識付き**で起案して、利用者の採否・修正・合成を確認する。**単一の推奨アンカーを置かない**——推しがあるなら「推し」と明示して判断材料として出す（隠れたアンカー＝案の書き分け・並び順・文量の温度差で事実上の推奨を仕込むこと＝が違反・Anti-550 単一案アンカー）。同じ案の言い換え・粒度違い・些末な変種を別案として数を膨らませない（Anti-552 網羅の水増し。網羅とは実質的に異なる選択肢の網羅であり、採否の実質 > 網羅の量）。
- **重さで出し分ける（DR198）**: **軽い判断点**は対話にそのまま全案を出す。**重い判断点**（後戻りが高い: 既定の変更・正本スキーマ・公開契約・不可逆決定）は全量を発行ディレクトリ内の文書（例: `proposals-<論点スラッグ>.md`）に置き、対話には要約+代表案+「全案はここ」の参照を出す。対話の1問で扱える選択肢の上限（**最大4択**）を超える網羅は、自動的に「文書+要約」側へ倒す。重さの判定は AI の読みで行い、**迷ったら文書側**（安全側）に倒す。
- **起案しても決めない（INV102・厳守）**: 呼び水・案・暫定回答はすべて推測標識付きの**叩き台**であり、**利用者の採否を経ずに canonical（intent-tree / compass / packet の確定内容）へ昇格しない**。利用者の無反応・話題の流れ・時間経過を承認として扱わない（Anti-551 起案の自己採用の禁止）。採否を経たものだけを手順 2.5 / 6.5 と同じ記録規律で canonical へ反映し、採否前の案は Assumptions / Open Questions に推測標識付きで置くに留める。

## 問いの平易さ点検（出力直前・共通）

利用者へ問い・確認を出す直前に、次の4点を点検する（通らなければ平易に書き直してから出す。書き直しで問いの意味・選択肢は変えない）:

1. **単体で通じるか**: その問いだけを読んで初見の読み手に意味が通るか。直前に読んだ内部文書（compass・packet・rules 等）の語彙をそのまま転写していないか。
2. **詰め込みすぎていないか**: 未説明の専門用語が1つの問いに3つ以上並んでいたら詰め込みすぎの合図＝分割するか言い換える。
3. **識別子に言い換えを添えたか**: コマンド名・記号・packet 名などの識別子を出すなら、初出に一行の普通の言葉の言い換えを添える。
4. **普通の言葉に狭い意味を載せていないか**: 字面が普通の言葉（例: 代行・配布）でも、このプロジェクト／ツール固有の狭い意味を載せて使っていないか。載せたのが自分（ツール・AI）なら、その対話・文書の初出に一行の普通の言葉の言い換えを添える（世間の意味のまま使われている一般語・既存技術用語には付けない）。

この点検は生成時の予防であり、事後検査（`/intent-validate` の造語検査）と対で働く（予防だけ・検査だけにしない）。
