# DB 固有検査オラクル：5検査軸・invariant 適合照合・不可逆性警告（warn-only・read-only）

`intent-db-design` skill が、射影した DB 設計（テーブル定義・制約・インデックス・命名）の**叩き台そのものの品質**を、DB 固有の検査軸で検査し、深刻度付きの所見として射影出力に併走させるための正本。SKILL.md は検査 Step（Step 3.5）から本ルールを参照するだけで、「どの検査軸を・機械と意味のどちらで・どの深刻度で・どう出力するか」は本ルールに閉じる。

本検査は射影骨格（seam）が出した叩き台の**品質検査**であって、実装スキーマ（migration/DDL）との突合（落差検出・別 spec）ではない。ここで検査するのは「叩き台そのもの」であり、実装との突合は持たない。検査はすべて **warn-only（gate ではない・誤検知前提・停止しない）・read-only（所見の提示に留め勝手に修正しない）** であり、canonical な成果物（intent-tree・intent-compass・packets）・既存スキーマ・export 下書きを一切変更しない。

## 検査の位置づけ（warn-only・intent-validate と同型）

- 本検査は intent-validate の「深刻度付き報告・修正は提案にとどめる・read-only」作法と**同型**に揃える。独自の検査出力契約を発明せず、別ファイル機構を増やさない。所見は seam の射影出力（`.intent/db-design/<packetスラッグ>/`）に併走させる。
- すべての検査は **warn-only** である。いかなる検査結果も射影・出力を停止しない（gate ではない）。不可逆性警告・品質検査は誤検知を前提とし、利用者に「気づかせる」ことが目的であって「指図して矯正する」ことではない（context-cost-cues の「指図せず気づかせる」と同型）。enforcement の gate（停止）と混同しない。
- 検査は**意味的な突き合わせ**であって機械判定の再実装ではない（INV2/A1・INV35(4)）。`intent-check.mjs` 等の機械検査スクリプトへ組み込まず、正規化妥当性等の意味判断を脆い grep に落とさない。検査は skill/rule の自然言語規約と LLM の意味判断に閉じる。
- read-only。射影元（対象 packet / compass / 既存スキーマ・migration）および canonical な成果物を一切変更しない（不可侵）。検査は所見の提示に留め、勝手にスキーマを修正しない。

## 5検査軸

射影出力（テーブル定義・制約・インデックス・命名）に対して、次の5軸で検査する。

1. **正規化崩れ**: 同じ事実が複数テーブル/カラムに重複して保持される、推移的関数従属で第3正規形を崩している、繰り返し列（`tag1`/`tag2`/`tag3` 等）で第1正規形を崩している等の構造を検査する。ただし非正規化が**意図的な性能設計**である場合もあるため、断定せず検討喚起に留める（意味軸）。
2. **欠落インデックス**: 外部キー列・頻出検索列（WHERE/JOIN/ORDER BY で使われそうな列）にインデックスが無い箇所を検査する。FK 列のインデックス欠落は JOIN 性能に直結するため拾う側に倒す。
3. **N+1 誘発スキーマ**: 親子1:N の取得で、子テーブルに親 ID のインデックスが無い・中間テーブルの設計が逐次取得を強いる等、アプリ側で N+1 クエリを誘発しやすいスキーマ構造を検査する（断定でなく検討喚起・意味軸寄り）。
4. **制約漏れ（NOT NULL・UNIQUE・FK・CHECK）**: データ整合性に直結する制約の欠落を検査する。具体的には (a) 参照を持つ列に **FK 制約**が無い、(b) 必須であるべき列に **NOT NULL** が無い、(c) 一意であるべき列（email・ログイン ID 等）に **UNIQUE** が無い、(d) 値域・状態遷移に **CHECK** が無い。**この軸はデータ整合性直結のため見逃しを嫌い感度高め**にする。
5. **命名一貫性**: テーブル/カラムの命名規約の不統一を検査する。`userId` と `user_id`（キャメル/スネークの混在）、テーブル名の複数形/単数形の不統一（`users` と `order`）、ID 列の接尾辞不統一（`user_id` と `userKey`）等。**この軸は機械的に拾える**。

## 機械軸/意味軸の仕分け（INV2・INV35(4)）

各検査軸を「機械的に拾える」か「LLM の意味判断に委ねる」かに仕分ける。意味判断を脆い grep に落とさない。

| 検査軸 | 区分 | 仕分けの根拠 |
|---|---|---|
| 命名一貫性 | **機械軸** | 識別子の表記（キャメル/スネーク・複数形/単数形・接尾辞）の不統一は文字列の照合で機械的に拾える。所見は「混在を検出」で断定してよい。 |
| 制約漏れ（構造的欠落） | **機械寄り** | 「FK を示す列名なのに FK 制約が無い」「`*_at` 列に NOT NULL が無い」等、構造的欠落の検出は機械的に拾える。ただし「その制約が**本当に必要か**」の妥当性は意味判断。 |
| 欠落インデックス | **機械寄り** | 「FK 列にインデックスが無い」は構造的に拾える。「頻出検索列か」の判断は意味（クエリパターンの推測）。 |
| N+1 誘発スキーマ | **意味軸** | アプリのアクセスパターンを推測する判断であり、構造だけでは断定できない。検討喚起に留める。 |
| 正規化崩れ（妥当性） | **意味軸** | 第3正規形を崩すべきか否か（意図的非正規化か事故か）は文脈依存の意味判断。断定せず検討喚起に留め、false-positive で利用者を矯正しない。 |

- 機械軸の所見は断定してよい（「`userId` と `user_id` の混在を検出」）。
- 意味軸の所見は断定でなく検討喚起にする（「`orders` と `order_item` で正規化方針が揺れている可能性。意図的非正規化なら問題ない」）。

## invariant 適合照合（意図 vs 出力スキーマ・INV35(3)）

「invariant に意図を書いた」だけでは「書いたが守られたか不明」（INV35(3)）。出力スキーマがその invariant に適合しているかを照合する。

- compass の Invariants（および対象 packet 固有の Safety）に **DB 関連の意図**（immutable・追記専用[append-only]・論理削除[soft delete]等）があるかを読む。
- 該当する invariant があれば、射影出力スキーマがそれに適合しているかを照合する。例:
  - invariant が「このテーブルは immutable（生成後に変更しない）」を意図しているのに、射影出力が UPDATE 可能なスキーマ（更新を許す列構成・更新タイムスタンプ等）になっている → **「invariant 違反候補」**として深刻度付きで標識する。
  - invariant が「追記専用（append-only）」を意図しているのに、削除/更新を前提としたスキーマになっている → 違反候補。
  - invariant が「論理削除（soft delete）」を意図しているのに、物理削除前提（`deleted_at` 等の列が無い）になっている → 違反候補。
- 標識は「違反候補」であって断定ではない（warn-only）。意図の読み違いもあり得るため、根拠（どの invariant と・どのスキーマ要素が衝突するか）を添えて検討喚起にする。
- **該当する DB 関連 invariant が無いときは、invariant 適合照合をスキップする**（該当なし）。

## 不可逆性警告（既存スキーマ差分・migration コスト・warn-only）

スキーマは不可逆性が高い（データが乗った後の migration コストが跳ねる）。既存スキーマ（seam の Grep 入力）と射影出力の差分に、データ投入後にコストが跳ねる変更があれば warn として標識する。

- 既存スキーマ vs 射影出力の差分のうち、次の変更を「不可逆性警告」として標識する:
  - **型変更**（例: `varchar` → `int`・データ再解釈や変換が要る）
  - **NOT NULL 追加**（既存 NULL 行があると失敗する・デフォルト埋めや段階移行が要る）
  - **カラム削除**（データ喪失・参照切れ）
  - **キー変更**（PK/UNIQUE/FK の変更・参照整合性の作り直しが要る）
- 警告には「データが乗った後だと migration コストが跳ねる」旨と、どの既存スキーマ要素に対する変更かを添える。
- **不可逆性警告は warn-only であり gate ではない**（誤検知前提・停止しない＝drift-watch の warn と同型）。警告が出ても射影・出力を停止しない。
- **既存スキーマが無い（新規 DB・比較対象なし）ときは、不可逆性警告を「比較対象なし」でスキップする**。品質検査（正規化・インデックス・制約・命名）は射影出力に対して実施する。

## 検査所見の出力形式（射影出力に併走・深刻度付き・intent-validate と同型）

- 検査所見は seam の射影出力（`.intent/db-design/<packetスラッグ>/db-design.md` の末尾または併走する形）に、**深刻度付き**で出力する。深刻度は intent-validate の報告と同型（要修正 / 推奨 / 情報、または severity: high / medium / low）に揃える。
- 各所見には **検査軸・深刻度・根拠（テーブル/カラム名）** を添える。例:
  - `制約漏れ severity: high — orders.user_id に FK 制約が無い（参照整合性が保証されない）`
  - `欠落インデックス severity: medium — users.email に検索用インデックスが無い`
  - `命名一貫性 severity: low — userId と user_id が混在`
  - `invariant 違反候補 severity: high — compass「このテーブルは immutable」に対し audit_log が UPDATE 可能なスキーマ`
  - `不可逆性警告 — users.age に NOT NULL を追加（既存データ投入後は migration コスト大・停止しない）`
- **感度方針（fit criterion 初期方針）**: データ整合性直結軸（FK/UNIQUE/NOT NULL 漏れ・制約漏れ）は**見逃しを嫌い感度高め**にする。主観的軸（正規化の是非・N+1 誘発）は warn/検討喚起に留め、false-positive で利用者を矯正しない。数値閾値は運用で再訪する（Deferred）。
- 所見は**修正の提案にとどめ**、勝手にスキーマを修正しない（intent-validate と同型）。

## 判定保留（Fail-Safe・捏造抑制）

- ある検査軸が**情報不足で判定不能**（例: クエリパターンが分からず欠落インデックスを断定できない・既存スキーマが部分的にしか同定できず差分を確定できない）なときは、**「判定保留＋理由」**を出力する。
- 判定保留の軸を**「合格」と誤標識しない**（Fail-Safe＝安全側は保留・捏造しない）。健全だと確認できていないものを「問題なし」と書かない。

## 不変条件

- すべての検査は **warn-only**（gate ではない・誤検知前提・停止しない）。enforcement の gate（停止）と混同しない。
- **read-only・canonical 不可侵**。射影元（対象 packet / compass / 既存スキーマ・migration）・canonical 成果物・export 下書きを一切変更しない。検査は所見の提示に留め、勝手に修正しない（修正は提案）。
- 検査は**意味的な突き合わせ**であって機械判定の再実装ではない（INV2/A1）。`intent-check.mjs` 等の機械検査スクリプトへ組み込まず、正規化妥当性等の意味判断を脆い grep に落とさない。命名一貫性等の機械的に拾える軸と、正規化妥当性等の意味判断を**分離**する（INV35(4)）。
- **永続ストア・外部接続なし**。検査は静的であり、実行時に外部 DB へ接続しない。永続ストア（データベース等）を導入しない（INV2）。
- invariant 適合照合は「違反候補」の標識であって断定ではない。該当する DB 関連 invariant が無いときはスキップする。
- 不可逆性警告は既存スキーマが無いとき「比較対象なし」でスキップし、品質検査は実施する。
- 判定不能な軸は「判定保留＋理由」とし、合格と誤標識しない（捏造しない）。
