# format 写像：顧客向け changelog（利用者影響先行）

`intent-release-note` skill が、git のコミット履歴と意図照合の結果を、**顧客・利用者向けの changelog**（製品を使う人が「自分にとって何が変わるか」を読む変更履歴）へ組み立てるための出力構造の定義。SKILL.md は手順と git 読み取り・intent 照合を担い、「どの素材をどの見出しへ・どの順で置くか」は本ルールを参照する。既存の `format-changelog.md`（開発者向け・種類別）／`format-github-releases.md`（物語＋一覧）と並ぶ第3の出力形。

## 責務の境界（出力構造の定義であって読み取り・照合ではない）

- 本ルールは **format 写像（出力構造の定義）** である。changelog のセクション構成・並び順・各項目の書き出しだけを定める。
- git log を読む・コミットを意図（packet name / parent intent / deltas）と照合する・range を解釈するといった手続きは **SKILL.md の責務**であり、本ルールは行わない。
- 出力先が派生ディレクトリ（`.intent/release-note/`・git 非追跡・read-only）であること、canonical・git を書き換えないことは skill 側の不変条件であり、本ルールはその前提のもとで構造のみを与える。

## 読み手と設計制約（利用者影響先行）

読み手は製品を**使う人**（顧客・エンドユーザー）を想定する。この読み手はコミット題名や内部実装の差分ではなく、「自分の使い方にとって何が変わるか」を知りたい。したがって本 format は **利用者影響先行**（DR93 の `changelog-for-humans`）を構成の核とする。

- **各項目は「利用者に何が変わるか」から書き出す。** コミット題名の転記や内部差分の羅列にしない。「〜できるようになりました」「〜が速くなりました」「〜の不具合を直しました」のように、利用者から見た変化を先頭に置く。
- **内部実装の語を開く（A33）。** 関数名・モジュール名・内部識別子（INV/DR/pkt-）・リファクタ用語を裸で出さない。利用者が使う言葉に言い換えるか、利用者に影響しない内部変更は載せない。
- **1出力1目的（種別分離）。** 顧客向け changelog は利用者影響の伝達に徹し、開発者向けの技術詳細・移行手順を同じ出力に混ぜない（別 format = 開発者向け changelog がそれを担う。DR93 の `doc-type-separation`）。
- **利用者に影響しない変更は省く。** 内部リファクタ・テスト追加・CI 変更など、利用者から見て変化がないコミットは、顧客向けでは載せない（開発者向け changelog の Other 枠が拾う）。

## 構成（上から下へ）

| 順 | セクション | 何を置くか |
|---|---|---|
| 1 | 見出し（バージョン / range と日付） | 対象 range（既定は前タグ〜HEAD）の識別と、生成時点の日付 |
| 2 | 新しくできること | 利用者が新たにできるようになったこと（新機能を利用者目線で） |
| 3 | 良くなったこと | 既存の使い勝手・速度・体験の改善（利用者から見た変化で） |
| 4 | 直したこと | 利用者が困っていた不具合の修正（何が直ったかを利用者目線で） |

- 各セクションの各項目は、利用者から見た変化を先頭に置く（コミット題名で始めない）。意図に紐づく「なぜ」は、利用者に伝わる範囲で一言添える（内部の理由は開かない）。
- 該当する利用者影響が無いセクションは見出しごと省く（空見出しを作らない）。
- 利用者に影響しないコミット（内部変更）は本 format には出さない（載せる先は開発者向け changelog）。
- 素材に無い変化を体裁のために創作しない。利用者影響が読み取れないコミットは、無理に利用者向けの文言をでっち上げず落とす（開発者向けが拾う）。

## 不変条件

- git・canonical を読み直さない・変更しない（読み取り・照合は SKILL.md、書き込みは派生ディレクトリへの SKILL.md の Write の責務）。
- 各項目を「利用者に何が変わるか」から書き出す順序を崩さない（利用者影響先行の核）。
- 内部実装の語・内部識別子を裸で利用者へ向けない（A33）。
- 利用者影響が読み取れない変更を、体裁のためにでっち上げない。
