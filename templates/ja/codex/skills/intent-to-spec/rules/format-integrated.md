# format 写像：統合仕様書向け（requirements 横断を統合）

`intent-to-spec` skill が、source scope 解釈が読み取った三層の素材を、**統合仕様書向けの自然言語 Spec**（横断した requirements と受入条件を読める形で1つにまとめた、実装に近い体裁）へ組み立てるための写像規則。SKILL.md は手順と報告形式のみを持ち、「どの素材をどの順で・どの見出しに配置するか」は本ルールを参照する。

## 責務の境界（写像であって読み取り・照合ではない）

- 本ルールは **format 写像** である。素材の構成・順序・見出しへの割り当てだけを担う。
- 射影元（intent-tree / compass / packets / steering）を**読み直さない**。独自のパーサ・スキーマを持たず、source scope 解釈が受け渡した素材（どの層・どの見出し・どの packet 由来かを保ったもの）をそのまま並べ替える。
- 各記述を射影元へ辿れる形にすること、射影元に無い記述を inferred として標識すること、不変則を省略・改変せず保持することは**捏造抑制ルールの責務**であり、本ルールは行わない。本ルールは canonical 由来の素材と inferred 由来の素材（Assumptions / Open Questions）を**区別したまま**受け取り、その区別を崩さずに配置することだけを守る。
- 中間 format（上流と統合の中間）は、独立した別ルールを持たない。本ルールと format-upstream（上流向け）が両極を表し、中間は両極の**度合い**として表現する（上流向けの「why 前面」を統合側へどれだけ寄せるかの調整であり、別ルールを増やさない）。

## 基本姿勢（横断した要求を実装に近い形へ統合する）

統合仕様書向け Spec は、**横断した requirements と受入条件を、実装者がそのまま読み下せる1つのまとまりへ統合する**体裁である。読み手は実装に着手する立場を想定する。したがって：

- 散在する個別要求を、機能のまとまりごとに**統合**して並べ、各要求にその受入条件（満たすべき条件）を添える。
- why（目的）と制約（不変則・判断基準）は、個別要求を**支える前提**として簡潔に保ち、上流向けほど前面化しない（前面化を強めるほど上流向けへ寄る＝度合いの調整）。
- 流暢さのために素材に無い詳細・受入条件を足さない（補完が要る箇所は捏造抑制ルールが inferred として扱うのであり、本ルールが本文に溶かし込んではならない）。
- 既定の読み手は実装者（当事者）なので記号・識別子は辿るための参照として残してよい。**読み手が非当事者（顧客等）と指定されたときは `rules/reader-vocabulary.md` に従い、記号（INV/DR）を台帳で言い換え・台帳に無い識別子を本文から省く**（開き方の本体は同ルールに一本化する）。

## 構成（横断要求を統合し、各要求に受入条件を添える）

source scope 解釈が受け渡した三層の素材を、次の順で配置する。各セクションには、その素材がどの層・どの見出し由来かが後段の捏造抑制ルールで辿れるよう、由来を保ったまま渡す。

| 順 | セクション | 何を置くか | 主な素材の由来 |
|---|---|---|---|
| 0 | 意図の階層マップ図（冒頭・素材があるとき） | 案件の意図の階層（L0–L4）を Mermaid 図で示し、読み手に全体像を先に掴ませる | intent-tree の L0–L4 の見出し階層（後述「意図の階層マップ図」） |
| 1 | 概要（何を作るか） | 統合した仕様が向かう範囲と、それが満たす目的の要約 | Intent の North Star（compass）＋ intent-tree の L0–L1（簡潔に） |
| 2 | 前提となる不変則・制約 | 全要求に共通して守るべき不変則と判断基準 | compass の Invariants / Anti-direction / Decision Rules（＋指定時のみ steering 制約） |
| 2.5 | 設計ビュー節群（素材があるときだけ） | 一般的な設計書項目（構成・アーキテクチャ／データ／画面・接点／運用・障害時 等）を、**対応する素材が読み取れたときだけ**節として立てる | 後述「設計ビュー節群」の対応表に従う素材（compass の設計意図・packets の Scope/Safety・intent-tree の画面ラフ参照・db-design 叩き台 等） |
| 3 | 統合要求と受入条件 | 横断 requirements を機能のまとまりごとに統合し、各要求にその受入条件を添えた本体 | 横断 requirements（packets の個別要求・依存・証拠）＋受入条件 |
| 4 | 前提・未確定（あれば） | 確定した上記と**混ぜずに**別枠で置く前提・未解決の問い | intent-tree の Assumptions / Open Questions（inferred 由来） |
| 5 | 標準項目との突合表（末尾・常置） | 一般的な設計書標準（arc42 / IEEE 1016）の主要項目 × 本仕様書での充足を1行ずつ示す（有＝節参照／未記入＝素材なし） | 後述「標準項目との突合表」の照合面 |

- セクション 1–3 は canonical 由来の素材で構成する。本体はセクション 3 であり、横断 requirements を機能のまとまりごとに統合し、各要求に対応する受入条件を添える（要求と受入条件を切り離さない）。
- **セクション 0（意図の階層マップ図）は冒頭に置き、素材（intent-tree の L0–L4）が読み取れたときだけ出す**。全体像を先に掴ませる目的で概要（1）より前に置く。素材が無い／intent-tree 不在のときは図を省き理由を明示する（空図を出さない）。詳細は後述「意図の階層マップ図」。
- セクション 2 の why・制約は本体（3）を支える前提として簡潔に置く。これを前面化するほど上流向けへ寄る（中間 format は本セクションと上流向けの「why 前面」のあいだの度合いとして表現する）。
- **セクション 2.5（設計ビュー節群）は素材があるときだけ立てる**。既存4節（1→2→3→4）の骨格と順序を保ち、設計ビューは第3節（統合要求）の前に、対応する素材が読み取れた節だけを挿入する（素材が無い節は立てない＝空見出しを作らない）。詳細は後述「設計ビュー節群」。
- セクション 4 は inferred 由来（Assumptions / Open Questions）の素材を、canonical の本文へ溶かさず**末尾の別枠**に置く。素材が無ければセクションごと省く（空見出しを作らない）。
- **セクション 5（標準項目との突合表）は末尾に常置する**。設計ビューと違い、素材の有無に関わらず表そのものは出す（「何が有り何が無いか」の可視化が網羅性の担保のため）。詳細は後述「標準項目との突合表」。
- 該当する素材が無い層はセクションを省く（無い情報を埋めない）。ただし統合要求と受入条件（3）は統合仕様書向け体裁の要であり、素材が読み取れているなら必ず置く。突合表（5）は素材が無くても常置する（例外）。

## 意図の階層マップ図（冒頭・全体像を先に掴ませる）

案件の意図の階層（intent-tree の L0–L4）を冒頭に **Mermaid 図** で同梱し、読み手が本文に入る前に全体像を掴めるようにする。理論的裏付け＝言葉に図を添えると言葉だけより理解が上がる（マルチメディア原則）・全体から詳細へ段階的に見せる（C4 モデルの発想）。

- **描画規約は overview の `rules/mermaid-tree.md` を参照する（DR187・複製しない）**: ノード ID の導出・ラベルの禁止文字（`(` `)` `[` `]` `"` `/` を入れない）・孤立ノードを作らない（L0 以外は親エッジを持たせる）・図の直後にテキスト階層を常に併記・intent-tree が空/不在なら図を省いて理由を明示、といった描画の規約は同ルールが単一正本である。本ルールはその規約に従って L0–L4 を図にすることだけを定め、規約本文を複製しない。
- **素材の射影に限る（INV100）**: 図のノードは intent-tree の見出しから射影し、素材に無いノードを作らない。図に描いた階層関係も素材（見出しの入れ子）へ辿れる形にする（捏造抑制ルールの照合対象）。
- **範囲に従う**: source scope が部分指定（サブツリー）のときは、その範囲の L0–L4 だけを図にする。
- **素材が無いときは図を出さない**: intent-tree が未記入/不在のときは冒頭図を省き、「意図の階層が未記入のため図を省略（`/intent-discover` で起票できる）」と理由を明示する（空図・推測ノードを出さない）。
- 図の描画は読み手の閲覧環境（Mermaid 対応の Markdown ビューア）に委ね、実行時に描画・変換ツールを呼ばない（INV2）。テキスト階層の併記により、図が描画されない環境でも通読が止まらない。

## 設計ビュー節群（一般的な設計書項目・素材があるときだけ立てる）

統合設計書として読み手が期待する一般的な設計項目を、**対応する素材が読み取れたときだけ**節として立てる。素材が無い項目は節を立てず、後述の突合表で「未記入」と示す（一般論・定型文で埋めない＝素材の射影のみ）。

| 設計ビュー節 | 立てる条件（素材） | 主な素材の由来 |
|---|---|---|
| 構成・アーキテクチャ | 構成方針・境界・依存方向・副作用の素材があるとき | intent-tree の L3（設計意図・境界・依存方向）／packets の Scope・Safety |
| データ | 永続データ・スキーマの素材があるとき（下記「データ節の素材」参照） | packets の Scope/Expected の永続データ記述／**db-design 叩き台**（あれば） |
| 画面・接点 | 画面ラフ参照や UI/UX 制約の素材があるとき | intent-tree の「画面ラフ参照」（対象外は立てない）／L3 の UI/UX 制約 |
| 運用・障害時 | 障害・縮退・タイムアウト・エラー意味論の素材があるとき | packets の Decisions（error-semantics 等）・Safety／compass の運用 Invariant |
| 主要な設計判断 | 判断素材（Decisions・compass DR の Annex・supersede 系譜・deltas 昇格）があるとき | 判断素材（射影方式は下記「主要な設計判断」を参照） |

- 立てる節の見出し名は普通語にし、新語を発明しない（glossary と照合）。
- 各設計ビューの記述も、他節と同じく素材からの射影に限る（素材に無い設計を書かない＝捏造抑制ルールの照合対象）。

### 主要な設計判断（判断・議論の過程を体系立ててまとめる）

「なぜ今の形に決まったか」を、これまでに積み上がった判断・議論の過程（source-scope が判断素材として読み取ったもの）から**全部列挙せず体系立ててまとめて**射影する。統合設計書では、この節を統合要求（3）の前（arc42 の「設計判断」章に対応する位置）に置く。**射影方式（3層構造の索引→要約→詳細・選定基準・まとめ方の境界）は `rules/format-upstream.md` の「主要な設計判断」節を単一正本として参照する**（上流向けと統合で射影方式を複製しない）。統合設計書での違いは配置（統合要求の前）と、詳細度に応じた厚みが本体（統合要求）と同じ段で効くことだけ。数値スコアは入れない・要約は捏造照合の内側（INV73）。

### データ節の素材（db-design 叩き台の射影）

- `.intent/db-design/<スラッグ>/db-design.md`（`intent-db-design` の叩き台 DB 設計）が存在するとき、その **テーブル定義・制約・インデックス・命名** をデータ節の素材として射影する。
- 叩き台の **inferred / unverified 標識と射影元トレースはそのまま保持する**（確定へ格上げしない）。叩き台は「設計の叩き台であって要件ではない」という db-design の出力契約を、消費側でも変えない。
- 読み取り境界は `.intent/` 配下のまま（コードの migration/DDL を直接読まない——実スキーマとの突合は `intent-db-design` と `intent-validate` の既存責務）。
- db-design 叩き台が無い案件では、データ節は packets/tree の永続データ記述の範囲に留める（叩き台の不在を推測で埋めない）。

## 標準項目との突合表（末尾・常置・網羅性の可視化）

一般的な設計書標準の主要項目に対し、本仕様書がどこまで充足しているかを1行ずつ示す表を末尾に置く。**「揃っている振り」ではなく「何が有り何が無いか」の可視化が網羅性の担保**である（素材が無い項目を一般論で埋めない＝Anti-538）。

- 照合面（項目リスト）は **arc42（12節）** と **IEEE 1016（設計視点）** の主要項目を静的に持つ（本ルール内に定義・外部照会しない）。arc42 の主要項目例: 序論と目標／制約／文脈と境界／ソリューション戦略／ビルディングブロック（構成）／実行時ビュー／配置ビュー／横断コンセプト／設計判断／品質要求／リスクと技術的負債／用語集。IEEE 1016 の設計視点例: 文脈／構成／論理／依存／情報（データ）／インターフェース／相互作用／状態動的／アルゴリズム。
- 各項目の行に、本仕様書での充足を示す: **有**＝対応する節への参照（どの節で扱ったか）／**未記入**＝素材が無く節を立てられなかった旨。
- **「未記入」の項目には根治の導線を一言添える**: どの素材（intent-tree/compass/packets のどの節）を書けばその項目が埋まるかを示す。特にデータ項目が未記入のときは「DB 設計を伴う案件は先に `/intent-db-design` を回すとデータ節が厚くなる」を添える。
- 突合表は素材の有無に関わらず常置する（欠落が見えること自体が価値）。ただし全項目を機械的に転記するのではなく、案件の性質に照らして主要な項目に絞ってよい（網羅の照合面であって全節の転記強制ではない）。

## 詳細度（output depth）ごとの厚み

同じ統合仕様書体裁の中で、確定した詳細度（概要 / 標準 / 詳細）に応じて**統合要求と受入条件（本体＝セクション3）の厚み**を変える。詳細度は体裁と直交する軸であり（DR111）、新しい format を増やすものではない——構成（概要 → 前提となる制約 → 統合要求と受入条件）は3段とも共通で、本体に素材をどこまで書き下すかが変わる。

- **概要**: 概要（1）と前提となる制約（2）を短く。統合要求（3）は機能のまとまりごとの要求名と、あれば1行の受入条件までに留める。
- **標準**: 概要に加え、各要求に受入条件（満たすべき条件の要点）を添える（要求と受入条件を切り離さない核はどの詳細度でも保つ）。
- **詳細**: 標準に加え、各要求を packet 本文由来の素材（Scope の範囲・守るべき制約・観測可能な受入基準）まで書き下し、依存・根拠のトレースを密にする。

厚くするのは**素材を書き下す密度**であって、素材に無い受入条件を発明することではない（INV73）。素材が読み取れていない要求は、どの詳細度でも「未記入／未観測」と明示して省く。

**設計ビュー節群・突合表も詳細度に従う**: 設計ビュー節（2.5）は、概要では各ビューの要点1〜2行／標準では素材の要点まで／詳細では素材を書き下すまで、と厚みを変える（素材の射影の密度であって、素材に無い設計を足さない）。突合表（5）はどの詳細度でも常置し、行数は詳細度に応じて主要項目に絞ってよい（有/未記入の可視化そのものはどの詳細度でも保つ）。

## 不変条件

- 射影元を読み直さない・変更しない（写像のみ。読み取りは source scope 解釈、書き込みは SKILL.md の派生 Write の責務）。
- 横断 requirements とその受入条件を切り離さない（各要求にその受入条件を添えた統合構成を崩さない）。
- 上流向けと統合向けの中間は度合いとして表現し、別ルールを作らない（本ルールと format-upstream が両極）。詳細度（厚み）は上流／統合の**中間の度合いとは別の軸**であり、どちらの体裁でも独立に効く（DR111）。
- canonical 由来の素材と inferred 由来の素材を混在させない（区別を保ったまま配置し、標識そのものは捏造抑制ルールに委ねる）。
- 素材に無い詳細・受入条件を、文章を整えるために足さない（詳細度を上げても同じ＝素材で深くする・INV73）。
- 設計ビュー節（2.5）は素材が読み取れた節だけを立て、空見出しを作らない。既存4節（概要→前提制約→統合要求→前提未確定）の骨格と順序を保つ（設計ビューは第3節の前への挿入・末尾突合表の追加であって、既存節の組み替えではない）。
- 標準項目との突合表（5）は素材が無くても常置し、未記入の項目を一般論で埋めない（欠落の可視化が網羅性の担保・Anti-538）。照合面（arc42/IEEE 1016 の項目リスト）は本ルール内に静的に持ち、外部照会しない（INV2）。
- db-design 叩き台をデータ節へ射影するとき、inferred/unverified 標識と射影元トレースを落とさない（叩き台を確定へ格上げしない）。読み取り境界は `.intent/` 配下のまま（コードの migration を直接読まない）。
- 意図の階層マップ図（0）は素材（intent-tree の L0–L4）があるときだけ冒頭に置き、描画規約は overview の `mermaid-tree.md` を参照する（複製しない・DR187）。図のノードは素材から射影し、素材に無いノードを作らない（INV100）。素材が無ければ図を省き理由を明示する（空図を出さない）。図生成の共通枠組みを新設しない（Anti-537）。
