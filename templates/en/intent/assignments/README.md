# assignments — implementation claims (local-only)

This directory holds declarations of **which packet is being implemented by whom (which session/agent)** during parallel implementation. When a session starts implementing `.intent/packets/active/<packet_id>.md`, it **creates one claim file** here.

```
.intent/assignments/
  README.md                        ← this file (tracked; explanatory)
  <packet_id>-<session-rand>.md    ← 1 claim = 1 file (git-untracked)
  <other-packet_id>-<session-rand>.md  ← a different claim (never collides across parallel sessions)
```

## Why "1 claim = 1 file" (dissolving write collisions structurally)

When several agents/sessions run the same repo in parallel, the claims themselves are written in parallel. If everyone appends to a single aggregate file (one `.intent/assignments.md`), you get append collisions; if you add an `## Assignment` section to the packet body, "two people touching the same packet" means **the very double-claim you want to prevent causes a file collision** — a self-contradiction.

So, using the **same technique as packets** (`.intent/packets/active/<packet_id>.md`), each claim goes in its own file. Each session **only creates its own file** (it never rewrites any other file). Because they are separate files, **parallel writes never collide structurally** (no locks, no mutual-exclusion machinery).

## Claim file name and schema

The file name is `<packet_id>-<session-rand>.md`:

- `<packet_id>`: the ID of the packet being implemented (taken from the file name under `.intent/packets/active/`).
- `<session-rand>`: 4 lowercase-alphanumeric chars `[a-z0-9]` generated by the shell at start (`LC_ALL=C tr -dc 'a-z0-9' </dev/urandom | head -c 4`) to avoid parallel collisions. It is **not derived from an agent name, process ID, or environment variable** (just a random token standing for that session).

The body only needs a minimal frontmatter schema (the body may hold an optional note):

```markdown
---
phase: implementing                # drafting | implementing. If omitted, read as implementing
packet_id: pkt-20260704-...-elcc   # the ID of the packet being implemented
declared_at: 2026-07-06T02:00:00Z  # claim timestamp (ISO 8601; from the shell `date`)
session: a3f2                      # <session-rand> (a random 4-char token for this session)
note: ""                          # optional note (which terminal/context, etc.; may be empty)
---
```

## Drafting claims — created automatically by discover

You can claim not only implementation but also **drafting** (while running `/intent-discover` -> `/intent-compass` -> `/intent-packets`). This stops another session from implementing a case that is still being drafted, without either session realizing it (**it only makes you aware; it never blocks**).

- **Creation is automatic**: `/intent-discover` creates one claim in the same step that creates the issue directory, so a claim can never be forgotten. The filename is `discovery-<issue-dir-name>-<session-rand>.md`.
- **Deletion is manual**: whether drafting has ended is a semantic judgment, so no machine makes it (auto-deletion would remove a still-live claim). The session itself deletes the file when it is no longer needed.
- **Keyed by the issue directory**: no packet exists yet while drafting, so `packet_id` stays empty and the claim is identified by `issue_dir` (**never fabricate a packet ID**).

```markdown
---
phase: drafting                          # drafting in progress
issue_dir: draft-phase-claim-pydw        # issue directory name (.intent/discovery/<this>/)
packet_id: ""                            # no packet exists yet while drafting (leave empty)
declared_at: 2026-07-12T08:04:46Z
session: m5gy
note: "drafting: draft-phase claims and when they get read"
---
```

**Backward compatible**: an existing claim with no `phase` is read as `implementing`.

## How double-claims surface (read-only; never stops)

`/intent-status` and `/intent-overview` read this directory **read-only** and derive (the full reader contract is in `skills/CONTRACT.md`):

- **A claimed/unclaimed list**: which packets have a claim.
- **A double-claim warning**: if **two or more** claim files carry the same `packet_id`, they are named as a double-claim (**warning only; never stops or refuses**). The parallel-operation judgment stays with the human.
- **Stale-claim observation**: a packet whose claim lingers while implementation has stalled is shown only as "elapsed since the claim date" (**no machine threshold on elapsed days, no auto-release**).

In a repo with no claim files at all, the existing flow's output is unchanged (backward compatible).

## Creating and releasing claims (manual; not automated)

- **Create**: the session starting implementation creates one claim file itself.
- **Release**: once implementation finishes and the packet moves done→archive, that session deletes its claim file (there is no auto-cleanup).

## Tracked vs untracked

- This directory itself (`README.md`) is git-tracked (for the explanation).
- Claim files (`<packet_id>-<session-rand>.md`) are **git-untracked** (local coordination data; the installer registers them in `.gitignore`).
- A claim is a **separate layer** from the packet's `state` (`draft`/`ready`/`implementing`/… value). A claim only records "who declared, and when"; it never rewrites the packet's frontmatter (12 keys fixed) or state.

## When unsure which packet to implement next (reference the work plan; optional)

When working through multiple unimplemented packets in parallel, to help decide "which to start next", look at the recommended order of work if `.intent/packets/plan.md` has a "Work plan" section (human-declared group headings and order of work). Read it top-to-bottom, skip anything that is `done`, has an unresolved dependency (`depends_on`), or **is already claimed by another session (a claim exists in this directory)**, and the first remaining item is the next candidate to start (when the order and dependencies conflict, dependencies win = DR139). This is only guidance for reference — you may start a packet other than the recommended one (it is not enforced). If there is no work plan, pick freely as before. For a case whose canonical journey file (`.intent/packets/journeys/*.md`, when present) carries the work plan, that file is the primary reference and the plan.md "Work plan" section is the permanent fallback (the DR140 note; with no journeys/, behave as before). This reference does not change the separate-layer discipline of claims (neither the claim nor the recommendation rewrites state).
