# Spec Ingest Derived Views

> `/intent-from-spec` reads an input natural-language specification and writes the extracted intent candidates and gaps under this directory as derived, easy-to-transcribe views. Everything except this README is **NOT tracked by Git (local-only)**. The views placed here are **derived, NOT the source of truth**. The authority remains the canonical files such as `.intent/intent-tree.md` and `.intent/intent-compass.md`; these views are merely a snapshot (candidates / hypotheses) reflecting the input spec at read time.

## Structure

```
.intent/spec-ingest/
├── README.md         # this description (tracked by Git)
└── spec-ingest.md    # the ingest view produced by /intent-from-spec (untracked, regenerated by full replacement)
```

## Nature of this directory

- **Derived / regenerable**: `spec-ingest.md` is regenerated by **full replacement** from the input specification every time `/intent-from-spec` runs. Hand edits are overwritten on the next run (do not edit).
- **Not the source of truth**: The authority for design intent stays in the canonical `.intent/intent-tree.md` / `.intent/intent-compass.md`. This view holds **candidates / hypotheses (Assumptions)** extracted from the spec; editing it here does not propagate to the canonical files. To adopt an item, review it and **manually transcribe** it into the discover / compass dialogue (there is no automated handoff).
- **Not tracked by Git**: `spec-ingest.md` and everything else except this README are local-only and not tracked (only this README is tracked). By design this avoids merge conflicts and loss from overwrites in team settings.
- **Read-only extraction layer + scoped Write**: `/intent-from-spec` does not modify the input specification (read-only) and does not modify the canonical artifacts. Its writes are confined to this `.intent/spec-ingest/` directory.
