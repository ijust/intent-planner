# Journey Plan (drafting and updating journeys)

The procedure for drafting/updating a journey (a unit that bundles one change spanning multiple packets) into `.intent/packets/journeys/` for a multi-packet case. Used in Step 4 (priority presentation) of `/intent-packets`. The canonical schema/placement conventions live in `.intent/packets/journeys/README.md`, and the readers' contract lives in `skills/CONTRACT.md`, "The journey reading contract" (do not duplicate them here).

## Firing conditions (fire only when all hold; otherwise stay silent and proceed as before)

- **This case has two or more packets** (two or more newly drafted, or an update adding packets to an existing journey's members). **Never propose one for a single-packet case** (no padding, no ritual).
- `.intent/packets/journeys/README.md` exists (in an older environment without it, do not fire and record to plan.md only, as before = backward compatible; INV103).
- Do not depend on designer-questions / mode values (judge only by whether the case is multi-packet).

## Procedure

1. **Propose to the user in one question**: "This case splits into multiple packets — shall I draft a journey bundling them into `.intent/packets/journeys/`?" (offer approve / no need / decide later; never force an answer). **On "no need" or deferral, create nothing and proceed as before** (plan.md recording stays as-is).
2. **On approval, draft it**: create `journeys/<slug>.md`.
   - `journey_id` is `jny-<YYYYMMDD>-<slug>-<rand>` (date taken with the shell; `<rand>` = 4 chars of `[a-z0-9]` generated by the shell; same shape as packet IDs). Derive the slug from `name` with the slug rules in `rules/packet-format.md`.
   - The frontmatter is **7 keys, fixed** (`journey_id` / `name` / `lifecycle: active` / `packets` = the member packets' packet_id list / `created_at` / `updated_at` = equal on creation / `summary`). **Add no further required keys.**
   - The body is free-form. Write here what used to go into the plan.md journey sections: the work plan (order and dependencies), shared contracts (what several packets keep together and the integration checks), deferred notes, and so on.
   - If a file with the same slug already exists, never overwrite silently — ask the user whether it is a different case or an update to this one.
3. **Do not also record to plan.md (the parallel run ended with the 2026-07-17 migration)**: this governs **after approval, once the journey is drafted**; it does not conflict with step 1 recording to plan.md as before when the user declines or defers (no journey created). the primary home for the work plan and shared contracts is the canonical journey file, and the plan.md journey sections became reference stubs (derived). Whatever other rules define as written to plan.md (the Walking Skeleton confirmation record, Recommended First Packet, etc.) stays as-is (this rule governs only where the journey body goes). Reading via the plan.md route remains a permanent fallback (INV103).

4. **Updating an existing journey**: when member packets are added or the work plan changes, update that file's `packets` list and body, and stamp `updated_at` (**never stamp on a re-run with no content change** = idempotent).
5. **Only a human closes it**: write `lifecycle: archived` only on a human's declaration after confirming "all member packets done + the integration checks green", then move the file to `journeys/archive/<year>/` (never delete). A machine never closes a journey automatically, and no machine threshold such as elapsed days exists.

## What must never be written (do not break the readers' derivation)

- **Never write a progress state into the frontmatter** (no `state:` key, no percent-complete, no done counts; readers derive progress from the member packets' `state` every time = DR200).
- **Never add a journey key to the packet-side frontmatter (12 keys, fixed)** (references go one way only, journey->packet = DR203; reverse lookup is a grep over journeys/).
- No numeric scores, date commitments, or velocity (the work plan is order and dependencies only; INV81).
- Do not coin new words beyond "journey" (no phases, epics, etc.) from here.

## Temperature

- This is a proposal, not a gate. Choosing not to create a journey is always valid, and the packets step completes as before without one (INV103).
- Never change implementation code (this is a planning-record procedure, not an implementation tool).
