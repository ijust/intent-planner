# Always Gate (single-question confirmation for `always` registration)

Before registering a new compass symbol (INV / DR / Anti) as `always` (cross-cutting across all domains), a gate that asks just one question: "does it really affect all domains, and which domains does it not affect?" federated-governance / C-fed3 · INV101 · Anti-543.

`always` is the "federal constitution" that domain-scoped execution (improve / validate) **always reads together**; the thinner it is, the more the domain scope saves. Without this gate, new symbols pile onto `always` by inertia and the savings erode (keeping the central constitution small is the crux of federation).

## Trigger (only when `always` is about to be chosen)

In `rules/domain-write.md` (P-fed3, the writing-side area derivation), fire this gate **only when `always` is about to be chosen** as the new symbol's `area`. Do not fire it when a domain tag (a concrete domain like `並行` / `語彙`) is chosen (close with the area-confirmation single question).

**No double question (integrated with domain-write; most important)**: do not ask the area-derivation confirmation and the always gate twice in the same drafting. When domain-write's area confirmation is about to fall to `always`, connect to this gate's single question; for a concrete domain, close with domain-write's single question. Fold into **one** question from the user's view (what is the area → if always, then "really all domains?" as one continuous flow).

In a repo with no `.intent/domains/`, or a legacy scaffold with no domain tags, do not fire and behave as before (backward compatible; a permanent fallback of the same kind as INV101/DR133).

## What the one question asks

When `always` is about to be chosen, confirm in one question:

> Does this symbol really affect all domains? List any domains it does not affect.

- If domains it does not affect are named, **propose** one or more concrete domain tags excluding those (e.g. "then how about the `並行` tag?"). The user decides.
- Once "affects all domains" is confirmed, register as `always`. **The user's choice is final** (see Temperature).

## Temperature (not a gate; one question at most)

Despite the name "gate," this does **not** stop the write. If the user chooses `always`, follow it (do not refuse; do not fire repeated questions; same kind as INV101/INV91). The gate imposes only **the friction of one question** — a violation if choosing `always` becomes two or more questions of friction.

- Do not touch existing `always` symbols (currently 82). Re-assignment progresses gradually through operation (this gate applies only at new-symbol drafting; user-confirmed 2026-07-15).
- No cap, threshold, or machine judgment on the `always` count (INV2). The "too many" judgment stays with the human.
- It stays effective with just the one question in the drafting flow, adding no helper script (same kind as DR71).
