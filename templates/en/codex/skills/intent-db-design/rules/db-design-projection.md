# db-design projection output format (machine-diffable · provisional contract)

The **canonical reference for the output format** when the `intent-db-design` skill maps the read three-layer material (intent / invariants / existing schema) into a DB design draft. SKILL.md holds only the procedure and report format; "which structure / which headings / which tables to output in, and where to write" defers to this rule. The discipline of trace attachment and inferred / unverified marking is the `db-design-fabrication-guard`'s (this rule defines the frame of the format and presumes each statement's attribution note rides on that frame); target identification and the three-layer read are the `db-design-input`'s.

The output format this rule defines satisfies a structure that the downstream inspection (gap detection) can **diff per item** against the implementation schema (migration/DDL). The output is closed to derived artifacts under `.intent/db-design/<packet-slug>/` and never modifies the canonical (intent-tree / compass / packets), the existing schema, or export drafts (`.intent/cc-sdd/` / `.intent/openspec/`).

## Positioning of the format (provisional contract · not frozen)

This output format is not a "frozen contract" but a **provisional format that satisfies the minimum diffability**. Because the side that actually diffs (intent-validate's gap detection) is still undesigned in another spec, there is a risk of unilaterally fixing the format of the side to be diffed.

- This format's responsibility runs only as far as satisfying **a floor of inspectability** — "the downstream validate can diff per item" — and it does not hold the diff logic itself.
- When the downstream gap-detection spec is designed, verify in the field whether tables / columns / constraints / indexes are diffable per item against the implementation schema, and revise this format if it falls short (do not freeze it).
- Therefore this format concentrates on providing **stable diffable-unit headings / table structure**, and does not embed semantic judgments (normalization validity · missing indexes · naming consistency, etc.) into the structure. Semantic judgment defers to the downstream inspection oracle (INV2 / A1 · do not lean too far into machine inspection).

## Output file and structure

Per single target packet, output one file `.intent/db-design/<packet-slug>/db-design.md` (slug derivation under "Output layout" below). The structure is the following machine-diffable Markdown.

```markdown
---
source_packet: "<target packet name>"
generated_at: <ISO 8601>
projection_sources: [packet, compass-invariant, existing-schema]
---

# DB design (draft) — <packet name>

> Derived artifact. Generated by `/intent-db-design`. Git-untracked. This is a design draft, not requirements.

## Table: <table_name>   <!-- diffable unit = heading -->
- **Source**: <packet-derived | invariant-derived | existing-schema-derived | inferred>
- **Columns**:
  | Column | Type | Constraints | Source |
  |---|---|---|---|
  | id | <type> | PK | inferred |
  | <col> | <type> | NOT NULL, FK→<ref> | packet 4.1 |
- **Indexes**: <column · kind · source>
- **Naming conventions**: <snake / camel etc. · source>

## inferred / unverified list   <!-- to confirm -->
- Column <y> of table <x>: no basis in any projection source (inferred)
- Table <z>: existing schema could not be fully identified, unconfirmed (unverified · may exist)
```

### Discipline of each element

- **frontmatter**: holds `source_packet` (target packet name · canonical key) / `generated_at` (ISO 8601 · obtained in the shell) / `projection_sources` (the layers used in the projection). Hold state with the schema discipline of frontmatter, and introduce no persistent store (DB, etc.) (INV2).
- **`## Table: <name>` heading**: one table = one heading. This is the **diffable unit**, and the downstream validate maps this heading to the implementation schema's table per item. Split a heading per table; do not bundle multiple tables into one heading.
- **Column table**: the fixed 4 columns `| Column | Type | Constraints | Source |`. One row = one column = a **diffable unit**. The `Source` cell carries each column's projection-source attribution (packet / invariant / existing schema / inferred / unverified) (the discipline of attribution is the `db-design-fabrication-guard`'s).
- **Indexes**: enumerate the column · kind (unique / btree, etc.) · source in an identifiable form. State explicitly which kind on which column, so the downstream validate can diff per index.
- **Naming conventions**: state the table / column naming convention (snake / camel, etc.) and its source explicitly.
- **inferred / unverified list**: list as items "to confirm" at the end of the output. Distinguish and list `inferred` (inference after confirming there is no basis in any projection source) and `unverified` (existing schema could not be fully identified, unconfirmed · may exist) (do not lump the two · the canonical reference for the distinction is `db-design-fabrication-guard`).

## Floor of inspectability (a structure the downstream validate can diff)

- Satisfies that table names · columns · constraints · indexes · naming conventions are **identifiable by structured headings / tables**, and diffable per item against the implementation schema (migration/DDL).
- The diffable units are these two: a table = the `## Table:` heading, a column = a row of the column table. The downstream validate diffs these headings / rows against the implementation schema.
- What is embedded in the structure is only "diffable identifiers and attribution"; **semantic judgments are not dropped into the structure** — normalization validity · missing indexes · N+1 induction · constraint omissions · naming consistency, etc. (INV2). Those are the downstream inspection oracle's.

## Output layout (slug rule and collision rule)

### Slug rule (deterministic)

Derive the output directory name (slug) from the packet name **deterministically** in the following order. The same packet name always yields the same slug. This rule is **identical** to the slug rule of `packet-format.md` and the export skills (`map-cc-sdd` / `map-openspec`) (the output-directory derivation is aligned). This subsection is a **verbatim copy** of the same-named subsection of `packet-format.md`; when changing it, revise both at the same time.

1. Apply NFC normalization.
2. Trim leading/trailing whitespace.
3. Lowercase ASCII uppercase letters.
4. Replace whitespace and path-dangerous characters (`/ \ : * ? " < > |`) with `-`.
5. Collapse consecutive `-` into one.
6. Strip leading/trailing `-`.

- Non-ASCII characters (Japanese etc.) are preserved as-is.
- If the result is an empty string, use `unnamed-packet` as the slug and notify the user.

### Collision rule

- Treat it as a collision only when the slug matches an existing directory and that directory's `db-design.md` (the frontmatter's `source_packet`) points to a **different** packet name. Assign an alternate name with a sequence starting at `-2` and notify the user of the packet-name → directory-name mapping. **Do not overwrite silently.**
- When it points to the **same** packet name, it is not a collision but regeneration, and update that same directory's draft in place (same-directory update).

## Invariants (read-only · output boundary)

- The output is closed to derived artifacts under `.intent/db-design/<packet-slug>/`. Never modify the canonical (intent-tree / compass / packets), the existing schema, or export drafts.
- Do not write the output into `.intent/cc-sdd/` or `.intent/openspec/` (export artifacts) (the draft is not requirements · do not mix into export).
- Do not write into another packet's directory (the write target is only under the target packet's slug).
- This format is a provisional contract and is not frozen. When it turns out the downstream gap detection cannot diff, revise this format (do not fix it unilaterally).
